## Стек приложения

- Язык и рантайм: Go 1.23+.
- API: RESTful JSON API, OpenAPI-first, контракты из `api/{auth,slots,bookings,common}/*.yaml` —
  это уже готовый и провалидированный контракт, а не черновик, писать заново нельзя.
- HTTP: `net/http` + `chi`, middleware для auth, request id, recovery, logging.
- OpenAPI: `oapi-codegen` для генерации типов/серверных интерфейсов с сохранением `operationId`
  (`register`, `login`, `logout`, `listSlots`, `getSlot`, `createBooking`, `listBookings`,
  `getBooking`, `cancelBooking`, `submitRating`).
- БД: PostgreSQL 16, транзакции и атомарный `UPDATE ... WHERE free_seats > 0 RETURNING` для
  бронирования, `pgx`/`pgxpool`.
- Миграции и SQL: `goose` для миграций, `sqlc` для типобезопасных запросов без ORM.
- Auth: **email + пароль** (не OTP/телефон, в отличие от референса) — `bcrypt` для хеша пароля,
  непрозрачный (`opaque-token`, см. `auth/models.yaml` → `bearerFormat`) bearer-токен, хранимый как
  хеш в таблице сессий. Токен — **не JWT**: контракт прямо фиксирует `opaque-token`.
- Тесты: Go unit/integration tests, concurrency tests для booking/cancel.
- Runtime: Docker Compose для API + PostgreSQL, конфигурация через env, structured logs через `slog`.

## Правила декомпозиции

- Один пункт TOC = одна итерация: минимальный вертикальный срез (миграция → repo → service →
  handler → тесты), проверка перед переходом к следующему пункту.
- Любое изменение публичного API начинается с правки `api/*.yaml` и проверки линтером
  (`api/README.md` → «Валидация спецификации»). Контракт уже прошёл ревью — менять его без причины
  нельзя.
- **Не реализовывать вне скоупа текущей поставки** (см. `2-requirements/functional-requirements.md`
  → «Границы скоупа», `5-mobile-app-spec/feature-list.md` §6):
  - интерфейсы/API Шефа и Владельца/Администратора, CRUD расписания, программ, шефов;
  - онлайн-оплата;
  - лист ожидания (waitlist) на заполненные слоты;
  - программа лояльности;
  - сбор данных об аллергиях клиента (аллергены **блюда** — это `Program.allergens`, они в скоупе);
  - резервный канал уведомлений (SMS/email);
  - профиль клиента (просмотр/редактирование данных сверх email);
  - лимит/счётчик прокатного фонда экипировки (FR-9 — фонд не ограничен, полей-счётчиков в БД быть
    не должно);
  - `seats_count`/групповая запись — одна запись всегда одно место.
- Поскольку в этой поставке (в отличие от исходного допущения ТЗ про «существующую
  инфраструктуру») бэкенд пишем мы сами с нуля, **программы/шефы/слоты нечем создавать** — для них
  нужна seed-миграция (dev/test данные), а не полноценный CRUD/админка (см. BE-03).

### BE-03. Спроектировать БД, миграции и seed-данные

Сделать:
- Таблицы: `clients`, `auth_sessions`, `programs`, `chefs`, `slots`, `bookings`.

- Индексы: `slots(start_at)`, `slots(status)`, `bookings(slot_id)`, `bookings(client_id)`,

- Seed-миграция: тестовые `programs`/`chefs`/`slots` — единственный способ получить данные без

### BE-04. Реализовать Auth: email/password и сессии

Сделать:
- `POST /auth/register`: валидация email (формат) и пароля (`minLength: 8`, см.
  `auth/models.yaml`), `bcrypt`-хеш, `409` при занятом email, выдача токена + `Client`.
- `POST /auth/login`: сверка пароля через `bcrypt.CompareHashAndPassword`, `401 unauthorized` при
  неверных данных без уточнения, что именно неверно (email или пароль).
- `POST /auth/logout`: инвалидировать текущую сессию (`revoked_at`), не трогать остальные сессии
  клиента (per-device logout — решение реализации, ТЗ не уточняет multi-device поведение).
- Хранить только `token_hash`, сырой токен клиенту отдаётся один раз, при выдаче.

Готово, когда:
- Unit-тесты: успешная регистрация, дубликат email, успешный вход, неверный пароль, logout делает
  токен недействительным для последующих запросов.
- Integration-тест проходит полный flow register → authenticated request → logout → 401 тем же
  токеном.

### BE-06. Реализовать атомарное создание записи

Сделать:
- `POST /bookings`: в транзакции атомарно `UPDATE slots SET free_seats = free_seats - 1 WHERE id=$1
  AND status='scheduled' AND free_seats > 0 RETURNING free_seats` — это ядро гарантии «0 двойных
  записей» (NFR-6, R-004): при конкуренции ровно один запрос получает свободное место.
- При `free_seats=0` (0 обновлённых строк из-за условия) → `409 slot_full` с
  `details.available_seats=0`.
- При `status='cancelled'` → `410 slot_cancelled`.
- `equipment_choice='rental'` — **никогда** не проверять доступность фонда, принимать всегда
  (FR-9); полей для проверки в БД нет по построению (BE-03).
- `price_total = slot.price` на момент создания (снимок, не ссылка).
- `free_cancellation_until` в ответе — вычисляется, не хранится (см. BE-03).
- Создать уникальный индекс-барьер (BE-03) как защиту от повторной активной записи того же клиента
  на тот же слот — обсудить, возвращать ли на это отдельный код ошибки, или трактовать как
  идемпотентный повтор (в контракте `createBooking` для этого случая явного кода нет — решение
  реализации, см. открытый пункт ниже).

Открытый пункт (требует решения перед реализацией, не покрыт контрактом однозначно):
- Контракт `createBooking` не содержит `422 slot_started` (в отличие от `cancelBooking`/
  `submitRating`, где это явно есть) — то есть попытка записи на уже стартовавший слот форматально
  не описана отдельным кодом. Варианты: (а) трактовать как `slot_full`/`slot_cancelled` в
  зависимости от реального состояния и не добавлять отдельную проверку на `start_at`, (б) явно
  дополнить `api/bookings/api.yaml` кодом `slot_started` для `createBooking` до реализации. **Нужно
  зафиксировать выбор с аналитиком/владельцем контракта, не решать молча в коде.**

Готово, когда:
- Concurrency-тест с параллельными `createBooking` на один слот не допускает `free_seats < 0` и не
  создаёт больше активных записей, чем было мест.
- `equipment_choice='rental'` никогда не приводит к отказу независимо от того, сколько раз подряд
  выбран.
- Открытый пункт выше зафиксирован решением (код ошибки либо явный отказ от проверки) до мержа.

### BE-08. Реализовать отмену записи (правило 24 часов)

Сделать:
- `POST /bookings/{bookingId}/cancel`: в транзакции заблокировать booking и slot.
- Отмена доступна только пока `slot.start_at` в будущем; после старта — `422 slot_started`.
- Если до старта `>= 24ч` → `status='cancelled'`, `free_seats += 1` в слоте (место возвращается).
- Если до старта `< 24ч` → `status='late_cancel'`, место **не** возвращается, штрафа нет (FR-16 —
  никаких дополнительных полей штрафа в модели, только предупреждение на стороне клиента).
- Ровно `24ч` считать ранней отменой (граница включительна, по аналогии с трактовкой границы в
  референсе, но с порогом 24, а не 2 часа — **это ключевое отличие от `primer`, не перепутать при
  копировании кода**).
- Повторная отмена → `409 already_cancelled`.
- `cancelled_by_studio` через этот endpoint не устанавливается — это отдельный переход, инициируемый
  не клиентом (см. BE-10 и «Правила декомпозиции» про отсутствие owner-инфраструктуры).

Готово, когда:
- Unit-тесты на границы: `24ч + 1с`, ровно `24ч`, `23ч59м59с`, после старта.
- Concurrency-тест: параллельные попытки отмены одной брони не возвращают место дважды и не дают
  два разных статуса.

### BE-09. Реализовать оценку шефа

Сделать:
- `POST /bookings/{bookingId}/rating`: доступно только если `slot.start_at` в прошлом,
  `booking.status='active'`, оценка ещё не поставлена (FR-20).
- `422 not_ratable` — если слот ещё не прошёл или бронь не `active` (отменена в любом статусе).
- `409 already_rated` — повторная попытка оценить; редактирование ранее поставленной оценки не
  предусмотрено (UC-4 E2, явно зафиксировано в описании `submitRating`).
- Сохранить `stars` (1–5), опциональный `comment`, `created_at`.

Готово, когда:
- Тесты покрывают: успешную оценку прошедшей неотменённой записи, `422` для будущего слота,
  `422` для отменённой записи (любой из трёх статусов отмены), `409` при повторной попытке.

### BE-10. Закрыть открытый вопрос push-инфраструктуры (FR-21/FR-22)

Контекст: `5-mobile-app-spec/09_Логики/LOGIC-004_...md` и `api/README.md` прямо фиксируют, что
регистрация push-подписки **не входит в проанализированный контракт** — предполагалось, что этим
занимается «существующая инфраструктура». В нашей поставке такой инфраструктуры больше нет, мы
пишем весь бэкенд сами, поэтому этот пункт больше не абстрактный открытый вопрос, а реальный пробел
для реализации. FR-22 (push при отмене студией) имеет приоритет **Must** — без решения по этому
пункту требование не может быть выполнено.

Сделать (после согласования — этот BE-пункт **начинается** с правки `api/`, не с кода):
- Дополнить `api/bookings/` или завести `api/push/` operationId для регистрации `PushSubscription`
  (endpoint, публикуемый клиентом после `granted` в LOGIC-004).
- Добавить минимальный внутренний механизм перевода слота в `status='cancelled'` с
  `cancellation_reason` и каскадным переводом связанных `active`/`late_cancel` броней в
  `cancelled_by_studio` — без полноценной owner-панели (вне скоупа), но без него FR-17/18/22 не
  проверить даже вручную. Достаточно защищённого внутреннего endpoint или скрипта для dev/QA.
- Фоновый воркер (`robfig/cron` или тикер): напоминание за 24 часа до `start_at` по всем `active`
  бронированиям (FR-21, приоритет Should — можно после FR-22).
- Доставка — Web Push API (см. NFR-3, LOGIC-004: PWA, не нативный SDK) через VAPID.

Готово, когда:
- Решение по расширению контракта зафиксировано и внесено в `api/`.
- Тест на перевод слота в `cancelled` подтверждает каскад `cancelled_by_studio` на связанных
  бронях.
- Напоминание формируется ровно один раз на бронь (нет дублей при перезапуске воркера).
