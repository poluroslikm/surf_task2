# Задачи и баги — журнал реализации

> Конвенция по промту [`../../promt/задачи-и-баги.md`](../../promt/задачи-и-баги.md): на каждую
> задачу или баг, встреченные в ходе реализации фич из
> [`../FEATURES_IMPLEMENTATION_PLAN.md`](../FEATURES_IMPLEMENTATION_PLAN.md) (и далее — любой
> последующей разработки), заводится отдельный файл в этой папке.

## Формат файла

Имя: `TASK-NNN-краткое-название.md` для запланированной работы (пункт плана, доработка) или
`BUG-NNN-краткое-название.md` для дефектов, найденных по ходу реализации. Нумерация в каждом типе
сквозная, не переиспользуется (как в `2-requirements/` — см. `promt/анализ.md`, промт 3, «Правила
ID и нумерации»).

Содержимое файла — четыре обязательные секции:

```markdown
# TASK-NNN / BUG-NNN — короткое название

**Симптом/Цель:** что не так (баг) или что нужно сделать (задача) — коротко, по факту.

**Требования:** ссылки на FR/NFR/UC/US из `2-requirements/`, `5-mobile-app-spec/`, если задача
опирается на уже зафиксированное требование; для BUG — на что нарушено ожидание. Если формального
требования нет (например, бонус-фича сверх ТЗ или технический баг без бизнес-контекста) — так и
писать: «требование не найдено — вне формального ТЗ» или «баг реализации, не требование».

**Реализация:** что сделано — затронутые файлы, ключевые решения, что осталось не сделано (если
задача закрыта частично).

**Коммит:** хэш коммита(ов), в которых это реализовано. Публичного удалённого репозитория у
проекта нет (см. `README.md` в корне/историю разработки) — ссылка означает `git show <hash>`
локально, не URL на GitHub/GitLab.
```

## Зачем отдельная папка, а не заметки в коде

Трассируемость (см. `promt/анализ.md`, промт 1, раздел 9): по TASK/BUG-ID должно быть видно, какое
требование его породило и каким коммитом оно закрыто — так же, как FR/UC/US у аналитика. Коммиты
без сопроводительного файла в этой папке — не по правилу, если они не тривиальны (правки
опечаток, форматирование и т.п. отдельного файла не требуют).
