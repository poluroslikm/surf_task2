## Декомпозиция

### FEAT-01. Контракт push-подписки (новый домен `api/push`)

Открытый вопрос, зафиксированный в `api/README.md` → «Открытые вопросы, влияющие на контракт» и
`5-mobile-app-spec/09_Логики/LOGIC-004_...md` → «API-запросы»: регистрация `PushSubscription` не
входит в проанализированный контракт. Решение для этой поставки: **новый домен `api/push/`**, а не
подпуть внутри `auth` или `bookings` — подписка на push является самостоятельным ресурсом с
собственным жизненным циклом (создаётся клиентом, привязана к устройству/браузеру, не к конкретной
записи), по той же логике, по которой `bookings` — отдельный домен, а не подпуть `slots` (см.
`api/README.md` → «Почему только 4 домена» — пятый домен здесь оправдан тем же принципом: это
реальный ресурс, а не read-only проекция).

Сделать:
- `api/push/models.yaml` — схема `PushSubscription { endpoint: string(url), keys: { p256dh:
  string, auth: string } }` (стандартная форма Web Push API, `PushSubscriptionJSON`).
- `api/push/api.yaml` — одна операция `registerPushSubscription` (`POST /push/subscriptions`,
  `security: bearerAuth`, upsert по `endpoint` — повторная регистрация той же подписки не ошибка).
  Ответы: `201` (создана) или `200` (уже существовала, обновлена), `400 bad_request` (невалидный
  `endpoint`/`keys`), стандартные `401/default` через `$ref` на `common/models.yaml`.
- Явного `unregisterPushSubscription` **не делать** — управления уведомлениями в MVP нет
  (`LOGIC-004`: «отдельного экрана управления уведомлениями в MVP нет»), браузер сам инвалидирует
  старые подписки.
- Обновить `api/README.md`: добавить `push` в список 4→5 доменов, снять пункт из «Открытые
  вопросы» (перенести в «решено», сославшись на этот план).
- Обновить `5-mobile-app-spec/09_Логики/LOGIC-004_...md` → «API-запросы»: заменить «вне текущего
  API-контракта» на ссылку на `registerPushSubscription`.

Готово, когда:
- `npx --yes @redocly/cli@latest lint api/push/api.yaml` проходит.
- Контракт не меняет ничего в `auth`/`slots`/`bookings`/`common`, кроме `README.md`/`LOGIC-004`
  как документации решения.

### FEAT-03. Backend: регистрация подписки и отправка push (VAPID)

Сделать:
- `internal/domain/push.go`, `internal/storage/push_repo.go` (upsert по `endpoint`),
  `internal/service/push_service.go`, `internal/httpapi/push_handler.go` —
  `POST /push/subscriptions` по образцу существующих доменов (auth/slots/bookings).
- Библиотека отправки: `github.com/SherClockHolmes/webpush-go` — VAPID-подписанные push-сообщения.
- Конфиг: `VAPID_PUBLIC_KEY`, `VAPID_PRIVATE_KEY`, `VAPID_SUBJECT` (env, генерируются один раз
  скриптом/командой, публичный ключ отдаётся клиенту через `GET /push/vapid-public-key` **или**
  захардкожен в `client/src/core/config.ts` по аналогии с `API_BASE_URL` — второе проще для этой
  поставки, если публичный ключ не считается секретом (это норма для VAPID: публичный ключ по
  определению не секретен).
- Внутренняя функция `SendPush(subscription, payload)` — переиспользуется FEAT-04 (напоминание) и
  FEAT-05 (отмена студией); при `410 Gone`/`404` от push-сервиса (означает, что подписка отозвана
  браузером) — удалять её из `push_subscriptions`, не считать ошибкой сервиса.

Готово, когда:
- `registerPushSubscription` сохраняет/обновляет подписку, повторная регистрация того же
  `endpoint` не создаёт дубликат.
- Ручной тест: `SendPush` реально доставляет уведомление в браузер (через тестовую подписку из
  `pushManager.subscribe`, см. FEAT-06).

### FEAT-04. Backend: воркер напоминаний за 24ч (FR-21)

Сделать:
- Фоновый тикер (например, каждые 5 минут): выбирает `bookings` со `status='active'`,
  `reminder_sent_at IS NULL`, у которых `slot.start_at - now()` попадает в окно `[24ч, 24ч - 5мин)`
  (окно размером с интервал тикера, чтобы не пропустить и не задвоить).
- Для каждой — берёт все `push_subscriptions` клиента, шлёт push с текстом «Напоминаем: класс
  [программа] начнётся через 24 часа», выставляет `reminder_sent_at = now()` **независимо от
  успеха доставки** (иначе воркер будет бесконечно ретраить недоставимые подписки — авто-удаление
  мёртвых подписок уже покрыто в FEAT-03).
- Приоритет **Should** (FR-21) — после FEAT-05 (FR-22 — Must), если не хватит времени на обе.

Готово, когда:
- Ручной тест: бронь с `start_at` через ~24ч получает ровно один push, повторный прогон воркера
  не шлёт дубликат (`reminder_sent_at` уже проставлен).

### FEAT-05. Backend: dev-инструмент принудительной отмены слота студией (FR-17/18/22)

Полноценной owner-панели нет и не будет в этой поставке (см. «Правила декомпозиции» в
`BE_IMPLEMENTATION_PLAN.md`) — но без **какого-то** способа перевести слот в `cancelled` сценарий
FR-17/18/22 нельзя даже вручную проверить, не то что показать. Решение: минимальный dev-only
endpoint, явно не часть `api/` (не публикуется в OpenAPI-контракте, не для клиента).

Сделать:
- `POST /internal/slots/{slotId}/force-cancel` (body: `{ "reason": string }`), защищён отдельным
  заголовком (`X-Internal-Token`, сверяется с env `INTERNAL_TOOLS_TOKEN`), не роутится через
  `RequireAuth` (это не клиентская сессия).
- В транзакции: `slots.status = 'cancelled'`, `cancellation_reason = reason`; каскадно все
  связанные `bookings` со `status IN ('active','late_cancel')` → `cancelled_by_studio`
  (`cancelled_at = now()`); место (`free_seats`) не трогать — слот целиком снят, вместимость не
  имеет значения (FR-18: повторная запись на этот слот запрещена всем, слот больше не
  показывается доступным).
- Для каждой каскадно отменённой брони — `SendPush` (FEAT-03) с текстом «Класс [программа]
  [дата] отменён студией. Причина: [reason]» всем `push_subscriptions` клиента.

Готово, когда:
- Ручной вызов endpoint переводит слот и все его активные брони в нужные статусы одной
  транзакцией, отправляет push подписанным клиентам, второй вызов на уже отменённый слот
  возвращает `409` (не дублирует переход).

### FEAT-06. Frontend: реальная push-подписка после `granted`

Сейчас (`BookingSuccessSheet.tsx`) `Notification.requestPermission()` вызывается, но
`PushManager.subscribe()` и передача подписки на сервер — нет (см. `client/README.md` → «Не
реализовано»). Закрывает это.

Сделать:
- После `permission === 'granted'`: `navigator.serviceWorker.ready`, затем
  `registration.pushManager.subscribe({ userVisibleOnly: true, applicationServerKey:

- `pushApi.register(subscription.toJSON())` — новый файл `client/src/api/pushApi.ts`, тонкая
  обёртка по образцу `bookingsApi.ts`.
- Не блокировать закрытие BS-002 сбоем регистрации (LOGIC-004 «Обработка ошибок» — сетевой сбой
  при передаче подписки не блокирует переход на SCR-005), не показывать снек об успехе (логика
  фоновая, без видимого эффекта — буквально по AC из LOGIC-004).

Готово, когда:
- В браузере с разрешённым Notification реальная подписка регистрируется на сервере (видно в
  `push_subscriptions`), при сетевом сбое переход на «Мои записи» всё равно происходит.

### FEAT-07. Frontend: кастомный service worker (push/notificationclick)

Текущий `vite-plugin-pwa` в режиме `generateSW` — только офлайн-кэш, без обработчика `push`.
Чтобы браузер реально показал системное уведомление при получении push, нужен кастомный SW.

Сделать:
- Переключить `vite-plugin-pwa` на стратегию `injectManifest` (собственный `src/sw.ts`,
  precache-манифест инжектится автоматически, офлайн-поведение не теряется).
- В `sw.ts`: `self.addEventListener('push', (event) => { const data = event.data.json();
  event.waitUntil(self.registration.showNotification(data.title, { body: data.body })) })`.
- `self.addEventListener('notificationclick', ...)` — закрыть уведомление и открыть/сфокусировать
  вкладку приложения (`clients.openWindow`/`clients.matchAll`).

Готово, когда:
- Push, отправленный вручную с бэкенда (FEAT-03/04/05), реально показывается системным
  уведомлением браузера с закрытой/свёрнутой вкладкой приложения.

### FEAT-09. Frontend: поиск по шефу/программе на SCR-002

Новая фича, не в исходном ТЗ — решения реализации:
- Чисто клиентский текстовый фильтр по уже загруженному `listSlots`-ответу (без пагинации,
  `listSlots` и так возвращает весь период целиком) — **без изменения контракта/бэкенда**.
- Поле поиска — над списком слотов на `SlotsScreen`, фильтрует по вхождению подстроки (без учёта
  регистра) в `slot.program.name` **или** `slot.chef.name`.
- Пустой результат поиска — не путать с Empty-состоянием LOGIC-005 (это не пустой ответ сервера, а
  локальная фильтрация): отдельная короткая подсказка «Ничего не найдено» внутри `content`-ветки,
  Content/Empty/Error-паттерн самого запроса не трогать.
- Сброс поиска — крестик в поле или очистка вручную; не персистентный (не сохраняется между
  открытиями экрана).

Готово, когда:
- Ввод части имени шефа/программы сужает список без нового запроса к API; очистка поля
  возвращает полный список; `tsc --noEmit`/`npm run build` проходят.
