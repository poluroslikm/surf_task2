## Обзор

Шторка над [SCR-002 «Список слотов»](SCR-002-slot-list.md), позволяющая клиенту расширить период
просмотра слотов за пределы дефолтных 7 дней. Содержит один параметр — верхнюю границу периода
(«Показать по [дата]»); нижняя граница периода не редактируется клиентом и всегда равна текущему
моменту (FR-3). Сама шторка не выполняет запросов к API — она только собирает значение и передаёт
его на SCR-002, который перезагружает список через [LOGIC-003](09_Логики/LOGIC-003_Фильтрация-слотов.md).

### User Story

> Как клиент, я хочу расширить период просмотра слотов фильтром дат,
> чтобы посмотреть расписание на будущее дальше 7 дней (US-3).

### Бизнес-ценность

- Клиент находит и планирует классы за пределами ближайшей недели, не пролистывая вручную длинный
  список слотов, объединяющий несколько недель расписания.
- Снижает риск того, что клиент решит, будто подходящих классов нет, только потому что не увидел
  их в дефолтном 7-дневном окне — поддерживает цель проекта «клиент сам записывается без путаницы»
  (BR-1, BR-2).

## Входные данные

| Название | Тип | Возможные значения | Описание |
|----------|-----|---------------------|----------|
| `dateTo` | Навигационный параметр | ISO-дата (`YYYY-MM-DD`) | Действующая на момент открытия шторки верхняя граница периода: конец дефолтного 7-дневного окна, если клиент ещё не применял фильтр, либо ранее выбранная клиентом дата, если фильтр уже был применён. Используется для предзаполнения поля «Показать по» (design-brief §6.1). |
| `isFilterActive` | Навигационный параметр | `true`, `false` | Признак того, действует ли сейчас нестандартный (расширенный) период. Определяет текст пояснения над полем — «Сейчас: ближайшие 7 дней» (`false`) или «Сейчас: по [дата]» (`true`, дата из `dateTo`). Не влияет на логику запроса — только на отображаемый текст. |

## Используемые запросы

> Все API-запросы экрана — по контракту [`../api/`](../api/). Общий паттерн обработки ответа
> (Loading/Content/Empty/Error) — [00-foundations.md §5](../3-design-brief/00-foundations.md#5-сквозной-паттерн-состояний-экрана),
> здесь не повторяется — указывается только специфика конкретного запроса.

BS-001 **не выполняет собственных запросов к API** (design-brief §7): она только собирает
значение верхней границы периода и закрывается. Фактический запрос `listSlots` выполняет
[SCR-002](SCR-002-slot-list.md) после закрытия шторки, по логике
[LOGIC-003](09_Логики/LOGIC-003_Фильтрация-слотов.md) — состав параметров и обработка ответа
(Content/Empty/Error) описаны там и в документе SCR-002, здесь не дублируются. Ниже — только
определение параметра, который формирует эта шторка, по контракту API.

### listSlots (выполняется на SCR-002, не в BS-001)

**Метод:** GET
**Спецификация:** [`../api/slots/api.yaml`](../api/slots/api.yaml) → `listSlots`
**Триггер:** Тап «Применить» или «Сбросить» в BS-001 закрывает шторку и запускает перезагрузку
списка на SCR-002 (запрос выполняется уже после закрытия шторки, не во время её показа).

**Параметры:**

| Параметр | Тип | Обязательность | Источник | Описание |
|----------|-----|-----------------|----------|----------|
| `date_to` | string (`date-time`) | Нет | Значение поля «Показать по» на момент тапа «Применить» | Верхняя граница периода — слоты со стартом не позже указанного момента (включительно). При тапе «Сбросить» параметр не передаётся — действует дефолт `date_from + 7 дней` (FR-3, R-027, `api/common/models.yaml` → `DateToParam`). |

> Параметр `date_from` этим запросом также поддерживается контрактом `listSlots`, но не
> редактируется в BS-001 — нижняя граница периода всегда равна текущему моменту и не входит в
> зону ответственности этой шторки (FR-3, design-brief §6.1, п.11.1).

**Обработка ответа:**

| Результат | Условие | UI-реакция |
|-----------|---------|------------|
| Успех / Empty / Error | Любой исход запроса `listSlots` после «Применить»/«Сбросить» | К этому моменту BS-001 уже закрыта и не отображает результат сама. Обрабатывается на SCR-002 по сквозному паттерну [00-foundations §5](../3-design-brief/00-foundations.md#5-сквозной-паттерн-состояний-экрана): Content — обновлённый список; Empty — «Пока нет доступных классов» (UC-3 E1, FR-5); Error — состояние ошибки с кнопкой «Обновить» (UC-3 E2). Снек об успехе не показывается — обратная связь через сам обновлённый список (00-foundations §6.1). |

## Состояния экрана

Базовый паттерн — [00-foundations.md §5](../3-design-brief/00-foundations.md#5-сквозной-паттерн-состояний-экрана).
Специфика этого экрана:

BS-001 не выполняет собственных запросов к API (см. «Используемые запросы»), поэтому сквозной
паттерн Loading/Content/Empty/Error к самой шторке не применяется напрямую (design-brief §7):

| Состояние | Условие | Отображение |
|-----------|---------|--------------|
| Loading | Не применимо | У шторки нет собственного запроса — поле открывается сразу предзаполненным известным значением (`dateTo`), без задержки загрузки |
| Content («готова к вводу») | Всегда, сразу при открытии шторки | Заголовок, текст текущего состояния, предзаполненное поле «Показать по», кнопки «Сбросить» и «Применить» |
| Empty | Не применимо к самой шторке | Пустой результат периода — состояние SCR-002 (UC-3 E1, FR-5), не BS-001 |
| Error | Не применимо к самой шторке | Сетевой сбой при перезагрузке списка — состояние SCR-002 (UC-3 E2), не BS-001 |

## Критерии приёмки

### Позитивные

| ID | Критерий |
|----|----------|
| AC-001 | **Дано** клиент на SCR-002 видит дефолтный список (ближайшие 7 дней), фильтр ранее не применялся, **Когда** клиент открывает BS-001 по иконке фильтра, **Тогда** поле «Показать по» предзаполнено датой конца дефолтного 7-дневного окна, и над полем отображается текст «Сейчас: ближайшие 7 дней» |
| AC-002 | **Дано** клиент открыл BS-001, **Когда** клиент выбирает дату дальше дефолтных 7 дней и нажимает «Применить», **Тогда** шторка закрывается, SCR-002 показывает список слотов до выбранной даты, и иконка фильтра в хедере SCR-002 переходит в активное состояние |
| AC-003 | **Дано** на SCR-002 ранее был применён расширенный период, **Когда** клиент открывает BS-001 и нажимает «Сбросить», **Тогда** шторка закрывается, SCR-002 показывает дефолтный список (ближайшие 7 дней), и иконка фильтра возвращается в обычное состояние |
| AC-004 | **Дано** на SCR-002 действует период «по 20 июля», **Когда** клиент повторно открывает BS-001, **Тогда** поле «Показать по» предзаполнено значением «20 июля», а не датой конца дефолтного окна |

### Негативные

| ID | Критерий |
|----|----------|
| AC-N01 | **Дано** клиент открыл BS-001 и изменил значение поля, не нажав «Применить», **Когда** клиент закрывает шторку тапом по бэкдропу, свайпом вниз или кнопкой ✕, **Тогда** список на SCR-002 остаётся в состоянии, которое было до открытия шторки, и новое значение поля не применяется |
| AC-N02 | **Дано** клиент нажал «Применить» в BS-001, **Когда** запрос `listSlots` на SCR-002 завершается сетевой ошибкой, **Тогда** SCR-002 показывает состояние ошибки с кнопкой «Обновить», и это поведение принадлежит SCR-002, а не BS-001 |

### Граничные

| ID | Критерий |
|----|----------|
| AC-E01 | **Дано** клиент открыл поле «Показать по», **Когда** клиент пытается выбрать дату раньше сегодняшней, **Тогда** такие даты недоступны для выбора в компоненте, и отдельное сообщение об ошибке не показывается |
| AC-E02 | **Дано** клиент выбрал в BS-001 дату, на которую в расписании нет ни одного слота, **Когда** клиент нажимает «Применить», **Тогда** шторка закрывается, и SCR-002 показывает empty state «Пока нет доступных классов» |
