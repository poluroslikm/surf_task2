## Обзор

Шторка критичного подтверждения деструктивного действия — **отмены записи клиентом**. Открывается
с [SCR-006 «Детали брони»](SCR-006-booking-details.md) по кнопке «Отменить» и требует явного,
осознанного подтверждения, прежде чем запрос на отмену будет отправлен в API. В зависимости от
времени до начала класса шторка показывает один из двух вариантов содержимого: без предупреждений
(ранняя отмена, ≥ 24 ч, FR-15) или с предупреждением о закупленных продуктах (поздняя отмена,
< 24 ч, FR-16). Предварительный выбор варианта делает клиент, ориентируясь на уже загруженное поле
`booking.free_cancellation_until`; какой вариант в итоге зафиксирован как результат — определяет
только ответ `cancelBooking` (LOGIC-002).

### User Story

> Как клиент, я хочу перед отменой записи увидеть явное подтверждение (и предупреждение о
> закупленных продуктах, если отменяю позже чем за 24 часа до начала),
> чтобы не отменить запись случайным тапом и не столкнуться с неожиданным последствием уже
> после того, как отмена стала необратимой.

### Бизнес-ценность

- Снижает число случайных отмен (защита от промаха на критичном необратимом действии),
  сохраняя заполняемость слотов для студии.
- При поздней отмене клиент видит предупреждение о закупленных продуктах **до**, а не после
  необратимого действия — меньше недоумения и обращений в поддержку по поводу того, что отмена
  всё равно засчиталась как поздняя.
- Спокойный тон без штрафов и давления (P6, FR-16) поддерживает доверие клиента к сервису даже
  в момент отказа от участия.

## Свойства Bottom Sheet

> *Секция только для типа "Bottom Sheet" — общие правила см.
> [00-foundations.md §4.3](../3-design-brief/00-foundations.md#43-bottom-sheet-шторки-bs-001bs-004).*

| Свойство | Значение |
|----------|----------|
| Высота | Динамическая (по контенту): грабер + заголовок + (опционально) блок предупреждения + кнопки, не более ~90% экрана. Вариант «поздняя отмена» выше варианта «ранняя» за счёт блока предупреждения. |
| Закрытие свайпом | Нет — критичное подтверждение деструктивного действия (00-foundations.md §4.3 прямо называет отмену записи исключением из общего правила свайп-закрытия). |
| Закрытие по тапу вне области | Нет — критичное подтверждение (00-foundations.md §4.3); закрытие шторки возможно только явной кнопкой. |
| Кнопка закрытия | Да — крестик в правом верхнем углу шторки. Действует как отказ: равнозначна кнопке «Не отменять» (шторка закрывается, статус записи не меняется, запрос к API не отправляется). |

> **Уточнение расхождения дизайн-брифа.** Раздел 9 и сценарий AC-003 черновика
> [3-design-brief/BS-003-cancel-confirm.md](../3-design-brief/BS-003-cancel-confirm.md) упоминают
> тап вне шторки и свайп вниз как способы закрыть шторку наравне с кнопками. Это противоречит
> собственному §3 того же документа и сквозному правилу
> [00-foundations.md §4.3](../3-design-brief/00-foundations.md#43-bottom-sheet-шторки-bs-001bs-004),
> которое явно исключает отмену записи из круга шторок с закрытием по бэкдропу/свайпу. Настоящее
> финальное ТЗ разрешает расхождение в пользу более строгого и явно сформулированного правила
> foundations: для BS-003 закрытие доступно **только** кнопками «Не отменять» и крестик; тап по
> бэкдропу и swipe-вниз для этой шторки не закрывают её вовсе (ни как подтверждение, ни как
> отказ) — см. также AC-N03.

## Используемые запросы

> Все API-запросы экрана — по контракту [`../api/`](../api/). Общий паттерн обработки ответа
> (Loading/Content/Empty/Error) — [00-foundations.md §5](../3-design-brief/00-foundations.md#5-сквозной-паттерн-состояний-экрана),
> здесь не повторяется — указывается только специфика конкретного запроса.
>
> Собственной загрузки данных при открытии у шторки нет: `booking` (включая
> `free_cancellation_until`) уже получен на SCR-006 через `getBooking` до открытия BS-003.

### cancelBooking

**Метод:** POST
**Спецификация:** [`../api/bookings/api.yaml`](../api/bookings/api.yaml) → `cancelBooking`
**Триггер:** Тап на кнопку «Отменить запись»

**Параметры:**

| Параметр | Тип | Обязательность | Источник | Описание |
|----------|-----|-----------------|----------|----------|
| `bookingId` | string (uuid) | Да | Path-параметр, значение — входной параметр шторки `bookingId` | Идентификатор отменяемой записи. Тело запроса отсутствует — отмена всегда целиком (одна запись = одно место). |

**Обработка ответа:**

| Результат | Условие | UI-реакция |
|-----------|---------|------------|
| Успех — ранняя отмена | `200`, `status = cancelled` | Кнопка выходит из состояния загрузки, шторка закрывается; SCR-006 получает обновлённый `booking` и показывает бейдж «Отменена» + снек «Запись отменена» (00-foundations.md §6.1, показывает экран-родитель §6.2). |
| Успех — поздняя отмена | `200`, `status = late_cancel` | Кнопка выходит из состояния загрузки, шторка закрывается; SCR-006 показывает бейдж «Отменена» + снек «Поздняя отмена: продукты на класс уже закуплены. Штраф не взимается.» (00-foundations.md §6.1). |
| Класс уже начался (гонка состояния) | `422`, `code = slot_started` | Шторка закрывается (повтор бессмысленен — операция более невозможна), статус записи не меняется; SCR-006 обновляет данные и показывает пояснение о недоступности отмены (аналог текста SCR-006 для disabled-состояния кнопки — «Класс уже начался, отменить запись нельзя», LOGIC-002 → «Обработка ошибок»). Обычно предотвращается заранее disabled-состоянием кнопки «Отменить» на SCR-006 (UC-2 E1); здесь — защита от гонки между открытием шторки и тапом подтверждения. |
| Уже отменена (гонка состояния) | `409`, `code = already_cancelled` | Шторка закрывается без сообщения об ошибке — результат («запись отменена») уже достигнут; SCR-006 обновляет данные и показывает текущий (уже отменённый) статус записи (UC-2 E2). |
| Токен истёк | `401` | Переход на экран входа [SCR-001](SCR-001-auth.md) — сквозное правило (`api-sequence.md`); отмена не считается выполненной. |
| Сетевой сбой / сервер недоступен | Нет ответа, `5xx` или таймаут | Шторка **остаётся открытой** (UC-2 E3), статус записи не меняется. Сама шторка (не SCR-006) показывает снек ошибки: «Не удалось выполнить. Проверьте соединение и повторите.» (00-foundations.md §6, §6.2). Кнопка «Отменить запись» разблокируется, повтор — тем же тапом. |

## Элементы экрана

> **Колонка «Валидация»** — для полей ввода правило и текст ошибки, для остального — «—».
> **Условия доступности** — для интерактивных элементов указываются после таблицы блока.

### 1. Шапка шторки

| Элемент | Описание | Источник данных | Валидация | Действие |
|---------|----------|------------------|-----------|----------|
| Грабер | Визуальный индикатор шторки сверху; чисто декоративный — свайп-закрытие для BS-003 отключён (см. «Свойства Bottom Sheet») | — | — | — |
| Кнопка-крестик | Иконка закрытия в правом верхнем углу | — | — | Тап → шторка закрывается, статус записи не меняется, запрос к API не отправляется (равнозначно «Не отменять») |
| Заголовок | Текст «Точно отменить запись?» — идентичен для обоих вариантов (ранняя/поздняя) | Статичная микрокопия (дизайн-бриф §8) | — | — |

**Условия доступности:**
- Кнопка-крестик активна всегда, кроме периода ожидания ответа `cancelBooking` (см. блок 3, «Условия доступности»).

### 2. Блок предупреждения (только вариант «поздняя отмена»)

| Элемент | Описание | Источник данных | Валидация | Действие |
|---------|----------|------------------|-----------|----------|
| Блок предупреждения | Дословная цитата строки «Правило отмены» из [00-foundations.md §6](../3-design-brief/00-foundations.md#6-tone-of-voice-и-общая-микрокопия): «Отмена не позднее чем за 24 часа до начала — без предупреждений, место освобождается. Позже — запись всё равно отменится, но продукты на класс уже закуплены. Штраф не взимается.» Показывается между заголовком и кнопками, так что клиент физически не может подтвердить, не увидев его. | Текст — статичная микрокопия; условие показа блока — сравнение текущего момента с `booking.free_cancellation_until` (LOGIC-002, предварительная оценка) | — | — |

**Условия доступности:**
- Блок показывается, если на момент открытия шторки текущий момент устройства ≥ `booking.free_cancellation_until` (вариант «поздняя отмена», §5.2 дизайн-брифа); в варианте «ранняя отмена» блок не рендерится вовсе (не просто скрыт визуально).

### 3. Кнопки действия

| Элемент | Описание | Источник данных | Валидация | Действие |
|---------|----------|------------------|-----------|----------|
| Кнопка «Отменить запись» | Деструктивное действие, визуально менее акцентно относительно кнопки «Не отменять», но всегда доступна для тапа независимо от варианта (FR-16 требует, что поздняя отмена выполняется, а не блокируется) | — | — | Тап → `POST cancelBooking`; кнопка переходит в состояние загрузки |
| Кнопка «Не отменять» | Безопасное, основное/акцентное действие — отказ от намерения отменить | — | — | Тап → шторка закрывается, статус записи не меняется, запрос к API не отправляется |

**Условия доступности:**
- Кнопка «Отменить запись» активна с момента открытия шторки в обоих вариантах; после тапа переходит в состояние загрузки/блокировки до ответа сервера — повторный тап исключён (защита от двойной отправки).
- Кнопка «Не отменять» и крестик активны с момента открытия шторки; на время ожидания ответа `cancelBooking` (после тапа «Отменить запись») блокируются вместе с остальными элементами шторки, до получения ответа API.

## Состояния экрана

Базовый паттерн — [00-foundations.md §5](../3-design-brief/00-foundations.md#5-сквозной-паттерн-состояний-экрана).
Специфика этого экрана: BS-003 — шторка действия, а не загрузки данных, поэтому классических
Loading/Empty при открытии у неё нет (все нужные данные, включая `free_cancellation_until`, уже
получены на SCR-006 до открытия шторки).

| Состояние | Условие | Отображение |
|-----------|---------|--------------|
| Loading | Не применимо — собственного запроса данных при открытии шторки нет | Не показывается |
| Content — ранняя отмена | Текущий момент < `booking.free_cancellation_until` (предварительная оценка, LOGIC-002) | Грабер + крестик + заголовок «Точно отменить запись?» + кнопки «Отменить запись» / «Не отменять», без блока предупреждения |
| Content — поздняя отмена | Текущий момент ≥ `booking.free_cancellation_until` (предварительная оценка, LOGIC-002) | Грабер + крестик + заголовок + блок предупреждения о закупленных продуктах + кнопки |
| Выполнение отмены | После тапа «Отменить запись», до получения ответа `cancelBooking` | Кнопка подтверждения в состоянии загрузки; кнопка «Не отменять» и крестик заблокированы, повторный тап по любому элементу исключён |
| Empty | Не применимо — шторка не отображает список или коллекцию данных, которая может быть пустой | — |
| Error | Ответ `cancelBooking` завершился сетевым сбоем/5xx/таймаутом (UC-2 E3) | Шторка остаётся открытой, статус записи не меняется; сама шторка показывает снек «Не удалось выполнить. Проверьте соединение и повторите.»; кнопка «Отменить запись» разблокирована для повторного тапа (кнопки «Обновить» здесь нет — это не загрузка данных, а повторяемое действие) |

| Действие | Элемент | Триггер | Результат |
|----------|---------|---------|-----------|

## Критерии приёмки

### Позитивные

| ID | Критерий |
|----|----------|
| AC-001 | **Дано** текущий момент раньше `booking.free_cancellation_until`, **Когда** клиент открывает шторку подтверждения отмены, **Тогда** показан заголовок «Точно отменить запись?» без блока предупреждения, доступны обе кнопки — «Отменить запись» и «Не отменять» |
| AC-002 | **Дано** текущий момент не раньше `booking.free_cancellation_until`, **Когда** клиент открывает шторку подтверждения отмены, **Тогда** показан тот же заголовок и блок предупреждения (дословный текст «Правила отмены» из foundations §6) между заголовком и кнопками, с явным указанием, что штраф не взимается |
| AC-003 | **Дано** шторка открыта в любом варианте, **Когда** клиент нажимает «Отменить запись» и сервер отвечает `200 {status: cancelled}`, **Тогда** кнопка переходит в состояние загрузки без возможности повторного тапа, при успехе шторка закрывается, клиент возвращается на SCR-006, который показывает обновлённый статус записи и снек «Запись отменена» |
| AC-004 | **Дано** шторка открыта в любом варианте, **Когда** клиент нажимает «Отменить запись» и сервер отвечает `200 {status: late_cancel}`, **Тогда** кнопка переходит в состояние загрузки без возможности повторного тапа, при успехе шторка закрывается, клиент возвращается на SCR-006, который показывает обновлённый статус записи и снек «Поздняя отмена: продукты на класс уже закуплены. Штраф не взимается.» |
| AC-005 | **Дано** шторка подтверждения отмены открыта (любой вариант), **Когда** клиент нажимает «Не отменять» или крестик, **Тогда** шторка закрывается без отправки запроса к API, статус записи не изменяется, снек не показывается |

### Негативные

| ID | Критерий |
|----|----------|
| AC-N01 | **Дано** клиент нажал «Отменить запись», **Когда** запрос `cancelBooking` завершается сетевой ошибкой, сервером недоступен или таймаутом, **Тогда** шторка остаётся открытой, статус записи не меняется, сама шторка показывает снек «Не удалось выполнить. Проверьте соединение и повторите.», кнопка «Отменить запись» доступна для повторного тапа |
| AC-N02 | **Дано** клиент нажал «Отменить запись», **Когда** сервер отвечает `401 Unauthorized` (истёкший токен), **Тогда** приложение переходит на экран входа [SCR-001](SCR-001-auth.md), отмена не считается выполненной |
| AC-N03 | **Дано** шторка подтверждения отмены открыта, **Когда** клиент тапает по области вне шторки или пытается закрыть её свайпом вниз, **Тогда** шторка не закрывается этим жестом — она может быть закрыта только тапом по «Не отменять» или крестику (00-foundations.md §4.3, критичное подтверждение) |

### Граничные

| ID | Критерий |
|----|----------|
| AC-E01 | **Дано** клиент нажал «Отменить запись», **Когда** к моменту обработки запроса сервер определяет, что класс уже начался (`422 {code: slot_started}` — гонка между открытием шторки и подтверждением), **Тогда** шторка закрывается, статус записи не меняется, SCR-006 обновляет данные и показывает клиенту, что отмена недоступна, так как класс уже начался |
| AC-E02 | **Дано** клиент нажал «Отменить запись», **Когда** к моменту обработки запроса запись уже отменена (`409 {code: already_cancelled}` — например, отменена с другого устройства между открытием шторки и подтверждением), **Тогда** шторка закрывается без сообщения об ошибке, SCR-006 обновляет данные и показывает клиенту текущий (уже отменённый) статус записи |
| AC-E03 | **Дано** предварительный вариант шторки (ранняя/поздняя), выбранный по локально сравненному `booking.free_cancellation_until`, разошёлся с фактическим результатом сервера (например, из-за рассинхронизации часов устройства или отмены точно на границе порога), **Когда** клиент подтверждает отмену, **Тогда** итоговый статус записи и текст снека на SCR-006 определяются исключительно полем `status` ответа `cancelBooking`, а не тем вариантом содержимого, который был показан в шторке до подтверждения |
