## 1. Назначение и контекст

SCR-001 — единственный экран неавторизованной зоны и точка входа в приложение: без учётной
записи клиент не может увидеть слоты и записаться (FR-3–FR-23 требуют авторизованного клиента,
см. `02-domain.md` → «Пользователи и роли»). Экран закрывает FR-1 (регистрация по email и
паролю) и FR-2 (авторизация по email и паролю), реализуя US-1 «зарегистрироваться и войти по
email и паролю, чтобы получить доступ к записи на классы».

Учётная запись здесь — не самоцель, а минимально необходимый механизм разграничения данных:
NFR-8 требует, чтобы клиент видел и управлял только своими записями и оценками — это возможно
только при наличии идентифицированной сессии, которую создаёт именно этот экран.

Экран не содержит анкеты, профиля или дополнительных полей (имя, телефон) — функциональные
требования описывают авторизацию **только по email и паролю** (FR-1, FR-2); поле для других
данных в требованиях отсутствует (см. также `00-foundations.md` §1, `design-brief.md` — «Почему
не 1-в-1 как в примере»: профиля как раздела в продукте нет).

## 2. Пользователь и цель

- **Роль:** Клиент — единственная роль приложения ([foundations §1](00-foundations.md#1-продукт-и-аудитория)).
- **Контекст использования:** два типичных случая —
  1. **Новый клиент** впервые открывает PWA (по ссылке от студии/рекомендации) и ещё не имеет
     учётной записи.
  2. **Существующий клиент** открывает приложение снова: после выхода из аккаунта через
     [BS-004 «Аккаунт»](BS-004-account.md), после переустановки/очистки данных браузера или на
     новом устройстве.

  В обоих случаях — mobile-first сценарий (NFR-1): с телефона, часто на бегу, до или в процессе
  выбора класса.
- **Цель пользователя:** получить доступ к записи на класс максимально быстро, минимумом полей,
  без лишних шагов (в духе P2 [foundations §2](00-foundations.md#2-дизайн-принципы), хотя P2
  сформулирован про NFR-2/запись — по аналогии применяем к входу: чем короче путь до
  списка слотов, тем лучше).
- **Цель системы:** создать или подтвердить учётную запись клиента и открыть авторизованную
  сессию, защищённую от доступа третьих лиц (NFR-7), обеспечивающую разграничение данных между
  клиентами (NFR-8).

## 3. Навигация

### Входящие
- Нет входящих переходов внутри приложения для **нового** посетителя — это первый экран,
  открывающийся при заходе в PWA без активной сессии (условие задания; в картах навигации
  ([foundations §4.4](00-foundations.md#44-карта-навигации-сквозная)) SCR-001 — корень графа).
- Из шторки [BS-004 «Аккаунт»](BS-004-account.md) при выходе из аккаунта (`logout`) — клиент
  возвращается именно сюда ([foundations §4.4](00-foundations.md#44-карта-навигации-сквозная),
  строка «Хедер SCR-002/SCR-005 → иконка аккаунта → BS-004 (выход)»).

### Исходящие
- **Успешный вход или успешная регистрация** → [SCR-002 «Классы»](SCR-002-slot-list.md) —
  стартовая вкладка таб-бара ([foundations §4.2](00-foundations.md#42-таб-бар-авторизованная-зона),
  [§4.4](00-foundations.md#44-карта-навигации-сквозная)). Отдельного экрана-приветствия или
  онбординга нет — ни FR, ни US его не описывают.
- Других исходящих переходов нет: на SCR-001 нет ссылок на внешние страницы, восстановление
  пароля или иные экраны (см. «Решения», п. 2).

Таб-бар на этом экране отсутствует — он определён только для авторизованной зоны
([foundations §4.2](00-foundations.md#42-таб-бар-авторизованная-зона)).

Каркас — по [foundations §4.1](00-foundations.md#4-каркас-экрана-и-навигация): фикс. хедер (без
кнопки «назад» — экран корневой) → скролл-контент → фикс. нижний CTA. Сверху вниз:

1. **Заголовок/лого студии** — название «Шеф-стол», без хедера с кнопкой назад (некуда
   возвращаться).
2. **Переключатель вкладок «Вход» / «Регистрация»** (см. «Решения», п. 1 и п. 5 — вкладка
   «Вход» активна по умолчанию).
3. **Поле Email.**
4. **Поле Пароль** (с переключателем видимости).
5. **Область сообщения об ошибке** (появляется только при ошибке — E1–E6, §8), не занимает
   место в Content-состоянии.
6. **Фикс. нижний CTA** — «Войти» или «Зарегистрироваться» в зависимости от вкладки.

## 6. Компоненты и поведение

### 6.1 Переключатель вкладок «Вход» / «Регистрация»

- Два таба, взаимоисключающие, стандартная тач-зона ≥ 44–48 pt
  ([foundations §3.1](00-foundations.md#31-тач-зоны-и-размеры)).
- Переключение вкладки **не отправляет форму** и не перезагружает экран — только меняет:
  подпись CTA, применимость проверки E2 (минимальная длина пароля — действует только на
  «Регистрации», см. §8), и текст сообщения E-неверные-данные при ошибке сервера (см. §8, E3/E4).
- При переключении вкладки поле `email` сохраняет введённое значение; поле `password`
  **очищается** — решение дизайнера, чтобы не переносить пароль по ошибке между сценариями входа
  и регистрации (см. «Решения», п. 6).

### 6.2 Поле Email

- Стандартный ввод с клавиатурой типа email на мобильных устройствах (атрибут `type=email`,
  `autocomplete=username`) — общая гигиена мобильной формы, не отдельное требование.
- Валидация формата — на клиенте, по потере фокуса (blur) и повторно при отправке: см. E1 (§8).

### 6.3 Поле Пароль

- Ввод скрыт по умолчанию (`type=password`), с иконкой-переключателем видимости
  (`aria-label`: «Показать пароль» / «Скрыть пароль»).
- На вкладке «Регистрация» — проверка минимальной длины по потере фокуса и при отправке: см. E2
  (§8, «Решения» п. 4). На вкладке «Вход» клиентская проверка длины **не выполняется** —
  историческая учётная запись могла быть создана до изменения политики пароля на сервере, поэтому
  для входа решает только ответ сервера (E3).

### 6.4 Кнопка CTA («Войти» / «Зарегистрироваться»)

- Во всю ширину контентной области, всегда видна над клавиатурой
  ([foundations §4.1](00-foundations.md#4-каркас-экрана-и-навигация)).
- **Disabled**, пока `email` или `password` пусты — до отправки, без обращения к серверу.
- **Enabled**, когда оба поля заполнены (финальную валидацию формата/сложности и существования
  учётной записи выполняет сервер).
- **Состояние отправки (loading):** по тапу — индикатор загрузки на кнопке, поля `email` и
  `password` блокируются от редактирования, повторный тап невозможен (защита от повторной
  отправки той же формы — общая гигиена форм, аналогично защите от двойного тапа на
  [SCR-004 §6.4](SCR-004-booking.md#64-кнопка-записаться-нижний-cta), но без отдельного FR/NFR
  именно для этого экрана).

## 7. Состояния экрана

Экран не выполняет GET-запрос для собственного отображения — форма статична и не требует
загрузки данных, поэтому полный паттерн Loading/Content/Empty/Error из
[foundations §5](00-foundations.md#5-сквозной-паттерн-состояний-экрана) к SCR-001 **не
применяется целиком**: нет состояний Loading/Empty при открытии экрана. Специфичные состояния
экрана:

| Состояние | Содержимое / поведение |
|---|---|
| **Content** (по умолчанию) | Пустая форма, активна вкладка «Вход» (см. «Решения», п. 5), CTA disabled до заполнения обоих полей. |
| **Ввод** | Клиент печатает; inline-валидация формата email и (на «Регистрации») длины пароля — по blur, не на каждое нажатие клавиши (решение дизайнера — не мешать вводу). |
| **Отправка (submitting)** | После тапа CTA — индикатор загрузки на кнопке, поля заблокированы, повторный тап невозможен (§6.4). |
| **Ошибка** | Сообщение под полями по кодам E1–E6 (§8); введённый `email` сохраняется, `password` очищается при ошибке сервера (E3, E4) — см. «Решения», п. 6. |
| **Успех** | Немедленный переход на [SCR-002](SCR-002-slot-list.md); отдельного экрана/снека успеха нет (см. «Решения», п. 7). |

## 8. Валидации и микрокопия

> Тон — простой и без вины ([foundations §6](00-foundations.md#6-tone-of-voice-и-общая-микрокопия)).
> Поскольку выделенного UC на этот экран нет (см. врезку в источниках), коды ниже — не потоки
> UC-исключений, а состояния ошибок, выведенные из FR-1/FR-2/US-1 и общих сквозных текстов
> ошибок [foundations §6](00-foundations.md#6-tone-of-voice-и-общая-микрокопия). Где текст не
> зафиксирован сквозным документом — это **допущение дизайнера**, помечено отдельно.

| Код | Условие | Текст | Поведение UI |
|---|---|---|---|
| **E1** | Некорректный формат email (клиентская проверка) | **«Введите корректный email»** *(допущение дизайнера)* | Inline-ошибка под полем email по blur/при отправке; CTA заблокирован, пока формат не исправлен. |
| **E2** | Пароль короче минимума **на вкладке «Регистрация»** (клиентская проверка) | **«Пароль должен быть не короче 8 символов»** *(минимум — допущение дизайнера, см. «Решения» п. 4)* | Inline-ошибка под полем пароля; применяется только при `activeTab = Регистрация` (§6.3). |
| **E3** | Неверный email или пароль при входе (ответ сервера, FR-2) | **«Неверный email или пароль»** *(намеренно не уточняется, какое из полей неверно — предотвращает подбор существующих email; допущение дизайнера в духе NFR-7)* | Сообщение над CTA; поле `password` очищается, `email` сохраняется (§6.1, «Решения» п. 6); фокус — на поле пароля. |
| **E4** | Email уже зарегистрирован при регистрации (ответ сервера, FR-1) | **«Этот email уже зарегистрирован. Войдите или используйте другой email.»** | Сообщение над CTA; предлагает переключиться на вкладку «Вход» (не выполняется автоматически — клиент решает сам). |
| **E5** | Сетевой сбой при отправке формы (вход или регистрация) | **«Не удалось выполнить. Проверьте соединение и повторите.»** ([foundations §6](00-foundations.md#6-tone-of-voice-и-общая-микрокопия)) | Сообщение над CTA с возможностью повторить; введённые `email`/`password` сохраняются (сеть — не ошибка данных). |
| **E6** | Ошибка сервера (5xx) | **«Что-то пошло не так. Попробуйте ещё раз позже.»** ([foundations §6](00-foundations.md#6-tone-of-voice-и-общая-микрокопия)) | Сообщение над CTA; форма остаётся заполненной, повторная отправка доступна. |

**Запреты:** CTA не отправляет форму, пока `email`/`password` пусты (§6.4); дублирующая отправка
во время `submitting` заблокирована (§6.4, §7).

## 9. Доступность и mobile-first

База — [foundations §3, §7](00-foundations.md#7-доступность). Специфика экрана:

- Оба поля формы имеют видимые лейблы, программно связанные с инпутами (`<label for=...>`),
  сообщения об ошибках связаны с полем через `aria-describedby` — не полагаемся только на цвет
  рамки ([foundations §3.2](00-foundations.md#32-контраст-и-читаемость-nfr-1)).
- Переключатель видимости пароля и вкладки «Вход/Регистрация» имеют доступные имена для
  screen reader («Показать пароль», «Вкладка: Вход», «Вкладка: Регистрация»).
- Тач-зоны вкладок, полей и CTA — ≥ 44–48 pt
  ([foundations §3.1](00-foundations.md#31-тач-зоны-и-размеры)).
- Поле пароля — атрибуты `autocomplete=current-password` (вкладка «Вход») /
  `autocomplete=new-password` (вкладка «Регистрация») для корректной работы автозаполнения и
  менеджеров паролей браузера — общая гигиена мобильной PWA-формы.
- Порядок фокуса: вкладки → email → пароль → переключатель видимости → CTA.
- Layout не ломается при системном увеличении шрифта; CTA не перекрывается клавиатурой
  ([foundations §4.1](00-foundations.md#4-каркас-экрана-и-навигация)).

## 10. Критерии приёмки дизайна (Gherkin)

### Позитивные

```gherkin
AC-001: Успешный вход
  Дано клиент на вкладке «Вход» с существующей учётной записью
  Когда он вводит верные email и пароль и нажимает «Войти»
  Тогда кнопка переходит в состояние загрузки
  И по успешному ответу сервера происходит переход на SCR-002

AC-002: Успешная регистрация
  Дано клиент на вкладке «Регистрация», email ещё не зарегистрирован
  Когда он вводит email и пароль длиной 8 и более символов и нажимает «Зарегистрироваться»
  Тогда кнопка переходит в состояние загрузки
  И по успешному ответу сервера происходит переход на SCR-002

AC-003: Переключение вкладок сохраняет email
  Дано клиент ввёл email на вкладке «Вход»
  Когда он переключается на вкладку «Регистрация»
  Тогда email остаётся заполненным
  И поле пароля пустое (§6.1)
```

### Негативные

```gherkin
AC-004: E1 — некорректный формат email
  Дано клиент вводит email без «@» или домена
  Когда поле email теряет фокус или клиент нажимает CTA
  Тогда показывается «Введите корректный email»
  И запрос на сервер не отправляется

AC-005: E2 — слабый пароль при регистрации
  Дано клиент на вкладке «Регистрация» вводит пароль короче 8 символов
  Когда поле пароля теряет фокус или клиент нажимает «Зарегистрироваться»
  Тогда показывается «Пароль должен быть не короче 8 символов»
  И запрос на сервер не отправляется

AC-006: E3 — неверные email или пароль при входе
  Дано клиент на вкладке «Вход» вводит email и пароль, не совпадающие с учётной записью
  Когда сервер отклоняет вход
  Тогда показывается «Неверный email или пароль»
  И поле пароля очищается, email остаётся заполненным
  И сообщение не уточняет, какое из полей неверно

AC-007: E4 — email уже зарегистрирован
  Дано клиент на вкладке «Регистрация» вводит email, который уже зарегистрирован
  Когда сервер отклоняет регистрацию
  Тогда показывается «Этот email уже зарегистрирован. Войдите или используйте другой email.»
  И учётная запись повторно не создаётся

AC-008: E5 — сетевой сбой
  Дано клиент нажал «Войти» или «Зарегистрироваться»
  Когда запрос не доходит до сервера из-за сбоя сети
  Тогда показывается «Не удалось выполнить. Проверьте соединение и повторите.»
  И введённые email и пароль сохраняются в полях

AC-009: Защита от двойного тапа
  Дано CTA в состоянии отправки (submitting)
  Когда клиент повторно тапает по кнопке
  Тогда повторный запрос не отправляется
```

## 11. Решения

1. **Одна форма с двумя вкладками «Вход» / «Регистрация», а не два отдельных экрана.** Оба
   сценария используют идентичный набор полей (FR-1, FR-2 — только email и пароль), отдельные
   экраны увеличили бы число переходов без функциональной пользы. §5, §6.1.
2. **Экран/ссылка «Забыли пароль?» не включены.** Ни FR-1/FR-2, ни NFR-7 не описывают
   восстановление пароля; механизм хранения и передачи учётных данных прямо назван «вне скоупа
   анализа требований» (NFR-7). Реализация восстановления потребовала бы дополнительной
   инфраструктуры (доставка письма/токена сброса), не описанной ни в одном источнике и не
   входящей в контракт API (`02-domain.md` → «Данные существующей инфраструктуры», R-015). Чтобы
   не изобретать функциональность вне требований, ссылка **не включена** в этот экран; это
   **открытый вопрос**, который стоит вернуть заказчику/аналитику до реализации — при отсутствии
   восстановления клиент, забывший пароль, окажется без пути возврата в приложение. §3, §4.
3. **Поле подтверждения пароля на регистрации не добавлено.** FR-1 описывает только email и
   пароль, без повторного ввода; лишнее поле увеличивает число шагов без явного требования. §4.
4. **Минимальная длина пароля — 8 символов, применяется только на «Регистрации».** NFR-7 прямо
   не задаёт политику пароля («остаётся на усмотрение реализации API»), поэтому конкретное число
   — **допущение дизайнера**, взятое как распространённый минимальный порог. Для входа проверка
   не выполняется на клиенте — старые пароли могли быть созданы до появления такого порога;
   финальное решение — за ответом сервера. §6.3, §8 (E2).
5. **По умолчанию активна вкладка «Вход», а не «Регистрация».** Единственный сценарий
   повторного попадания на этот экран изнутри приложения — выход из аккаунта (BS-004), то есть у
   клиента уже есть учётная запись; для первого посещения переключение на «Регистрация» занимает
   один тап. §5, §6, §7 (состояние Content).
6. **При ошибке неверных данных (E3) поле пароля очищается, email — сохраняется.** Снижает риск
   повторной отправки неверного пароля вслепую и не заставляет клиента заново набирать email;
   общепринятая практика форм входа, не отдельное требование. §6.1, §7, §8.
7. **Отдельного экрана/снека успеха для входа и регистрации нет.** Переход на SCR-002 сам по
   себе — очевидная обратная связь (аналогично правилу «не дублировать обратную связь» из
   [foundations §6.2](00-foundations.md#6-tone-of-voice-и-общая-микрокопия) для очевидных
   переходов, например logout). §3, §7.
