# Требования на дизайн · BS-003 · Подтверждение отмены записи

**ID:** BS-003 · **Тип:** Bottom Sheet · **Зона:** АЗ · **Приоритет:** Must · **Статус:** Черновик · **Версия:** 0.1 · **Дата:** 2026-07-02

> Шторка критичного подтверждения деструктивного действия — **отмены записи** клиентом.
> В зависимости от времени до начала класса показывает один из двух вариантов содержимого
> (ранняя / поздняя отмена, порог — 24 часа, FR-15/FR-16) и требует явного подтверждения.
> Сквозные правила (каркас шторок, состояния, микрокопия, доступность) — в
> [`00-foundations.md`](00-foundations.md); здесь **не дублируются**, а уточняются под кейс.

## 1. Назначение и контекст

Шторка вызывается, когда клиент на [SCR-006](SCR-006-booking-details.md) нажимает «Отменить» и
должен осознанно подтвердить необратимое действие (US-10, UC-2 шаги 1–2).

Назначение:
- Дать клиенту **последний шанс передумать** перед отменой (FR-14).
- При **поздней отмене** — явно предупредить о последствии (продукты на класс уже закуплены)
  **до** подтверждения, чтобы клиент подтверждал отмену осознанно (FR-16, US-10, критерий
  приёмки US-10).
- При **ранней отмене** — не создавать лишнего трения: простое подтверждение без
  предупреждений (FR-15).
- Ни в каком варианте не упоминать денежный штраф как последствие — его нет ни при какой
  отмене (FR-16 explicit; принцип P6 foundations — «спокойный тон, без штрафов и давления»).

Кто определяет, какой из двух вариантов показать: **сервер** — источник истины по времени до
начала класса (UC-2, шаг 3: «Система определяет время до начала — источник истины сервер»).
Шторка не пересчитывает порог самостоятельно на клиенте (по аналогии с P3 foundations — «числа
только из API» — то же правило распространяется на серверное время).

## 2. Пользователь и цель

- **Роль:** Клиент (единственная роль приложения).
- **Цель пользователя:** отказаться от участия в классе, если планы изменились, без сюрпризов
  и без ощущения, что его наказывают (US-10).
- **Цель продукта:** при ранней отмене — вернуть место в слот другим клиентам (FR-15); при
  поздней — всё равно выполнить отмену (FR-16 явно требует не блокировать её), но сделать
  предупреждение о закупленных продуктах видимым и осознанным до подтверждения.
- **Задача шторки:** предотвратить случайную отмену (защита от промаха) и, для позднего случая,
  убедиться, что клиент увидел предупреждение прежде, чем оно станет необратимым.

## 3. Навигация

### Входящие
- Кнопка **«Отменить»** на экране [SCR-006 · Детали брони](SCR-006-booking-details.md). Кнопка
  недоступна, если класс уже начался или запись уже отменена (UC-2, исключения E1, E2) — это
  состояние самого SCR-006, к этой шторке отношения не имеет и здесь не переописывается.

### Исходящие
- **Подтверждение** («Отменить запись») → запрос отмены к API → закрытие шторки → возврат на
  [SCR-006](SCR-006-booking-details.md) с обновлённым статусом записи и снеком-итогом. Снек
  показывает **экран-родитель SCR-006**, не шторка (foundations §6.1/§6.2):
  - ранняя отмена — снек «Запись отменена»;
  - поздняя отмена — снек «Поздняя отмена: продукты на класс уже закуплены. Штраф не
    взимается.»
- **«Не отменять»** или закрытие шторки (крестик) → шторка просто закрывается, статус записи не
  меняется, никакого снека не показывается.

### Свойства шторки

Общие правила bottom sheet — [§4.3 foundations](00-foundations.md#43-bottom-sheet-шторки-bs-001bs-004).
Уточнение для BS-003 как **критичного подтверждения**: foundations прямо называет отмену записи
исключением из правила «закрытие по тапу вне шторки» — для BS-003 закрытие возможно **только
явной кнопкой** («Не отменять» или крестик). Тап по бэкдропу и swipe-вниз для этой шторки не
применяются как способ подтвердить действие; деструктивное действие выполняется только по
явному тапу «Отменить запись».

## 5. Структура и иерархия

### 5.1 Вариант «ранняя отмена» (≥ 24 часов до начала, FR-15)

Сверху вниз: грабер → заголовок → кнопки.

```
┌─────────────────────────────┐
│            ▔▔▔▔              │  ← грабер
│                              │
│  Точно отменить запись?      │  ← заголовок
│                              │
│  [ Отменить запись ]         │  ← деструктивная
│  [    Не отменять   ]        │  ← безопасная, акцентная
└─────────────────────────────┘
```

Никаких предупреждений не показывается — FR-15 явно требует отмену «без предупреждений».

### 5.2 Вариант «поздняя отмена» (< 24 часов до начала, FR-16)

Сверху вниз: грабер → заголовок → блок предупреждения → кнопки.

```
┌─────────────────────────────┐
│            ▔▔▔▔              │  ← грабер
│                              │
│  Точно отменить запись?      │  ← заголовок (тот же, что в п. 5.1)
│                              │
│  ┌─────────────────────────┐ │  ← блок предупреждения
│  │ Отмена не позднее чем за │ │     (дословно из foundations §6,
│  │ 24 часа до начала — без  │ │     строка «Правило отмены»)
│  │ предупреждений, место    │ │
│  │ освобождается. Позже —   │ │
│  │ запись всё равно         │ │
│  │ отменится, но продукты   │ │
│  │ на класс уже закуплены.  │ │
│  │ Штраф не взимается.      │ │
│  └─────────────────────────┘ │
│                              │
│  [ Отменить запись ]         │  ← деструктивная
│  [    Не отменять   ]        │  ← безопасная, акцентная
└─────────────────────────────┘
```

Предупреждение стоит **между заголовком и кнопками**, чтобы клиент физически не мог
подтвердить, не увидев его (задача постановки: «предупреждение... ПЕРЕД подтверждением»).

> Иерархия следует [§3.3 foundations](00-foundations.md#33-типографическая-иерархия-уровни-не-шрифты):
> заголовок экрана/шторки → основной текст (предупреждение) → кнопки.

## 6. Компоненты и поведение

### 6.1 Определение варианта

Вариант шторки (5.1 или 5.2) выбирается по времени до начала класса, которое сообщает сервер
(UC-2, шаг 3). Порог — 24 часа (FR-15/FR-16): «не позднее чем за 24 часа» трактуется как
**граница входит в раннюю отмену** (≥ 24 ч — ранняя; < 24 ч — поздняя) — формулировка домена
(`02-domain.md` → «Политика отмен») и foundations §6 однозначно ставят порог на стороне «без
предупреждений» для «не позднее чем 24 часа».

### 6.2 Блок предупреждения (только поздняя отмена)

Текст — **дословная цитата** строки «Правило отмены» из [§6 foundations](00-foundations.md#6-tone-of-voice-и-общая-микрокопия):
не переформулируется, не сокращается. Смысл (что запись всё равно отменится, но без штрафа)
не передаётся только цветом/иконкой — дублируется текстом (§3.2 foundations).

### 6.3 Кнопки

- **«Отменить запись»** — деструктивное действие, визуально менее акцентно относительно
  безопасной кнопки (стиль `destructive`, конкретный цвет — на усмотрение дизайнера, без
  хардкода), но всегда доступно для тапа: FR-16 прямо требует, что даже поздняя отмена
  **выполняется**, а не блокируется.
- **«Не отменять»** — безопасное действие, основное/акцентное, не должно визуально маскироваться
  под деструктивное («не ловушка»).
- Обе кнопки — тач-зоны ≥ 44–48 pt с отступом, исключающим случайный тап не по цели
  ([§3.1 foundations](00-foundations.md#31-тач-зоны-и-размеры)).

### 6.4 Состояние выполнения (loading)

После тапа «Отменить запись» — кнопка переходит в состояние загрузки/блокировки, повторные тапы
исключены; остальные элементы шторки (включая «Не отменять» и крестик) блокируются до ответа
API — защита от повторной/двойной отправки запроса отмены.

## 7. Состояния шторки

Применяется сквозной паттерн [§5 foundations](00-foundations.md#5-сквозной-паттерн-состояний-экрана);
специфика BS-003 (это шторка действия, а не загрузки данных — состояний Loading/Empty при
открытии у неё нет, так как время до начала класса приходит вместе с данными записи, уже
загруженными на SCR-006):

| Состояние | Содержимое / поведение |
|-----------|-------------------------|
| **Content — ранняя отмена** (≥ 24 ч, FR-15) | Заголовок + кнопки, без предупреждений (см. §5.1). |
| **Content — поздняя отмена** (< 24 ч, FR-16) | Заголовок + блок предупреждения (§5.2) + кнопки. |
| **Выполнение отмены** | Кнопка подтверждения в состоянии загрузки; элементы заблокированы (§6.4). |
| **Ошибка отмены** (UC-2, исключение E3 — сетевой сбой) | Шторка **остаётся открытой**, статус записи не меняется; показывается снек ошибки — по правилу [§6.2 foundations](00-foundations.md#62-кто-показывает-снек-при-закрытии-шторки) снек ошибки, при которой шторка не закрывается, показывает **сама шторка**, а не SCR-006. Текст — сквозная сетевая ошибка действия из [§6 foundations](00-foundations.md#6-tone-of-voice-и-общая-микрокопия): «Не удалось выполнить. Проверьте соединение и повторите.» Клиент может повторить тап «Отменить запись». |

> Случай «запись уже отменена» (UC-2, исключение E2) и случай «класс уже начался» (UC-2,
> исключение E1) предотвращаются на [SCR-006](SCR-006-booking-details.md) — кнопка «Отменить»
> там недоступна/не показывается ещё до открытия этой шторки; сама BS-003 эти случаи не
> обрабатывает.

## 8. Валидации и микрокопия

Точные тексты:

- **Заголовок** (оба варианта): «Точно отменить запись?»
- **Предупреждение при поздней отмене** — дословная цитата «Правило отмены» из
  [§6 foundations](00-foundations.md#6-tone-of-voice-и-общая-микрокопия):
  > «Отмена не позднее чем за 24 часа до начала — без предупреждений, место освобождается.
  > Позже — запись всё равно отменится, но продукты на класс уже закуплены. Штраф не
  > взимается.»
- **Кнопка подтверждения:** «Отменить запись»
- **Кнопка отказа:** «Не отменять»
- **Ошибка выполнения отмены** (сеть, шторка остаётся открытой) — сквозной текст из
  [§6 foundations](00-foundations.md#6-tone-of-voice-и-общая-микрокопия), строка «Сетевая ошибка
  при действии»:
  > «Не удалось выполнить. Проверьте соединение и повторите.»

Снеки после закрытия шторки (показывает SCR-006, не сама шторка) — точные тексты уже
зафиксированы в [§6.1 foundations](00-foundations.md#61-каталог-снеков-успеха) и здесь не
переопределяются:
- ранняя отмена → «Запись отменена»;
- поздняя отмена → «Поздняя отмена: продукты на класс уже закуплены. Штраф не взимается.»

> Числа (время до начала, дата/время класса) в тексты шторки не зашиваются и не выводятся в
> ней вовсе (см. §4) — они уже показаны на SCR-006; шторка использует только факт варианта
> (ранняя/поздняя), полученный от сервера (§6.1 этого документа).

## 9. Доступность и mobile-first

Базовые требования — [§7 foundations](00-foundations.md#7-доступность) и
[§3.1](00-foundations.md#31-тач-зоны-и-размеры). Специфика BS-003:

- **Защита от случайного нажатия:** деструктивное действие («Отменить запись») требует явного
  тапа по кнопке; тап по бэкдропу/swipe-вниз закрывает шторку как отказ, а не как подтверждение
  (§3 этого документа).
- **Безопасная кнопка «Не отменять»** имеет понятную акцентную форму, доступное имя для screen
  reader и не маскируется под деструктивную.
- Различие вариантов (наличие/отсутствие предупреждения) передаётся текстом и структурой
  (выделенный блок), не цветом в одиночку ([§3.2 foundations](00-foundations.md#32-контраст-и-читаемость-nfr-1)).
- Крупные тач-зоны кнопок с отступом между ними ([§3.1 foundations](00-foundations.md#31-тач-зоны-и-размеры)).

## 10. Критерии приёмки дизайна

```gherkin
Функция: Подтверждение отмены записи (BS-003)

  Сценарий: AC-001 Ранняя отмена — без предупреждений
    Дано до начала класса осталось 24 часа или больше
    Когда клиент открывает шторку подтверждения отмены
    Тогда показан заголовок «Точно отменить запись?»
    И не показывается предупреждение о продуктах
    И доступны кнопки «Отменить запись» и «Не отменять»

  Сценарий: AC-002 Поздняя отмена — предупреждение перед подтверждением
    Дано до начала класса осталось меньше 24 часов
    Когда клиент открывает шторку подтверждения отмены
    Тогда показан заголовок «Точно отменить запись?»
    И показано предупреждение из foundations §6 о закупленных продуктах
    И явно указано, что штраф не взимается
    И предупреждение расположено до кнопок подтверждения

  Сценарий: AC-003 Отказ от отмены
    Дано шторка подтверждения отмены открыта (любой вариант)
    Когда клиент нажимает «Не отменять», крестик, тапает вне шторки или свайпает вниз
    Тогда шторка закрывается
    И статус записи не изменяется
    И снек не показывается

  Сценарий: AC-004 Подтверждение ранней отмены
    Дано шторка в варианте ранней отмены открыта
    Когда клиент нажимает «Отменить запись»
    Тогда показывается состояние выполнения без возможности повторного тапа
    И при успехе шторка закрывается, клиент возвращается на SCR-006
    И SCR-006 показывает обновлённый статус записи и снек «Запись отменена»

  Сценарий: AC-005 Подтверждение поздней отмены
    Дано шторка в варианте поздней отмены открыта и предупреждение показано
    Когда клиент нажимает «Отменить запись»
    Тогда показывается состояние выполнения без возможности повторного тапа
    И при успехе шторка закрывается, клиент возвращается на SCR-006
    И SCR-006 показывает обновлённый статус записи и снек «Поздняя отмена: продукты на класс уже закуплены. Штраф не взимается.»

  Сценарий: AC-006 Ошибка выполнения отмены (сетевой сбой)
    Дано клиент нажал «Отменить запись» в любом варианте шторки
    Когда запрос отмены завершается сетевой ошибкой
    Тогда шторка остаётся открытой
    И статус записи не изменяется
    И сама шторка показывает снек «Не удалось выполнить. Проверьте соединение и повторите.»
    И клиент может повторить попытку тапом «Отменить запись»
```

## 11. Решения

| # | Решение |
|---|---------|
| 1 | **Обратный отсчёт до 24 часов в шторке не показываем** — лишний шум; вариант (ранний/поздний) шторка получает уже готовым от сервера при открытии, конкретное «сколько осталось» не требуется ни одним FR (допущение дизайнера, по аналогии с решением не показывать таймер в BS-003 у примера SUP-проекта). |
| 2 | **Заголовок один и тот же для обоих вариантов** («Точно отменить запись?») — разница между вариантами выражается наличием/отсутствием блока предупреждения, а не отдельными заголовками; это меньше меняющихся элементов интерфейса между состояниями. |
| 3 | **Параметры брони (дата/время/программа) в шторке не повторяются** — контекст уже виден на [SCR-006](SCR-006-booking-details.md); шторка фокусируется только на предупреждении (если нужно) и подтверждении. |
| 4 | **Пограничный случай «ровно 24 часа» — ранняя отмена**, без предупреждений — прямое прочтение формулировки FR-15/`02-domain.md` «не позднее чем за 24 часа» как включающей саму границу. |
| 5 | **Никакого упоминания денежного штрафа при ранней отмене** — раз штрафов нет вообще ни при какой отмене (FR-16), при ранней отмене тема штрафа не поднимается вовсе (не нужно объяснять отсутствие того, чего не ожидается) — сообщается прямо и по делу только в позднем варианте, где предупреждение уже показывается по другой причине (продукты). |
