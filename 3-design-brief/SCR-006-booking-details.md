## 1. Назначение и контекст

Экран показывает **полную информацию об одной записи (брони) клиента** и служит точкой
**запуска её отмены**. Это вложенный экран, открываемый из списка
[«Мои бронирования» (SCR-005)](SCR-005-my-bookings.md) тапом по конкретному бронированию
(FR-13, US-9).

Три задачи экрана:

1. **Просмотр.** Собрать в одном месте параметры записи, статус и выбранный вариант экипировки,
   чтобы клиент понимал, на что и когда он записан и что оплатить офлайн (FR-13, US-9, US-10,
   US-11).
2. **Отмена.** Дать единственное действие над записью — отменить участие. Само подтверждение,
   логика раннего/позднего порога и текст предупреждения выполняются в шторке
   [BS-003 «Подтверждение отмены»](BS-003-cancel-confirm.md) — экран лишь запускает её и
   отображает результат после возврата (FR-14, FR-15, FR-16, UC-2).
3. **Переход к оценке шефа.** Для прошедшего и неотменённого класса — точка входа в
   [SCR-007 «Оценка шефа»](SCR-007-chef-rating.md) (FR-20, UC-4 «Предусловия»).

Экран **не показывает карту** — ни маршрута, ни адреса на карте: в домене «Шеф-стол» нет
геопривязанного перемещения (класс идёт в лофте студии по расписанию), в требованиях (FR-1…FR-23,
UC-1…UC-4) карта нигде не упоминается. Адрес студии как поле данных упомянут в
[`02-domain.md`](../1-elicitation/02-domain.md) («канонической схемой считается контракт API...
адрес студии»), но конкретный экран для его показа заказчиком/аналитиком не задан — на SCR-006 он
не выводится, чтобы не изобретать содержимое сверх FR-13.

## 2. Пользователь и цель

- **Роль:** Клиент (единственная роль приложения, [foundations §1](00-foundations.md#1-продукт-и-аудитория)).
- **Цель пользователя:** убедиться в деталях своей записи и при необходимости отменить её
  заранее, понимая последствия; для прошедшего класса — оценить шефа.
- **User stories:**
  - [US-10](../2-requirements/user-stories.md) — «Как клиент, я хочу отменить свою запись до
    начала класса, чтобы освободить место, если планы изменились».
  - [US-11](../2-requirements/user-stories.md) — «Как клиент, я хочу видеть, что класс отменён
    студией, и знать причину, чтобы не приходить зря и понимать, что произошло».
- **Сценарий:** [UC-2 «Отмена записи»](../2-requirements/use-cases.md) — основной поток, A1
  (поздняя отмена), A2 (отмена студией как входящее состояние экрана), исключения E1–E3. Переход
  к оценке — предусловия [UC-4](../2-requirements/use-cases.md).
- **Принцип «только свои данные»** ([foundations P5](00-foundations.md#2-дизайн-принципы)):
  экран показывает данные исключительно текущего клиента, без доступа к чужим бронированиям.

## 3. Навигация

**Входящие:**
- Тап по бронированию в списке [«Мои бронирования» (SCR-005)](SCR-005-my-bookings.md).
- Возврат с [SCR-007 «Оценка шефа»](SCR-007-chef-rating.md) после отправки оценки (`submitRating`)
  — SCR-006 как экран-родитель показывает снек **«Спасибо за оценку!»**
  ([foundations §6.1/§6.2](00-foundations.md#61-каталог-снеков-успеха)); поле `rating_submitted`
  (§4) обновляется, CTA «Оценить шефа» заменяется пометкой «Оценка отправлена» (§6.4).

**Исходящие:**
- **«Отменить»** (нижний CTA) — доступна, только пока класс не начался (FR-14; UC-2 E1) →
  открывает шторку [BS-003 «Подтверждение отмены»](BS-003-cancel-confirm.md). После подтверждения
  отмены управление возвращается на SCR-006 с обновлённым статусом и снеком-итогом («Запись
  отменена» / «Поздняя отмена: продукты на класс уже закуплены. Штраф не взимается.») — SCR-006, а
  не [SCR-005](SCR-005-my-bookings.md), показывает эту обратную связь (см.
  [foundations §6.2](00-foundations.md#62-кто-показывает-снек-при-закрытии-шторки)); возврат в
  список бронирований — отдельным действием клиента через «Назад».
- **«Оценить шефа»** (нижний CTA) — доступна, только если класс прошёл, запись не была отменена
  ни клиентом, ни студией, и оценка ещё не поставлена (FR-20, UC-4 «Предусловия», §6.4) → переход
  на [SCR-007 «Оценка шефа»](SCR-007-chef-rating.md).
- **«Назад»** (в хедере) → возврат на [SCR-005](SCR-005-my-bookings.md).

Таб-бар на этом экране **скрыт** — экран вложенный
([foundations §4.2](00-foundations.md#42-таб-бар-авторизованная-зона)).

> «Отменить» и «Оценить шефа» взаимоисключающи по своим условиям (класс либо ещё не начался, либо
> уже прошёл) — на экране в любой момент времени актуален не более чем один из двух CTA, см. §5,
> §6.3–6.4.

## 4. Контент-инвентарь

Источник состава полей — FR-13 («статус, параметры слота, вариант экипировки») и переиспользуемый
набор параметров слота из FR-3/FR-6 (используется также на SCR-002/SCR-003).

| Блок | Данные | Требование | Примечание |
|------|--------|------------|------------|
| Статус | Бейдж: Активна / Отменена / Отменён студией / Прошла | FR-13; [foundations §3.5](00-foundations.md#35-иконки-и-индикаторы) | Ровно 4 канонических состояния (см. §6.2). |
| Причина отмены студией | Текст причины | FR-17 | Показывается **только** при статусе «Отменён студией». |
| Когда | Дата и время начала класса | FR-13; FR-3/FR-6 | Крупно, контрастно ([foundations §3.2](00-foundations.md#32-контраст-и-читаемость-nfr-1)). |
| Программа | Название программы + уровень сложности | FR-13; FR-6 | Тот же состав, что на карточке слота (SCR-003), без полного списка ингредиентов/аллергенов — это информация для решения о записи (уже принято), не для просмотра готовой брони; **допущение дизайнера**, при уточнении заказчика список можно вернуть. |
| Шеф | Имя шефа | FR-13; FR-6 | Read-only справочные данные. |
| Экипировка | «Своя экипировка» / «Прокатная экипировка» | FR-8, FR-13 | Лейблы — из [foundations §6](00-foundations.md#6-tone-of-voice-и-общая-микрокопия). |
| Цена | Сумма + напоминание об офлайн-оплате | FR-23 | Текст оплаты — единый, из foundations §6 («Оплата на месте: наличные или перевод на карту.»). |
| Момент бесплатной отмены | «Бесплатно отменить можно до `<free_cancellation_until>`» | FR-15, FR-16 | Только для статуса «Активна»; значение — **готовое поле из API** (P3, [foundations §3 принцип «Числа только из API»](00-foundations.md#2-дизайн-принципы)), а не вычисляется на клиенте вычитанием 24 часов из времени начала — порог остаётся на стороне сервера (см. [BS-003 §1](BS-003-cancel-confirm.md)), клиент только отображает присланное значение. |
| Оценка поставлена (`rating_submitted`) | Флаг: есть ли уже оценка по этой записи | FR-20, UC-4 E2 | Не отображается напрямую; определяет показ CTA «Оценить шефа» vs пометки «Оценка отправлена» (см. §6.4). Только для статуса «Прошла». |

> Числа (цена) не хардкодятся — берутся из данных API ([foundations P3](00-foundations.md#2-дизайн-принципы)).

## 5. Структура и иерархия

Каркас — [foundations §4.1](00-foundations.md#41-базовый-каркас): фиксированный хедер,
скролл-контент, фиксированный нижний CTA. Таб-бар отсутствует.

- **Хедер (фикс.):** кнопка «Назад» + заголовок экрана «Детали записи».
- **Скролл-контент:** бейдж статуса (вверху, заметный) → (если статус «Отменён студией» — блок
  причины) → блок «Когда» → блок «Программа и шеф» → блок «Экипировка» → блок «Цена» (+
  напоминание об офлайн-оплате) → (если статус «Активна») строка «Бесплатно отменить можно до…».
- **Фикс. нижний CTA:** одна кнопка на экран, контекстно зависящая от статуса (см. §6.3–6.4):
  «Отменить» (статус «Активна»), «Оценить шефа» (статус «Прошла»), либо кнопка «Отменить»,
  показанная неактивной с пояснением (статусы «Отменена» / «Отменён студией»).

## 6. Компоненты и поведение

### 6.1 Блоки деталей
Сгруппированные секции в одну колонку
([foundations §3.4](00-foundations.md#34-плотность-и-сетка)). Каждая секция — заголовок-лейбл +
значение; ключевые числа (время начала, цена) — крупные и контрастные
([foundations §3.2](00-foundations.md#32-контраст-и-читаемость-nfr-1)). Все данные read-only, без
редактирования.

### 6.2 Бейдж статуса
Отражает статус записи. Состояния не передаются только цветом — обязательны иконка/текст
([foundations §3.2, §3.5](00-foundations.md#35-иконки-и-индикаторы)). Ровно четыре состояния:

- **Активна** — предстоящая запись, класс ещё не начался, не отменена.
- **Отменена** — клиент отменил запись (ранняя или поздняя — по политике FR-15/FR-16 отмена в
  обоих случаях выполняется, отдельного постоянного статуса «поздняя отмена» нет; различие видно
  только в снеке в момент отмены, см. [foundations §6.1](00-foundations.md#61-каталог-снеков-успеха)).
- **Отменён студией** — класс отменён организатором/форс-мажором; текст — «Класс отменён студией.
  Причина: [reason]. Повторная запись на этот слот недоступна.» ([foundations §6](00-foundations.md#6-tone-of-voice-и-общая-микрокопия),
  FR-17). Бронь **не удаляется** из истории.
- **Прошла** — время начала уже наступило, запись не была отменена ни клиентом, ни студией.

### 6.3 CTA «Отменить» (нижний, контекст «Активна»)
- Крупная, во всю ширину ([foundations §3.1](00-foundations.md#31-тач-зоны-и-размеры)).
- **Enabled** — только когда статус «Активна» (класс ещё не начался). Тап → открывает
  [BS-003](BS-003-cancel-confirm.md) (FR-14).
- **Disabled, с обязательным пояснением рядом** (не скрывается) — для статусов «Прошла» (UC-2 E1)
  и «Отменена» / «Отменён студией» (UC-2 E2). Тексты пояснений — см. §8.
- **Отмена — только целиком.** Частичная отмена в MVP не поддерживается: это прямое следствие
  допущения «одна запись = одно место» (см. `functional-requirements.md`, врезка перед FR-7, и
  `02-domain.md` → «Открытые вопросы по домену», п. 1) — отменять «часть» записи из одного места
  логически невозможно, поэтому единственная разумная трактовка — отмена целиком.
- Само действие отмены (запрос к API, определение раннего/позднего порога, текст предупреждения)
  выполняется в [BS-003](BS-003-cancel-confirm.md), а не на этом экране — см. §3.

### 6.4 CTA «Оценить шефа» (нижний, контекст «Прошла»)
- Показывается вместо «Отменить», когда статус — «Прошла» (класс состоялся и не был отменён ни
  клиентом, ни студией — FR-20, UC-4 «Предусловия») **и** по этой записи ещё не поставлена оценка.
- Поле `rating_submitted` (есть/нет оценки по этой записи) — часть контент-инвентаря (§4),
  источник — API; определяет, показывается ли CTA. Если оценка уже поставлена, вместо CTA
  показывается нейтральная пометка **«Оценка отправлена»** (не кнопка) — это защищает от повторной
  отправки (UC-4 E2) уже на этом экране, а не только на SCR-007.
- Тап → переход на [SCR-007 «Оценка шефа»](SCR-007-chef-rating.md).
- Для статусов «Отменена» и «Отменён студией» кнопка «Оценить шефа» **не показывается** — FR-20
  прямо разрешает оценку только для неотменённых прошедших записей.

### 6.5 Экипировка
Значение read-only, лейблы «Своя экипировка» / «Прокатная экипировка» — единые для всего
приложения ([foundations §6](00-foundations.md#6-tone-of-voice-и-общая-микрокопия), FR-8).
Прокатный фонд не ограничен и никогда не влиял на исход записи (FR-9) — на этом экране
дополнительно не отображается и не комментируется.

### 6.6 Запрет повторной записи на слот, отменённый студией (для полноты картины)
Если статус записи — «Отменён студией», повторная запись клиента на этот же слот запрещена
(FR-18). Это состояние **не относится к CTA этого экрана** (на SCR-006 в принципе нет действия
«записаться»), а проявляется на [SCR-002](SCR-002-slot-list.md) и
[SCR-003](SCR-003-slot-card.md): для отменённого студией слота там недоступен CTA «Записаться».
Упомянуто здесь для целостности картины состояния «Отменён студией».

## 7. Состояния экрана

Базовый паттерн — [foundations §5](00-foundations.md#5-сквозной-паттерн-состояний-экрана).
Специфика SCR-006:

| Состояние | Содержимое / поведение |
|-----------|-------------------------|
| **Loading** | Скелетон в форме блоков деталей и бейджа статуса (не пустой экран). |
| **Content** | Полные детали записи + статус + контекстный нижний CTA по правилам §6.3–6.4. |
| **Error** | Заглушка ошибки + кнопка «Обновить» (текст — [foundations §6](00-foundations.md#6-tone-of-voice-и-общая-микрокопия): «Не удалось загрузить. Проверьте соединение и попробуйте снова.»). |
| **Отменена клиентом** | Бейдж «Отменена». CTA «Отменить» неактивна с пояснением (UC-2 E2). |
| **Отменён студией** | Бейдж «Отменён студией» + причина (FR-17). CTA «Отменить» неактивна с пояснением. |
| **Класс начался/прошёл, запись активна до этого момента** | Бейдж «Прошла». CTA сменяется на «Оценить шефа» (если запись не была отменена, FR-20) либо, если условие не выполняется (запись была отменена до старта), CTA-раздел показывает соответствующий disabled-статус «Отменена»/«Отменён студией», не «Прошла» — приоритет у факта отмены. |

Empty state неприменим — экран всегда открывается для одной существующей записи; переход на
несуществующую/чужую запись трактуется как ошибка (Error), а не как пустое состояние — это
согласуется с принципом «только свои данные» (foundations P5).

## 8. Ошибки и исключения (UC-2) и микрокопия

Ниже — точные формулировки альтернативных потоков и исключений UC-2, относящихся к этому экрану и
к запускаемой с него шторке BS-003.

**Альтернативные потоки:**
- **A1. Поздняя отмена (менее 24 часов до начала).** «На шаге 4 запись всё равно отменяется (не
  блокируется), но клиенту показывается предупреждение о том, что продукты на класс уже
  закуплены; денежного штрафа нет (Q-007, FR-16).» — предупреждение и подтверждение показываются
  в [BS-003](BS-003-cancel-confirm.md), а не на этом экране; после возврата на SCR-006 отражается
  итоговый статус «Отменена» (см. §6.2) и снек-текст по [foundations §6.1](00-foundations.md#61-каталог-снеков-успеха).
- **A2. Класс отменён студией (форс-мажор).** «Бронь клиента не удаляется, а переходит в статус
  «Отменён студией» с указанием причины; клиенту приходит push-уведомление. При недоступности
  push клиент увидит статус при следующем открытии приложения — резервного канала нет (Q-014,
  FR-17, FR-22). Повторная запись на этот слот запрещена (R-008, FR-18).» — это **входящее
  состояние** экрана (клиент не инициирует его сам), см. §6.2, §6.6.

**Исключения / ошибки:**
- **E1.** «Класс уже начался или прошёл. Кнопка отмены недоступна.» — на SCR-006 реализуется как
  disabled-состояние CTA «Отменить» при статусе «Прошла», с пояснением (см. §6.3, §7).
- **E2.** «Запись уже отменена (повторная попытка). Повторная отмена не выполняется, показывается
  текущий статус записи.» — на SCR-006 реализуется как disabled-состояние CTA «Отменить» при
  статусах «Отменена» / «Отменён студией», с показом текущего статуса (см. §6.2, §6.3).
- **E3.** «Сетевой сбой при отмене. Приложение показывает сообщение об ошибке и предлагает
  повторить попытку; финальный статус запись получает по ответу сервера.» — сама попытка отмены и
  обработка сбоя происходят в [BS-003](BS-003-cancel-confirm.md) (там же — повтор попытки); на
  SCR-006 в этом случае статус записи остаётся прежним до получения ответа сервера.

**Микрокопия disabled-пояснений (по мотивам E1/E2, тон — [foundations §6](00-foundations.md#6-tone-of-voice-и-общая-микрокопия), без вины пользователя):**

| Ситуация | Пояснение рядом с неактивной кнопкой |
|----------|----------------------------------------|
| E1 — класс уже начался/прошёл | «Класс уже начался — отменить запись нельзя.» |
| E2 — запись уже отменена (клиентом) | «Запись уже отменена.» |
| Запись отменена студией | «Класс отменён студией — отменять уже нечего.» |

> Полный текст правила отмены (порог 24 часа, отсутствие штрафа) **не дублируется** здесь —
> единый источник: [foundations §6](00-foundations.md#6-tone-of-voice-и-общая-микрокопия). На
> SCR-006 рядом с активной кнопкой «Отменить» показывается только вычисленный момент отсечки
> («Бесплатно отменить можно до `<start_at − 24ч>`», см. §4), без пересказа самого правила.

## 9. Доступность и mobile-first

Базовые требования — [foundations §7](00-foundations.md#7-доступность) и
[§3](00-foundations.md#3-структурные-токены-без-бренда). Специфика экрана:

- **Читаемый статус** — бейдж дублирует смысл текстом и иконкой, не только цветом
  ([foundations §3.2, §3.5](00-foundations.md#35-иконки-и-индикаторы)).
- **Крупный контекстный CTA** — кнопка («Отменить» либо «Оценить шефа») во всю ширину, тач-зона
  ≥ 44–48 pt; disabled-состояние визуально и программно (доступное имя для screen reader)
  считывается однозначно.
- **Осторожность с деструктивным действием** — отмена не происходит по одному тапу на этом
  экране: обязательное подтверждение вынесено в [BS-003](BS-003-cancel-confirm.md)
  ([foundations §4.3](00-foundations.md#43-bottom-sheet-шторки-bs-001bs-004) — критичные
  подтверждения закрываются только явной кнопкой).
- Layout не ломается при системном увеличении шрифта; время начала и цена остаются крупными и
  контрастными.

## 11. Решения

| # | Решение |
|---|---------|
| 1 | **Карты на экране нет.** В домене нет геопривязанного маршрута/адреса как отдельного UI-требования (FR-1…FR-23 карту не упоминают) — в отличие от примера с SUP-прогулками, здесь класс проходит в лофте студии по расписанию. |
| 2 | **Отмена — только целиком.** Частичная отмена невозможна логически: одна запись = одно место (допущение аналитика, `functional-requirements.md` перед FR-7; `02-domain.md` → «Открытые вопросы», п. 1). |
| 3 | **Штрафов нет ни в каком виде.** Поздняя отмена — предупреждение о закупленных продуктах, без денежных последствий (FR-16); экран не показывает никаких «сумм штрафа». |
| 4 | **Один контекстный CTA вместо двух одновременно видимых.** «Отменить» и «Оценить шефа» логически взаимоисключающи (класс либо ещё не начался, либо уже прошёл) — экран показывает только релевантный, чтобы не перегружать интерфейс (P6, спокойный тон). |
| 5 | **Постоянного статуса «поздняя отмена» нет.** Разница ранней/поздней отмены (FR-15 vs FR-16) видна только в снеке в момент самой отмены ([foundations §6.1](00-foundations.md#61-каталог-снеков-успеха)); долгосрочный статус записи — просто «Отменена», по канону из [foundations §3.5](00-foundations.md#35-иконки-и-индикаторы). |
| 6 | **Список ингредиентов/аллергенов программы не повторяется на этом экране** — это информация для принятия решения перед записью (FR-6, показывается на SCR-003), после подтверждения брони её ценность на экране статуса ниже; допущение дизайнера, подлежит уточнению у заказчика. |
