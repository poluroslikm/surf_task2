## 1. Назначение и контекст

Экран отвечает на задачу клиента «понять, подходит ли мне класс, перед записью» (US-4). Это
второй шаг основного потока **UC-1 «Запись на класс»**: «Открывает карточку выбранного слота и
просматривает параметры: дата/время, программа (фото, ингредиенты/аллергены), уровень
сложности, шеф, всего/свободно мест» (UC-1, шаг 2).

Экран — **чисто просмотровый**: он даёт информацию для решения «записываться или нет» и не
содержит формы записи. Выбор экипировки и подтверждение брони — отдельный экран
[SCR-004 «Оформление записи»](SCR-004-booking.md) (реестр экранов, [design-brief.md](design-brief.md));
здесь эта форма не повторяется, только описан переход к ней (см. §3).

Все данные слота, программы и шефа — **read-only справочные данные из API**
(`02-domain.md` → «Ключевые объекты домена»: «Класс/слот... клиенту доступен только для чтения»;
«Программа... справочные данные, read-only»; «Шеф... справочные данные слота (read-only)»).
Клиент ничего не редактирует на этом экране.

Принцип **P2 «Короткий путь к записи»** (`00-foundations.md` §2, NFR-2): карточка — второй из
ориентировочно ≤ 3 экранов до подтверждения (список → **карточка** → оформление → подтверждение).

## 2. Пользователь и цель

- **Роль:** Клиент (единственная роль приложения, `00-foundations.md` §1).
- **Контекст использования:** mobile-first (`00-foundations.md` §1, §2 P1); клиент только что
  тапнул по карточке слота в списке (SCR-002) и хочет за несколько секунд оценить, подходит ли
  ему класс — по сложности программы, составу блюда/аллергенам, времени, шефу, наличию мест и
  цене — прежде чем переходить к записи (US-4).
- **Цель пользователя:** получить полный набор параметров слота, чтобы принять решение
  «записываться / не записываться», не переходя пока к форме записи (US-4, UC-1 шаг 2).
- **Цель системы:** показать все параметры слота, точно как их вернул API, ничего не
  пересчитывая и не дублируя бизнес-логику вместимости на клиенте (FR-12, `00-foundations.md`
  §2 P3).

## 3. Навигация

### Входящие
- **Основной путь.** С [SCR-002 «Список слотов»](SCR-002-slot-list.md) по тапу на карточку слота
  в списке (реестр экранов, `design-brief.md`; UC-1, переход шаг 1 → шаг 2).
- **Состояние «слот уже заполнен».** Функциональные требования отдельно оговаривают случай,
  когда клиент всё же попадает на карточку полностью заполненного слота (например, список успел
  устареть за время его сессии, либо он вернулся на карточку через историю браузера/прямую
  ссылку). Это не отдельный сценарий навигации, а требование к **состоянию экрана**: в списке
  такие слоты уже показываются с пометкой «мест нет» и без возможности записи (FR-6a,
  [SCR-002](SCR-002-slot-list.md)); на карточке в этом случае действует то же правило — см. §7.
- **Состояние «слот отменён студией».** Аналогично, если клиент попадает на карточку слота,
  который к этому моменту отменила студия ([SCR-002 §6.2a](SCR-002-slot-list.md#62a-слот-отменённый-студией)),
  карточка отображает бейдж «Отменён студией», а не пытается показать обычное состояние
  вместимости — см. §6.4a, §7.

### Исходящие
- **«Назад»** (в хедере) → возврат на [SCR-002 «Список слотов»](SCR-002-slot-list.md).
- **«Записаться»** (нижний CTA, активен только если `free_seats > 0`) → переход на
  [SCR-004 «Оформление записи»](SCR-004-booking.md) (реестр экранов; UC-1, переход шаг 2 → шаг 3).
  На этом переходе зона ответственности SCR-003 заканчивается: выбор экипировки и подтверждение
  брони описаны в SCR-004, здесь не дублируются.

Таб-бар на этом экране **скрыт** — это вложенный экран потока записи
(`00-foundations.md` §4.2: «Таб-бар скрывается на вложенных экранах (карточка слота, оформление
записи...)»).

## 4. Контент-инвентарь

Полный состав параметров слота, обязательный к показу (FR-6, US-4):

| Поле | Обязательность | Источник |
| :-- | :-- | :-- |
| Дата и время начала | Обязательно | FR-6, US-4, UC-1 шаг 2 |
| Название программы | Обязательно | FR-6, US-4 |
| Фото блюда | Обязательно | FR-6, US-4 |
| Список ингредиентов/аллергенов блюда | Обязательно | FR-6, US-4 |
| Уровень сложности программы — **отдельное явное поле**, не только текст в названии | Обязательно | FR-6 (сформулировано явно: «явное поле, а не только текст в названии») |
| Шеф (имя) | Обязательно | FR-6, US-4 |
| Всего мест / свободно мест | Обязательно, **значение строго из API** | FR-6, FR-12, US-4, US-8 |
| Статус «Отменён студией» (если применимо) | Обязательно при наличии | FR-17, FR-18; §6.4a, §7 |
| Цена класса | Обязательно | US-14 («в карточке слота и при подтверждении записи показана цена»); FR-23 |
| CTA «Записаться» (доступен/недоступен) | Обязательно | FR-6a, FR-18; §7 ниже |

> **Цена — вне первичного перечня FR-6, но требуется явно US-14.** Реестр экранов
> (`design-brief.md`) указывает для SCR-003 в качестве основных FR-6/US-4/UC-1(шаг2); при этом
> US-14 отдельно и буквально требует показ цены **на карточке слота** («В карточке слота и при
> подтверждении записи показана цена»). Включаю блок цены в макет на этом основании, со ссылкой
> на US-14 и FR-23.

**Явно не показывается на этом экране** (чтобы не выходить за рамки требований):
- Карта/маршрут, геолокация студии — не упомянуты ни в FR-6, ни в глоссарии; в проекте нет карты.
- Форма ввода аллергий клиента — список ингредиентов/аллергенов блюда описывает **состав блюда**
  (read-only данные программы), а не собирает данные о здоровье клиента; сбор данных об аллергиях
  клиента прямо вынесен вне скоупа (`02-domain.md` → «Вне скоупа»: «Сбор данных об аллергиях
  клиента»).
- Выбор количества мест/гостей — «одна запись — одно место» (допущение по `functional-requirements.md`,
  врезка перед FR-7; `00-foundations.md` §1); на этом экране вообще нет полей ввода.
- Онлайн-оплата — оплата офлайн (FR-23); на карточке — только показ цены, без формы оплаты.
- Фото/аватар шефа — домен описывает шефа для клиента только как справочные данные слота
  (`02-domain.md` → «Ключевые объекты домена»: «Шеф... для клиента — справочные данные слота
  (read-only)»), без указания фото; не добавляю лишнего поля.

## 5. Структура и иерархия

Каркас — по [`00-foundations.md` §4.1](00-foundations.md): фикс. хедер → скролл-контент → фикс.
нижний CTA. Сверху вниз:

1. **Хедер** — «Назад» + заголовок (название программы или обобщённое «Класс»).
2. **Фото блюда** — крупное изображение на всю ширину, первый визуальный якорь карточки.
3. **Заголовок программы + бейдж уровня сложности** — название программы рядом с явным
   отдельным полем сложности (FR-6).
4. **Блок даты/времени начала.**
5. **Блок шефа** — имя.
6. **Блок вместимости** — «Всего N мест · Свободно M» либо бейдж «Мест нет», если `free_seats = 0`
   (FR-6a, §7).
7. **Блок ингредиентов/аллергенов** — список состава блюда.
8. **Блок цены** — цена класса + напоминание об офлайн-оплате (текст из
   [`00-foundations.md` §6](00-foundations.md)).
9. **Фикс. нижний CTA «Записаться»** — активен/неактивен по правилу §7.

## 6. Компоненты и поведение

### 6.1 Фото блюда
- Изображение на всю ширину контентной области, фиксированное соотношение сторон (не «прыгает»
  при загрузке — скелетон занимает то же место, см. §7).
- Доступное имя для screen reader — описывает блюдо/программу (`00-foundations.md` §7).

### 6.2 Бейдж уровня сложности
- Отдельный, визуально самостоятельный элемент рядом с названием программы, а не часть заголовка
  текстом — прямое требование FR-6 («явное поле, а не только текст в названии»).
- Значение — как есть из данных программы (например, «Новичковый» / «Для опытных»,
  `02-domain.md` → глоссарий «Программа»); не передаётся только цветом (`00-foundations.md` §3.2).

### 6.3 Блок даты/времени и блок шефа
- Дата/время — крупный, контрастный текст (важное число, `00-foundations.md` §3.2).
- Шеф — имя текстом; данные read-only, без интерактивности (`00-foundations.md` §8.2).

### 6.4 Блок вместимости (всего/свободно)
- Значения «Всего» и «Свободно» — **строго как получены из API**, без пересчёта и без
  хардкода на клиенте (FR-12, US-8, `00-foundations.md` §2 P3).
- Крупный, контрастный текст (важное число, `00-foundations.md` §3.2).
- Если `free_seats = 0` — вместо числа «Свободно» показывается бейдж **«Мест нет»** (то же
  словесное обозначение, что и в списке слотов, FR-6a), не только цветом — текстом и формой
  (`00-foundations.md` §3.2, §3.5).

### 6.4a Слот, отменённый студией

- Если слот пришёл из API со статусом «отменён студией» (форс-мажор или решение организатора,
  FR-17), блок вместимости (§6.4) заменяется бейджем **«Отменён студией»** — визуально отличным
  от бейджа «Мест нет» (другой текст/иконка, не только цвет), с указанием причины, если она
  доступна на слоте.
- CTA «Записаться» **недоступен** в этом состоянии — повторная запись на слот, отменённый
  студией, запрещена для всех клиентов (FR-18), это правило действует одинаково на
  [SCR-002](SCR-002-slot-list.md#62a-слот-отменённый-студией), здесь и на SCR-004 (клиент не
  может дойти до оформления записи на такой слот).
- Карточка остаётся доступной для просмотра — это не ошибка загрузки, а обычное состояние Content
  (см. §7).

### 6.5 Блок ингредиентов/аллергенов
- Список состава блюда конкретной программы (FR-6). Это справочная информация о блюде, **не**
  форма сбора данных о собственных аллергиях клиента (см. §4 «явно не показывается»).

### 6.6 Блок цены
- Цена класса — крупно, рядом текст-напоминание об офлайн-оплате: «Оплата на месте: наличные
  или перевод на карту.» (точный текст из [`00-foundations.md` §6](00-foundations.md), FR-23).
- Цена не пересчитывается на этом экране (на карточке нет выбора экипировки и, соответственно,
  нет переменных, влияющих на итоговую стоимость, — в отличие от SCR-004, где к цене может
  добавляться тариф проката, FR-8).

### 6.7 Кнопка «Записаться» (нижний CTA)
- Во всю ширину контентной области, тач-зона ≥ 44–48 pt (`00-foundations.md` §3.1).
- **Enabled**, когда `free_seats > 0` — тап переводит на [SCR-004](SCR-004-booking.md) (UC-1,
  переход шаг 2 → шаг 3).
- **Disabled**, когда `free_seats = 0` (слот полностью заполнен) — FR-6a: «...без возможности
  записи на них»; кнопка не реагирует на тап, состояние недоступности читается не только по
  виду кнопки, но и по бейджу «Мест нет» рядом с вместимостью (§6.4).
- **Disabled**, когда слот отменён студией — FR-18; кнопка не реагирует на тап, недоступность
  читается по бейджу «Отменён студией» (§6.4a), а не только по виду кнопки.
- Цена не дублируется на самой кнопке (в отличие от кнопки на SCR-004, где показывается итоговая
  сумма брони) — на этом экране цена уже показана крупным отдельным блоком (§6.6), а сама кнопка
  ведёт не к оплате, а к следующему шагу оформления. *Решение дизайнера, см. §11.1.*

## 7. Состояния экрана

Базовый паттерн — [`00-foundations.md` §5](00-foundations.md) (Loading → Content / Empty / Error).
Специфика SCR-003:

| Состояние | Содержимое / поведение |
| :-- | :-- |
| **Loading** | Скелетон в форме будущей карточки: прямоугольник на месте фото, полосы на месте текстовых блоков (`00-foundations.md` §5). |
| **Content — есть свободные места** | Основной сценарий: все поля §4 заполнены, CTA «Записаться» активен. |
| **Content — слот полностью заполнен** | То же наполнение, но блок вместимости показывает бейдж **«Мест нет»** вместо числа свободных мест, а CTA «Записаться» **недоступен** (FR-6a). Это ожидаемое, не ошибочное состояние — актуально в том числе для случая, когда клиент попал на карточку уже заполненного слота напрямую (см. §3, «Состояние „слот уже заполнен“»). |
| **Content — слот отменён студией** | Блок вместимости заменяется бейджем **«Отменён студией»** (+ причина, если есть); CTA «Записаться» **недоступен** (FR-17, FR-18, §6.4a). Тоже ожидаемое состояние, не ошибка. |
| **Error** | Ошибка загрузки данных карточки — заглушка + кнопка **«Обновить»**, текст «Не удалось загрузить. Проверьте соединение и попробуйте снова.» (точный текст из `00-foundations.md` §5–§6, общий паттерн для экранов с запросом к API). |

> **Empty-состояние не применимо.** В отличие от списка слотов (SCR-002), карточка всегда
> открывается для одного конкретного, уже выбранного слота — «пустого» набора данных для одного
> объекта не бывает; либо данные загружены (Content), либо запрос завершился ошибкой (Error).

## 8. Валидации и микрокопия

> Тон — спокойный, без вины (`00-foundations.md` §6, P6). Раздел ограничен исключениями, которые
> относятся к **просмотру карточки перед записью** (по заданию — часть UC-1, касающаяся шага 2);
> исключения самого процесса записи (нехватка мест в момент подтверждения, отмена слота студией,
> сетевой сбой при подтверждении и т. д.) относятся к следующему шагу и описаны в
> [SCR-004](SCR-004-booking.md), здесь не дублируются.

| Ситуация | Текст / поведение | Источник |
| :-- | :-- | :-- |
| Слот полностью заполнен (`free_seats = 0`), в т.ч. при попадании на карточку напрямую | Бейдж **«Мест нет»** у вместимости; CTA «Записаться» неактивен, тап по нему ничего не делает | FR-6a |
| Слот отменён студией | Бейдж **«Отменён студией»** (+ причина, если есть) вместо блока вместимости; CTA «Записаться» неактивен, тап по нему ничего не делает | FR-17, FR-18 |
| Сбой сети при загрузке карточки | «Не удалось загрузить. Проверьте соединение и попробуйте снова.» + кнопка «Обновить» | `00-foundations.md` §5–§6 (общий паттерн) |

> **Микрокопия рядом с неактивной кнопкой — допущение дизайнера.** Каталог сквозных текстов
> (`00-foundations.md` §6) фиксирует формулировку «мест нет» как пометку в **списке** слотов
> (FR-6a), но не задаёт отдельный поясняющий текст рядом с неактивным CTA на карточке. Здесь
> переиспользуется то же слово «Мест нет» в виде бейджа у вместимости (§6.4) без дополнительного
> текста на самой кнопке — так как причина неактивности уже видна на экране. Это решение
> дизайнера, а не отдельно зафиксированное требование; при необходимости можно уточнить у
> заказчика/аналитика более развёрнутую формулировку.

## 9. Доступность и mobile-first

База — [`00-foundations.md` §3, §7](00-foundations.md). Специфика экрана:

- Фото блюда имеет доступное текстовое описание (alt) с названием программы.
- Кнопка «Записаться» имеет доступное имя, отражающее состояние, например: «Записаться на класс
  «Паста ручной работы», 4 июля» (активна) / «Записаться недоступно, мест нет» (неактивна) —
  по образцу примера доступного имени из `00-foundations.md` §7.
- Важные числа — дата/время, свободно/всего мест, цена — крупные и контрастные, не ниже WCAG AA
  (`00-foundations.md` §3.2).
- Бейджи уровня сложности и «Мест нет» не передаются только цветом — дублируются текстом
  (`00-foundations.md` §3.2, §3.5).
- Тач-зона CTA и всех интерактивных элементов — ≥ 44–48 pt (`00-foundations.md` §3.1).
- Layout не ломается при системном увеличении шрифта; длинный список ингредиентов скроллится
  вместе с остальным контентом, не обрезается.

## 10. Критерии приёмки дизайна (Gherkin)

### Позитивные

```gherkin
AC-001: Просмотр карточки слота со свободными местами
  Дано клиент открыл карточку слота, у которого free_seats > 0
  Тогда видны: дата/время, программа (название, фото, ингредиенты/аллергены),
    явное поле уровня сложности, шеф, «Всего/Свободно мест», цена
  И кнопка «Записаться» активна

AC-002: Переход к оформлению записи
  Дано клиент на карточке слота со свободными местами
  Когда он нажимает «Записаться»
  Тогда происходит переход на SCR-004 «Оформление записи»

AC-003: Цена показана на карточке
  Дано клиент открыл карточку любого слота
  Тогда на экране показана цена класса и текст «Оплата на месте: наличные или перевод на карту.»

AC-004: Вместимость — точно из API
  Дано API вернул для слота значения total_seats и free_seats
  Тогда на карточке отображаются ровно эти значения без клиентского пересчёта
```

### Негативные

```gherkin
AC-005: Слот полностью заполнен
  Дано клиент открыл карточку слота, у которого free_seats = 0
  Тогда вместо числа свободных мест показан бейдж «Мест нет»
  И кнопка «Записаться» неактивна
  И тап по кнопке не переводит на SCR-004

AC-006: Попадание на карточку заполненного слота напрямую
  Дано клиент открывает карточку слота, который стал полностью заполненным
    после того, как клиент увидел его в списке (например, через историю браузера)
  Тогда карточка отображает актуальное состояние из API: free_seats = 0
  И кнопка «Записаться» недоступна — как в AC-005

AC-006a: Слот отменён студией
  Дано клиент открыл карточку слота со статусом «отменён студией»
  Тогда вместо блока вместимости показан бейдж «Отменён студией» (+ причина, если есть)
  И кнопка «Записаться» неактивна
  И тап по кнопке не переводит на SCR-004

AC-007: Ошибка загрузки карточки
  Дано сеть недоступна в момент открытия карточки слота
  Тогда показывается заглушка ошибки с текстом
    «Не удалось загрузить. Проверьте соединение и попробуйте снова.» и кнопкой «Обновить»
  И повторный тап «Обновить» повторяет запрос данных карточки
```

### Граничные

```gherkin
AC-008: Последнее свободное место
  Дано free_seats = 1
  Тогда вместимость отображается как «Всего N · Свободно 1»
  И кнопка «Записаться» активна (специального предупреждения «последнее место»
    требованиями не предусмотрено — не добавляется)
```

## 11. Решения

1. **Цена не дублируется на кнопке CTA.** В отличие от SCR-004, где кнопка показывает итоговую
   сумму брони (зависящую от выбора экипировки), на SCR-003 нет переменных, влияющих на цену —
   она уже показана отдельным крупным блоком (§6.6). Решение дизайнера, не зафиксировано
   отдельным FR.
2. **Точный текст рядом с неактивным CTA — допущение дизайнера**, см. §8: переиспользован
   бейдж «Мест нет» из FR-6a без дополнительной поясняющей строки на кнопке.
3. **Фото шефа не включено в макет** — домен и FR-6 описывают шефа только как имя (справочные
   данные слота), фото не упомянуто; не добавляется, чтобы не изобретать поле сверх требований.
4. **Карта/адрес студии отсутствуют** — в проекте нет геолокации/карты маршрута (в отличие от
   примера из другого проекта, использованного только для калибровки формата документа); FR-6
   этого поля не требует.
5. **Список ингредиентов/аллергенов — это состав блюда, не форма ввода аллергий клиента.** Сбор
   данных об аллергиях клиента явно вне скоупа (`02-domain.md` → «Вне скоупа»); экран этого не
   делает и не должен провоцировать восприятие как формы для клиента.
