## 1. Назначение и контекст

Экран даёт клиенту **контроль над своими записями** на кулинарные классы — отдельно
**предстоящими** и **прошедшими** (FR-13, US-9). Это домашняя точка вкладки «Мои записи»
таб-бара ([foundations §4.2](00-foundations.md#42-таб-бар-авторизованная-зона)): здесь клиент
проверяет, куда и когда он записан, в каком статусе находится запись, и переходит в детали, чтобы
при необходимости отменить бронь (UC-2, шаг 1) или (для прошедшего класса) оценить шефа
(FR-19, FR-20 — сама форма оценки не входит в этот документ, см. [SCR-007](SCR-007-chef-rating.md)).

Контекст использования — клиент открывает приложение «в свободную минуту», не обязательно стоя у
плиты ([foundations §1](00-foundations.md#1-продукт-и-аудитория)): экран должен позволять быстро,
одним взглядом, вспомнить параметры ближайшей записи и её статус, не углубляясь в детали.

---

## 2. Пользователь и цель

- **Роль:** Клиент (единственная роль приложения).
- **Цель пользователя:** увидеть свои бронирования — предстоящие и прошедшие — со статусом,
  параметрами слота и выбранным вариантом экипировки (FR-13); при необходимости перейти к деталям
  и отменить предстоящую запись (UC-2, шаг 1); при пустом списке — понять, как записаться на класс.
- **Покрываемая история:** [US-9](../2-requirements/user-stories.md) — «Как клиент, хочу видеть
  список своих бронирований, чтобы контролировать предстоящие и прошедшие классы». Критерий
  приёмки US-9: «Вижу свои записи (предстоящие и прошедшие) со статусом, параметрами слота и
  выбранным вариантом экипировки».

## 3. Навигация

### Входящие
- **Таб «Мои записи»** в таб-баре с любого корневого экрана АЗ, в т.ч. с
  [SCR-002 «Классы»](SCR-002-slot-list.md) ([foundations §4.2](00-foundations.md#42-таб-бар-авторизованная-зона)).
- Точка возврата после **успешной записи** — из шторки
  [BS-002 «Подтверждение записи»](BS-002-booking-success.md)
  ([foundations §4.4](00-foundations.md#44-карта-навигации-сквозная)).
- Обычный возврат по кнопке «Назад» с [SCR-006 «Детали брони»](SCR-006-booking-details.md) —
  список к этому моменту уже обновлён (статус изменился на SCR-006). **Уточнение:** после
  подтверждения отмены на [BS-003](BS-003-cancel-confirm.md) клиент попадает не сразу сюда, а
  сначала возвращается на SCR-006 (там показывается обновлённый статус и снек-итог отмены,
  см. [SCR-006 §3](SCR-006-booking-details.md#3-навигация)); на SCR-005 он возвращается только
  отдельным дальнейшим действием («Назад»).

### Исходящие
- **Тап по карточке записи** → [SCR-006 «Детали брони + отмена»](SCR-006-booking-details.md)
  (UC-2, шаг 1: «Клиент открывает «Мои бронирования» и выбирает предстоящую запись»). Это верно
  для карточки в любом статусе — деталь показывает полный набор параметров и, для предстоящей
  записи, действие «Отменить»; для прошедшей неотменённой записи — вход к оценке шефа
  (FR-19, FR-20, форма — [SCR-007](SCR-007-chef-rating.md), здесь не описывается).
- **Иконка аккаунта в хедере** → [BS-004 «Аккаунт»](BS-004-account.md)
  ([foundations §4.2](00-foundations.md#42-таб-бар-авторизованная-зона)).
- Переключение таб-бара → [SCR-002 «Классы»](SCR-002-slot-list.md).
- Из **Empty-состояния** — действие «Записаться на класс» → [SCR-002](SCR-002-slot-list.md).

### Каркас и таб-бар
Корневой экран вкладки — таб-бар виден; на дочернем экране деталей (SCR-006) таб-бар скрывается
по правилу [foundations §4.2](00-foundations.md#42-таб-бар-авторизованная-зона). Базовый каркас —
[foundations §4.1](00-foundations.md#41-базовый-каркас). Собственного нижнего фиксированного CTA у
экрана нет — FR-13 не описывает действий записи/отмены прямо со списка, они выполняются через
SCR-006.

## 4. Контент-инвентарь

FR-13 требует показывать по каждой записи: **статус**, **параметры слота** и **выбранный вариант
экипировки**. Ниже — как это раскладывается на карточку списка.

### 4.1 Карточка записи

| Поле | Источник требования | Назначение на карточке |
|------|---------------------|--------------------------|
| Статус записи | FR-13, FR-17 | Бейдж: «активна» / «отменена» / «Отменён студией» / «прошла». Текст + форма, не только цвет (см. §5 ниже и [foundations §3.2, §3.5](00-foundations.md#32-контраст-и-читаемость-nfr-1)). |
| Дата и время начала класса | FR-13 (параметры слота); тот же состав данных, что в FR-3/FR-6 | Крупно, основной ориентир при опознании записи. |
| Программа (название) | FR-13 (параметры слота); домен → «Ключевые объекты домена: Программа» | UI-лейбл «Программа» ([foundations §9](00-foundations.md#9-глоссарий-дизайн-специфичный-срез)). |
| Шеф (имя) | FR-13 (параметры слота); домен → «Ключевые объекты домена: Шеф» | Второстепенная строка, помогает узнать запись. |
| Вариант экипировки | FR-13, FR-8 | «Своя экипировка» / «Прокатная экипировка» — лейблы из [foundations §6](00-foundations.md#6-tone-of-voice-и-общая-микрокопия). |
| Цена класса | *Решение дизайнера*, опирается на FR-23 (цена показывается клиенту при записи) | Показана для справки/напоминания — не требование FR-13 буквально, но согласуется с тем, что цену клиент уже видел на SCR-003/SCR-004/BS-002. |

> **Карточка — сводка, а не полная деталь.** FR-13 не перечисляет исчерпывающий состав
> «параметров слота» для списка (в отличие от FR-6, где для карточки слота их состав чётко задан:
> фото, ингредиенты/аллергены, уровень сложности, всего/свободно мест). *Решение дизайнера*: на
> карточке списка — только минимум, достаточный для опознания записи (дата/время, программа, шеф,
> экипировка, статус); уровень сложности, фото, ингредиенты/аллергены, вместимость слота и полная
> причина отмены студией — в деталях на [SCR-006](SCR-006-booking-details.md). Это ограничивает
> плотность карточки согласно [foundations §3.4](00-foundations.md#34-плотность-и-сетка).

### 4.2 Причина отмены студией на карточке
Полный текст причины (шаблон «Класс отменён студией. Причина: [reason]...» —
[foundations §6](00-foundations.md#6-tone-of-voice-и-общая-микрокопия)) длинный для карточки
списка. *Решение дизайнера*: на карточке — только бейдж «Отменён студией»; полная причина — на
[SCR-006](SCR-006-booking-details.md), куда ведёт тап по карточке (FR-17 требует, чтобы причина
была доступна клиенту, но не предписывает конкретный экран для неё).

## 5. Статусы записи и бейджи

Источники: FR-13 (статус — обязательный элемент карточки), FR-15/FR-16 (правила отмены клиентом),
FR-17 (отмена студией с причиной), UC-2 (потоки и постусловия).

| Отображаемый статус | Когда показывается | Источник |
|---|---|---|
| **«активна»** | Запись не отменена, время начала класса ещё не наступило. | FR-13 |
| **«отменена»** | Запись отменена клиентом — независимо от того, была отмена ранней (≥24 ч) или поздней (<24 ч). | FR-15, FR-16 (см. допущение ниже) |
| **«Отменён студией»** | Класс отменён студией (форс-мажор); бронь не удаляется из истории. | FR-17 |
| **«прошла»** | Запись не отменена, время начала класса уже наступило (класс состоялся). | FR-13 (раздел «прошедшие») |

> **Допущение дизайнера — ранняя/поздняя отмена не различаются как отдельные бейджи.**
> FR-15 и FR-16 описывают отмену клиентом как одно действие с разным сопутствующим предупреждением
> в зависимости от момента (до/после 24 часов), а не как два разных исхода записи; штрафа нет ни в
> одном случае. Постусловие UC-2 упоминает «отменена» / «отменена, поздно» / «Отменён студией» —
> формулировка допускает прочтение с отдельным статусом для поздней отмены. Здесь принято более
> экономное прочтение: **разница между ранней и поздней отменой — это предупреждение, показанное
> клиенту в момент самой отмены** (на [SCR-006](SCR-006-booking-details.md) /
> [BS-003](BS-003-cancel-confirm.md), тексты — [foundations §6](00-foundations.md#6-tone-of-voice-и-общая-микрокопия)), а не постоянный статус, который нужно
> заново различать при каждом последующем просмотре списка. Список показывает единый бейдж
> «отменена». Если это прочтение расходится с намерением заказчика, требует уточнения перед
> реализацией.

> **Допущение дизайнера — «прошла» как производное состояние.** Ни FR-13, ни use cases не
> описывают отдельное поле статуса API для «прошедшего» класса — домен вводит понятие «прошла» как
> временную характеристику относительно даты старта («после посещения — оценить шефа», домен →
> «Процесс»). Бейдж «прошла» на экране вычисляется дизайн-логикой (запись не отменена И время
> начала уже в прошлом), а не читается как отдельное персистентное значение статуса из API.

## 6. Разделение списка на «Предстоящие» / «Прошедшие»

> **Решение дизайнера, не описанное явно в FR-13.** FR-13 требует показывать «предстоящие и
> прошедшие» записи в одном списке, но не указывает механизм разделения. Решение — разбить список
> на две группы (сегмент-переключатель вверху **или** sticky-секции — выбор конкретного паттерна за
> дизайнером на этапе макета), чтобы клиент не пролистывал прошедшие записи в поисках ближайшей.

Правило отнесения:
- **Предстоящие** — только записи со статусом «активна» (время начала ещё не наступило).
- **Прошедшие** — записи со статусом «прошла», а также **все отменённые записи** («отменена» и
  «Отменён студией») независимо от того, наступило время начала или нет.

> *Обоснование решения:* отменённая запись, даже если формальное время старта класса ещё впереди,
> не требует от клиента никаких действий по подготовке — смысловая ценность строки «Предстоящие»
> (что мне нужно помнить/готовиться) для неё утрачена; логичнее держать её в «Прошедшие» рядом с
> другими завершившимися для клиента бронированиями. Это решение дизайнера по аналогии с общим
> принципом «список предстоящих = то, что ещё требует внимания клиента», отдельно заказчиком не
> зафиксировано.

По умолчанию открыта группа **«Предстоящие»**. Прошедшие сортируются по убыванию даты начала
(последние по времени — сверху); предстоящие — по возрастанию (ближайшие — сверху), как и в
списке слотов (FR-3).

## 7. Структура и иерархия

Сверху вниз: **хедер** (заголовок + иконка аккаунта) → **переключатель/секции** → **скролл-список
карточек** → **таб-бар**.

```
┌──────────────────────────────────┐
│  Мои записи                  (⚙) │  ← хедер: заголовок + иконка аккаунта → BS-004
│  [ Предстоящие | Прошедшие ]     │  ← сегмент-переключатель (или sticky-секции, см. §6)
├──────────────────────────────────┤
│ ┌──────────────────────────────┐ │
│ │ сб, 4 июля · 18:00           │ │  ← дата/время старта (крупно)
│ │ Программа: Итальянская       │ │
│ │ кухня для начинающих         │ │
│ │ Шеф: Анна                    │ │
│ │ Своя экипировка               │ │
│ │ 2500 ₽              [активна]│ │  ← бейдж статуса (текст + форма)
│ └──────────────────────────────┘ │
│ ┌──────────────────────────────┐ │
│ │ вс, 5 июля · 12:00            │ │
│ │ Программа: Для опытных       │ │
│ │ Шеф: Игорь                    │ │
│ │ Прокатная экипировка          │ │
│ │ 3200 ₽              [активна]│ │
│ └──────────────────────────────┘ │
├──────────────────────────────────┤
│    Классы     ·     Мои записи   │  ← таб-бар (foundations §4.2)
└──────────────────────────────────┘
```

Пример из группы «Прошедшие» (иллюстрация бейджа отмены студией):

```
│ ┌──────────────────────────────┐ │
│ │ пн, 22 июня · 19:00           │ │
│ │ Программа: Для опытных       │ │
│ │ Шеф: Игорь                    │ │
│ │ Прокатная экипировка          │ │
│ │ 3200 ₽      [Отменён студией]│ │  ← причина — на SCR-006
│ └──────────────────────────────┘ │
```

> Названия программ, имена шефов и цены на макетах — иллюстративные примеры для калибровки
> плотности карточки, а не зафиксированный заказчиком контент (справочные данные приходят из API,
> `02-domain.md` → «Ключевые объекты домена»).

Типографическая иерархия — [foundations §3.3](00-foundations.md#33-типографическая-иерархия-уровни-не-шрифты):
дата/время старта — самый контрастный элемент карточки (основной ориентир опознания записи),
следом статус-бейдж и цена.

## 9. Состояния экрана

Применяется сквозной паттерн [foundations §5](00-foundations.md#5-сквозной-паттерн-состояний-экрана)
(Loading → Content / Empty / Error). UC-2 не описывает отдельных исключений для этапа открытия
списка (шаг 1) — его исключения (E1–E3) относятся к самой отмене на SCR-006/BS-003, не к загрузке
этого экрана. Поэтому ошибки/пустые состояния списка регулируются целиком общим паттерном
[foundations §5](00-foundations.md#5-сквозной-паттерн-состояний-экрана), без экранно-специфичных
исключений сверх него.

| Состояние | Содержимое / Поведение |
|-----------|--------------------------|
| **Loading** | Скелетоны карточек записи в форме будущего контента, без блокирующего спиннера (P7, [foundations §5](00-foundations.md#5-сквозной-паттерн-состояний-экрана)). |
| **Content** | Список карточек, разделённый на «Предстоящие» (только активные) и «Прошедшие» (прошедшие + все отменённые), см. §6. |
| **Empty (нет записей вообще)** | Заглушка «У вас пока нет записей» + действие «Записаться на класс» → [SCR-002](SCR-002-slot-list.md). Формулировка — *допущение дизайнера в духе tone of voice* ([foundations §6](00-foundations.md#6-tone-of-voice-и-общая-микрокопия)): текст не зафиксирован заказчиком ни в брифе, ни в FR-13/US-9. |
| **Empty в группе «Предстоящие»** (есть только прошедшие) | Короткая подсказка «Нет предстоящих записей» + действие «Записаться на класс»; группа «Прошедшие» при этом доступна и не скрывается — предсказуемость навигации, решение дизайнера по аналогии с общим правилом «не скрывать структуру интерфейса», отдельно заказчиком не зафиксировано. |
| **Error** | Заглушка ошибки нейтральным тоном + кнопка «Обновить»; текст — общий сетевой/серверный из [foundations §6](00-foundations.md#6-tone-of-voice-и-общая-микрокопия) («Не удалось загрузить. Проверьте соединение и попробуйте снова.» / «Что-то пошло не так. Попробуйте ещё раз позже.»). |

## 10. Валидации и микрокопия

### Доступ к данным
Клиент видит **только свои записи** (P5, [foundations §8.2](00-foundations.md#82-безопасность-данных-в-ui-nfr-8)).
Чужих бронирований, контактов или данных управления расписанием на экране нет.

### Тексты бейджей статуса
См. таблицу §5: «активна», «отменена», «Отменён студией», «прошла».

### Лейблы экипировки и оплаты
«Своя экипировка» / «Прокатная экипировка», текст об офлайн-оплате — переиспользуются из
[foundations §6](00-foundations.md#6-tone-of-voice-и-общая-микрокопия) и
[§9 (глоссарий)](00-foundations.md#9-глоссарий-дизайн-специфичный-срез), не вводятся заново.

### Empty-состояние
- Заголовок: «У вас пока нет записей» *(допущение дизайнера, формулировка не зафиксирована
  заказчиком)*.
- Действие: «Записаться на класс» → [SCR-002](SCR-002-slot-list.md).
- Тон — спокойный, без давления, в духе [foundations §6](00-foundations.md#6-tone-of-voice-и-общая-микрокопия) (принцип P6).

### Оценка шефа — только вход, не форма
Для прошедшей неотменённой записи возможность оценить шефа (FR-19, FR-20) доступна **только через
переход в детали** [SCR-006](SCR-006-booking-details.md) — на самой карточке списка нет отдельной
кнопки «Оценить». Сама форма оценки описана в [SCR-007](SCR-007-chef-rating.md) и не входит в
этот документ.

## 11. Доступность и mobile-first

Базовые правила — [foundations §7](00-foundations.md#7-доступность) и
[§3](00-foundations.md#3-структурные-токены-без-бренда). Экранная специфика:

- Клиент часто открывает список «в свободную минуту» с телефона
  ([foundations §1](00-foundations.md#1-продукт-и-аудитория)) — карточка должна опознаваться
  беглым взглядом: дата/время и статус — самые заметные элементы.
- Бейдж статуса дублирует смысл текстом и формой — считывается и при плохом освещении экрана, и
  при дальтонизме ([foundations §3.2](00-foundations.md#32-контраст-и-читаемость-nfr-1)).
- Формат даты/времени старта — день недели + число + время, без двусмысленности.
- Переключатель/секции и карточки — с доступными именами для screen reader (например, «Запись на
  класс «Итальянская кухня для начинающих», 4 июля, статус: активна»); фокус-состояния обязательны
  ([foundations §7](00-foundations.md#7-доступность)).
- Тач-зона всей карточки ≥ минимального размера ([foundations §3.1](00-foundations.md#31-тач-зоны-и-размеры)).

## 12. Критерии приёмки дизайна

```gherkin
Сценарий: AC-001 Разделение на предстоящие и прошедшие
  Дано у клиента есть записи с разными статусами и датами начала
  Когда он открывает экран «Мои бронирования»
  Тогда записи разделены на группы «Предстоящие» и «Прошедшие»
  И по умолчанию отображается группа «Предстоящие»
  И в «Предстоящие» попадают только записи со статусом «активна»

Сценарий: AC-002 Отменённая запись — в «Прошедшие» независимо от даты начала
  Дано запись отменена клиентом или студией, а время начала класса ещё не наступило
  Когда клиент открывает экран
  Тогда эта запись показана в группе «Прошедшие», а не в «Предстоящие»

Сценарий: AC-003 Отображение статуса записи текстом и формой
  Дано запись имеет статус «активна», «отменена», «Отменён студией» или «прошла»
  Когда клиент видит её карточку
  Тогда на карточке показан бейдж с соответствующим текстом статуса
  И статус передан текстом и формой (не только цветом)

Сценарий: AC-004 Параметры слота и вариант экипировки на карточке
  Дано у клиента есть запись на класс
  Когда он видит карточку записи в списке
  Тогда на карточке показаны дата/время начала, программа, шеф и выбранный вариант экипировки
  («Своя экипировка» или «Прокатная экипировка»)

Сценарий: AC-005 Отмена студией — причина доступна через детали
  Дано запись имеет статус «Отменён студией»
  Когда клиент тапает по карточке такой записи
  Тогда открываются детали брони (SCR-006) с указанием причины отмены

Сценарий: AC-006 Пустой список
  Дано у клиента нет ни одной записи
  Когда он открывает экран
  Тогда показана заглушка «У вас пока нет записей»
  И доступно действие «Записаться на класс», ведущее к списку слотов (SCR-002)

Сценарий: AC-007 Пусто только в «Предстоящих»
  Дано у клиента есть только прошедшие/отменённые записи
  Когда он открывает группу «Предстоящие»
  Тогда показана подсказка «Нет предстоящих записей» с действием «Записаться на класс»
  И группа «Прошедшие» остаётся доступной и не скрывается

Сценарий: AC-008 Только свои записи
  Дано в системе есть записи разных клиентов
  Когда клиент открывает «Мои бронирования»
  Тогда он видит исключительно собственные записи
  И на экране нет чужих или административных данных

Сценарий: AC-009 Переход к деталям и отмене
  Дано на экране показана карточка предстоящей записи
  Когда клиент тапает по карточке
  Тогда открывается экран деталей брони (SCR-006), где доступно действие «Отменить»

Сценарий: AC-010 Обновление списка
  Дано экран в состоянии Content
  Когда клиент выполняет жест pull-to-refresh
  Тогда список перезапрашивается и отображает актуальные записи и статусы

Сценарий: AC-011 Ошибка загрузки списка
  Дано запрос списка бронирований завершился ошибкой
  Когда клиент открывает экран
  Тогда показана заглушка ошибки нейтральным тоном с кнопкой «Обновить»
  И список не отображается, пока клиент не повторит запрос
```

## 13. Решения

1. **Разделение на «Предстоящие» / «Прошедшие»** (сегмент или sticky-секции) — решение дизайнера,
   FR-13 требует показывать оба среза, но не задаёт механизм (§6).
2. **Отменённые записи (клиентом или студией) — всегда в «Прошедшие»**, независимо от даты начала
   класса; «Предстоящие» — только записи со статусом «активна» (§6).
3. **Ранняя/поздняя отмена клиентом — один бейдж «отменена»**, без отдельного статуса для поздней
   отмены; различие выражено только предупреждением в момент самой отмены (FR-15, FR-16), на
   SCR-006/BS-003, а не постоянным статусом в списке (§5, отмечено как допущение дизайнера).
4. **«Прошла» — производное состояние**, вычисляемое по дате начала для неотменённой записи, а не
   отдельное значение статуса API (§5, допущение дизайнера).
5. **Карточка — сводка**: дата/время, программа, шеф, вариант экипировки, статус, цена; полная
   деталь (уровень сложности, фото, ингредиенты/аллергены, вместимость, причина отмены студией) —
   на [SCR-006](SCR-006-booking-details.md) (§4.1–4.2).
6. **Оценка шефа — только вход через детали**, без отдельной кнопки на карточке списка; форма
   оценки — вне этого документа, см. [SCR-007](SCR-007-chef-rating.md) (§10).
7. **Empty-состояние — авторская формулировка**, не зафиксированная заказчиком, выдержана в тоне
   [foundations §6](00-foundations.md#6-tone-of-voice-и-общая-микрокопия) (§9).
8. **Ошибки/исключения экрана — целиком по общему паттерну** [foundations §5](00-foundations.md#5-сквозной-паттерн-состояний-экрана);
   UC-2 не описывает специфичных исключений для просмотра списка (его E1–E3 относятся к самой
   отмене на SCR-006/BS-003) (§9).
