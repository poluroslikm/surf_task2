## 1. Назначение и контекст

Экран — стартовая точка сценария записи: список предстоящих кулинарных классов (слотов) на
ближайшие 7 дней по умолчанию, с возможностью расширить период через фильтр дат (FR-3, FR-4,
R-027). Это **корневая вкладка** таб-бара («Классы»), первый экран, который клиент видит после
успешного входа (см. [00-foundations.md §4.2](00-foundations.md#42-таб-бар-авторизованная-зона)).

Экран открывает цепочку записи: список → карточка слота → оформление → подтверждение, где путь
до подтверждения ориентирован на **≤ 3 экрана** (NFR-2). SCR-002 — первый из них.

> **Важная оговорка по FR-6a (заполненные слоты).** Требование показывать полностью заполненные
> слоты в списке (с пометкой «мест нет», не скрывая) — это **допущение аналитика, не
> подтверждённое заказчиком** (см. врезку над FR-6a в `functional-requirements.md` и аналогичную
> оговорку в UC-3). Приоритет FR-6a — **Should**, а не Must. Экран спроектирован из этого
> допущения; при уточнении заказчика (например, если решат скрывать заполненные слоты) раздел
> «Заполненный слот» ниже потребует пересмотра. Подробнее — раздел «Решения», п. 1.

## 2. Пользователь и цель

- **Роль:** Клиент (единственная роль приложения, [00-foundations.md §1](00-foundations.md)).
- **Контекст:** клиент открывает приложение с телефона, чаще заранее и не «здесь и сейчас», чтобы
  выбрать подходящий класс на неделю вперёд (домен: «на неделю вперёд» — горизонт расписания,
  R-027).
- **Цель пользователя:** быстро увидеть, какие классы предстоят, и понять — куда есть смысл
  записаться (дата подходит, есть свободные места, программа и шеф интересны) — US-2.
- **Цель системы:** отдать актуальный, отсортированный список слотов из API за нужный период,
  не подменяя и не пересчитывая числа мест (FR-12, P3).

## 4. Контент-инвентарь

### 4.1 Карточка слота в списке (превью, не полный контент — полный набор смотри в
[SCR-003](SCR-003-slot-card.md), FR-6)

| Поле | Источник / правило | Примечание |
|------|--------------------|------------|
| Дата и время начала | API, поле слота | Формат: день недели, число месяца, время («Чт, 3 июля · 18:00») |
| Программа (название) | API, поле слота | Без фото/ингредиентов — это уже уровень SCR-003 (FR-6), в списке только название (FR-3) |
| Уровень сложности | API, явное поле | Отображается как отдельная метка/тег (FR-3, FR-6 — «явное поле, а не только текст в названии») |
| Шеф | API, поле слота | Имя шефа |
| Всего мест / свободно мест | API, поля слота | Оба числа берутся из ответа сервера как есть, не пересчитываются (FR-12, P3) |
| Бейдж «Мест нет» | Производится из `свободно = 0` | Заменяет/дополняет число свободных мест при заполненном слоте (FR-6a) |
| Бейдж «Отменён студией» | Производится из статуса слота, приходящего из API | Заменяет число свободных мест; CTA записи недоступен (FR-18) — см. §6.2a |

> Цена класса в списке **не показывается** — FR-3 не включает цену в состав полей карточки
> списка; цена появляется на [SCR-003](SCR-003-slot-card.md) и при оформлении записи (US-14,
> FR-23). Это осознанное сужение состава полей именно для превью в списке.

| Элемент | Назначение |
|---------|------------|
| Заголовок хедера «Классы» | Идентифицирует раздел (корневая вкладка) |

| Иконка аккаунта (хедер) | Точка входа в [BS-004](BS-004-account.md) ([00-foundations.md §4.2](00-foundations.md)) |

(фикс.). Сверху вниз:

1. **Хедер** (фикс.) — заголовок «Классы», иконка фильтра, иконка аккаунта.

## 6. Компоненты и поведение

### 6.1 Карточка слота (список)

- Вся площадь карточки — единая тач-зона (≥ 44–48 pt, [foundations §3.1](00-foundations.md)).
- Порядок полей внутри карточки (сверху вниз): дата/время → программа + уровень сложности → шеф
  → свободно/всего мест **или** бейдж «Мест нет» (FR-3).
- Число свободных мест — крупное, контрастное (важное число, [foundations §3.2](00-foundations.md)).
- Уровень сложности — отдельный визуальный тег/метка (не растворён в названии программы), чтобы
  клиент мог сравнивать классы по сложности на глаз, не открывая карточку (FR-6, применительно
  к составу полей превью).
- Тап по карточке → [SCR-003](SCR-003-slot-card.md), независимо от того, есть ли свободные
  места (см. §6.2 и «Решения», п. 2).

### 6.2 Заполненный слот (`свободно = 0`)

- Показывается в общем списке на своей позиции по времени начала — **не скрывается и не
  переносится** в конец (FR-6a).
- Число свободных мест заменяется бейджем **«Мест нет»** — состояние не передаётся только
  цветом, дублируется текстом и иконкой ([foundations §3.2, §3.5](00-foundations.md)).
- Карточка визуально притушена (например, сниженный контраст фона/иконок при сохранении
  читаемости текста ≥ WCAG AA), чтобы явно отличаться от доступных слотов, не переставая быть
  разборчивой.
- Карточка **остаётся кликабельной** — тап открывает [SCR-003](SCR-003-slot-card.md) в
  режиме просмотра; действие записи там недоступно (кнопка «Записаться» неактивна при
  отсутствии свободных мест — это поведение уже описано на стороне SCR-003/SCR-004 и не
  дублируется здесь). Обоснование решения — §11, п. 2.
- На карточке **нет** элементов записи (ни кнопки, ни свайпа) — «без возможности записи»
  реализовано отсутствием CTA записи именно в списке, а не блокировкой перехода к деталям
  (FR-6a).

### 6.2a Слот, отменённый студией

- Отдельное от «Мест нет» состояние карточки: слот, который студия отменила (форс-мажор — срыв
  поставки продуктов, поломка оборудования — или решение организатора), приходит из API с
  соответствующим статусом (FR-17, FR-18, `02-domain.md` → «Отмена класса студией»).
- Число свободных мест заменяется бейджем **«Отменён студией»** — визуально отличным от бейджа
  «Мест нет» (другая иконка/текст, не только цвет — [foundations §3.2, §3.5](00-foundations.md)),
  чтобы клиент понимал разницу между «мест не осталось» и «класс не состоится».
- Карточка **остаётся кликабельной** для просмотра деталей на [SCR-003](SCR-003-slot-card.md) —
  по аналогии с §6.2, п. «Карточка остаётся кликабельной».
- CTA «Записаться» на этом слоте недоступен **и в списке, и на карточке слота (SCR-003), и при
  прямом переходе на оформление записи (SCR-004)** — повторная запись на отменённый студией слот
  запрещена для всех клиентов (FR-18). Это состояние — прямая причина, по которой SCR-006 может
  ссылаться на данный раздел как на место, где правило FR-18 реализовано (см.
  [SCR-006 §6.6](SCR-006-booking-details.md#66-запрет-повторной-записи-на-слот-отменённый-студией-для-полноты-картины)).

### 6.3 Индикатор активного фильтра

- Показывается **только**, если выбранный период отличается от дефолтных «ближайшие 7 дней»
  (UC-3, основной поток vs A1).
- Вид — компактный чип под хедером с текстом диапазона дат и кнопкой сброса (✕).
- Тап по ✕ на чипе — мгновенный сброс к дефолтному периоду без открытия BS-001 (быстрый путь для
  UC-3 A2), список перезагружается на «ближайшие 7 дней».
- Иконка фильтра в хедере визуально помечена (например, точкой-индикатором), когда фильтр
  активен, — чтобы факт нестандартного периода был заметен даже до прокрутки к чипу.
- Открытие [BS-001](BS-001-date-filter.md) по тапу на иконку фильтра — единственная точка
  задания произвольного периода (FR-4); содержимое самой шторки не описывается в этом документе.

### 6.4 Список и сортировка

- Сортировка — по времени начала, ближайшие слоты выше (FR-3); порядок сохраняется при смене
  периода фильтром (UC-3, шаг 4).
- **Pull-to-refresh** — обновляет список при том же активном периоде (фильтр не сбрасывается
  сам по себе), согласно общему паттерну состояний ([foundations §5](00-foundations.md)).

### 6.5 Таб-бар и хедер

- Таб-бар и хедер — без специфики экрана сверх [foundations §4](00-foundations.md); иконка
  аккаунта одинакова на SCR-002 и SCR-005.

## 7. Состояния экрана

Базовый паттерн — [foundations §5](00-foundations.md). Специфика SCR-002:

| Состояние | Содержимое / Поведение |
|-----------|--------------------------|
| **Loading** | Скелетон из 4–6 карточек-заглушек в форме карточки слота (не пустой экран), ориентир отклика — NFR-5 (< 2–3 с). |
| **Content** | Список карточек за активный период (дефолт — 7 дней, либо выбранный фильтром), отсортированный по времени начала; включает заполненные слоты с бейджем «Мест нет» (§6.2) и слоты, отменённые студией, с бейджем «Отменён студией» (§6.2a). |
| **Empty** | Текст «Пока нет доступных классов» (FR-5) + контекстное действие (см. §8) — по умолчанию для любого периода, где слотов нет (UC-3, E1). |
| **Error** | Заглушка ошибки + кнопка «Обновить» (текст — [foundations §6](00-foundations.md): «Не удалось загрузить. Проверьте соединение и попробуйте снова.») — для сбоя первичной загрузки (UC-3, E2). |
| **Обновление (pull-to-refresh) — ошибка** | Контент экрана сохраняется, ошибка сообщается снеком (не Error-заглушкой), по правилу [foundations §6.3](00-foundations.md). |

Индикатор активного фильтра (§6.3) — не отдельное состояние экрана, а элемент, накладывающийся
на состояние Content/Empty/Error в зависимости от того, какой период выбран.

## 8. Валидации и микрокопия

> Экран не содержит форм ввода — «валидации» здесь относятся к обработке периода и сетевых
> сбоев при загрузке списка (UC-3, исключения E1–E2). Тон — нейтральный, без вины пользователя
> ([foundations §6](00-foundations.md)).

| Код | Условие | Точный текст | Поведение UI |
|-----|---------|--------------|--------------|
| **E1** (UC-3) | В выбранном периоде нет ни одного слота | **«Пока нет доступных классов»** (FR-5) | Empty state. Если активен дефолтный период — дополнительная подсказка и кнопка **«Выбрать другой период»** (открывает [BS-001](BS-001-date-filter.md)). Если активен нестандартный период (фильтр применён) — дополнительно кнопка **«Сбросить фильтр»** (возврат к 7 дням). |
| **E2** (UC-3) | Сетевой сбой при первичной загрузке списка | **«Не удалось загрузить. Проверьте соединение и попробуйте снова.»** ([foundations §6](00-foundations.md)) | Error-заглушка + кнопка **«Обновить»** (§7, [foundations §5](00-foundations.md)); повторный запрос — за тот же активный период. |
| **E3** (доп., pull-to-refresh) | Сетевой сбой при обновлении списка потягиванием вниз (контент уже отображён) | **«Не удалось загрузить. Проверьте соединение и попробуйте снова.»** (тот же текст, что и в E2 — переиспользуется по [LOGIC-005](../5-mobile-app-spec/09_Логики/LOGIC-005_Паттерн-состояний-экрана.md), отдельная формулировка не заводится) | Снек ошибки, список на экране не очищается ([foundations §6.3](00-foundations.md)). |

**Лейблы и сквозные тексты:**
- Бейдж заполненного слота: **«Мест нет»** (FR-6a).
- Числа свободных/всего мест — подставляются из данных слота, не хардкодятся (P3, FR-12).
- Диапазон дат в чипе фильтра — читаемый формат «3–20 июля» (без года, если период в пределах
  текущего года — уточняется дизайнером при вёрстке BS-001).

## 9. Доступность и mobile-first

База — [foundations §3, §7](00-foundations.md). Специфика экрана:

- Карточка слота — единая тач-зона ≥ 44–48 pt; между карточками — отступ, исключающий случайный
  тап по соседней (§3.1).
- Бейдж «Мест нет» и притушенный стиль заполненной карточки — состояние читается не только по
  цвету, но по тексту и иконке (§3.2, §3.5).
- Доступное имя карточки для screen reader включает ключевые поля и статус, например:
  «Класс «Паста ручной работы», новичковый уровень, чт 3 июля 18:00, шеф Анна, свободно 4 из 8
  мест» / «…, мест нет» — по аналогии с примером в [foundations §7](00-foundations.md).
- Индикатор активного фильтра имеет доступное имя, поясняющее и диапазон, и возможность сброса
  («Фильтр дат активен: 3–20 июля. Сбросить фильтр»).
- Layout — одна колонка, не ломается при системном увеличении шрифта; таб-бар и хедер остаются
  на своих местах при скролле контента.

```gherkin
AC-001: Список по умолчанию — ближайшие 7 дней, сортировка по времени
  Дано клиент открывает вкладку «Классы» без ранее заданного фильтра
  Тогда показываются слоты на ближайшие 7 дней от текущего момента
  И слоты отсортированы по времени начала, ближайшие — выше
  И карточки содержат дату/время, программу, уровень сложности, шефа, всего/свободно мест

AC-002: Карточка отображает полный набор полей превью
  Дано в списке есть слот со свободными местами
  Тогда карточка показывает дату/время начала, программу, явный тег уровня сложности, шефа,
    число свободных и общее число мест
  И числа мест совпадают со значениями из ответа API (без клиентского пересчёта)

AC-003: Заполненный слот виден с бейджем и доступен для просмотра
  Дано в выбранном периоде есть слот, где свободно мест = 0
  Тогда слот остаётся на своей позиции в списке (по времени начала)
  И вместо числа свободных мест показан бейдж «Мест нет»
  И тап по карточке открывает карточку слота (SCR-003) в режиме просмотра
  И на карточке списка нет элемента для записи на этот слот

AC-003a: Слот, отменённый студией, виден с бейджем и без возможности записи
  Дано в выбранном периоде есть слот со статусом «отменён студией»
  Тогда слот остаётся на своей позиции в списке (по времени начала)
  И вместо числа свободных мест показан бейдж «Отменён студией» (не «Мест нет»)
  И тап по карточке открывает карточку слота (SCR-003) в режиме просмотра
  И ни на карточке списка, ни на SCR-003, ни на SCR-004 запись на этот слот невозможна (FR-18)

AC-004: Расширение периода через фильтр
  Дано клиент открывает фильтр дат и выбирает период длиннее 7 дней
  Тогда список обновляется, показывая слоты за выбранный период
  И под хедером появляется чип с обозначением активного периода

AC-005: Сброс фильтра
  Дано активен нестандартный период (не дефолтные 7 дней)
  Когда клиент нажимает ✕ на чипе фильтра
  Тогда список возвращается к дефолтным ближайшим 7 дням
  И чип фильтра исчезает
```

### Негативные

```gherkin
AC-006: E1 — нет слотов в выбранном периоде
  Дано выбранный период (дефолтный или заданный фильтром) не содержит ни одного слота
  Тогда показывается empty state с текстом «Пока нет доступных классов»
  И предлагается действие (выбрать другой период / сбросить фильтр) в зависимости от того,
    активен ли нестандартный фильтр

AC-007: E2 — сетевой сбой при первичной загрузке
  Дано клиент впервые открывает вкладку «Классы» и запрос к API завершается ошибкой сети
  Тогда показывается Error-заглушка с текстом «Не удалось загрузить. Проверьте соединение и
    попробуйте снова.» и кнопкой «Обновить»
  И повторный тап на «Обновить» повторяет запрос за тот же период

AC-008: E3 — сетевой сбой при pull-to-refresh
  Дано список уже отображает данные
  Когда клиент тянет список вниз для обновления и запрос завершается ошибкой
  Тогда показывается снек «Не удалось загрузить. Проверьте соединение и попробуйте снова.»
  И ранее показанный список остаётся на экране без изменений
```

### Граничные

```gherkin
AC-009: Слот на границе периода (7-й день)
  Дано слот стартует ровно на границе интервала «текущий момент + 7 дней»
  Тогда слот включён в список без фильтра наравне с более ранними слотами

AC-010: Свободно мест = 0 (полностью граничное значение)
  Дано слот имеет свободно мест = 0 и всего мест > 0
  Тогда карточка показывает бейдж «Мест нет», а не число «0»
  И слот не скрывается из списка (FR-6a)

AC-011: Свободно мест = всего мест (класс полностью свободен)
  Дано слот имеет свободно мест = всего мест
  Тогда карточка показывает оба числа как есть, без бейджа «Мест нет»
```

## 11. Решения

1. **Заполненные слоты не скрываются, а показываются с бейджем «Мест нет» — по FR-6a.** Это
   требование помечено в `functional-requirements.md` как **допущение аналитика, не
   подтверждённое заказчиком** (приоритет Should, а не Must). Решение принято, чтобы клиент видел
   полную картину расписания и не путался, почему часть классов «пропала» из списка — но при
   уточнении заказчика (например, если решат скрывать заполненные слоты) этот раздел (§6.2, AC-003,
   AC-010) потребует пересмотра.
2. **Карточка заполненного слота остаётся кликабельной для просмотра деталей.** FR-6a запрещает
   только «возможность записи» с заполненного слота, но не просмотр. Оставить карточку
   кликабельной — осознанное дизайнерское решение (явно не подтверждено заказчиком отдельно, как
   и сам FR-6a): (а) избегает «мёртвой» неинтерактивной карточки в списке, что хуже для
   восприятия, чем явно неактивная кнопка записи на самой карточке слота; (б) согласуется с тем,
   что [SCR-003](SCR-003-slot-card.md) уже штатно поддерживает состояние «свободных мест нет» с
   заблокированной кнопкой «Записаться» (см. документ [SCR-004](SCR-004-booking.md), где отдельно
   отмечено: «кнопка на SCR-003 неактивна, если свободных мест нет»). Альтернатива (полностью
   некликабельная карточка) осталась бы валидной, но потребовала бы дублирования логики бейджа
   на превью, тогда как полный контекст (программа, ингредиенты) остаётся ценным для клиента даже
   без возможности записи прямо сейчас.
3. **Цена не входит в состав полей карточки списка.** FR-3 явно перечисляет состав полей превью
   (дата/время, программа, уровень сложности, шеф, всего/свободно мест) без цены; цена появляется
   на [SCR-003](SCR-003-slot-card.md) (FR-6) и при оформлении записи (US-14, FR-23). Решение —
   не добавлять цену на этот экран сверх заявленного состава полей.
4. **Слоты, которые уже начались (прошли по времени старта относительно текущего момента),
   в список не попадают.** Явно не оговорено в FR/UC, но логически следует из формулировки
   «предстоящих слотов… на ближайшие 7 дней от текущего момента» (FR-3, R-027) — допущение
   дизайнера для однозначной трактовки границы периода: интервал **закрытый с обеих сторон**
   `[текущий момент, текущий момент + 7 дней]` — слот ровно на границе «+7 дней» **включён**
   в дефолтную выдачу (см. AC-009). Открыта только левая сторона по смыслу («уже начавшиеся
   слоты не попадают»), правая — включительно.
5. **Индикатор активного фильтра — чип с быстрым сбросом, а не только пометка на иконке.**
   Решение упрощает UC-3 A2 («сброс фильтра дат») до одного тапа без открытия BS-001, снижая
   число действий (согласуется с общим духом NFR-2 — минимизировать шаги в основных сценариях).
