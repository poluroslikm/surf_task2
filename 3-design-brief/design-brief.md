## Цель и контекст

«Шеф-стол» заменяет ручную запись на кулинарные классы через WhatsApp и Google-таблицу,
устраняя двойные записи и путаницу по выходным (BR-1). Клиент самостоятельно просматривает
слоты, записывается, выбирает экипировку, отменяет записи, получает напоминания и оценивает
шефа после класса (BR-2, BR-5, BR-6). Платформа — **веб-приложение (PWA)**, открываемое в
браузере на телефоне, без публикации в App Store / Google Play (NFR-3). Только русский язык
интерфейса (NFR-4).

**Скоуп — только роль «Клиент».** Шеф и владелец работают через существующую
инфраструктуру/админку и в приложение не входят. Слоты, программы, шефы — read-only из API;
оплата — офлайн (наличные / перевод), приложение лишь показывает цену и фиксирует запись
(FR-23).

**Ключевые решения, принятые в требованиях (влияют на дизайн):**
- **Одна запись — одно место** (допущение аналитика, см. `functional-requirements.md`, врезка
  перед FR-7) — в оформлении записи нет счётчика количества мест.
- **Экипировка (FR-8, FR-9):** выбор «своя / прокатная» — бинарный, прокатный фонд не
  ограничен, никогда не блокирует запись.
- **Вместимость — только из API (FR-12):** числа свободных/всего мест не пересчитываются и не
  хардкодятся на клиенте.
- **Отмена (FR-14–FR-16):** порог 24 часа; после порога отмена всё равно проходит, но с
  предупреждением, без штрафа.
- **Оценка шефа необязательна** (FR-19): 1–5 звёзд + опциональный комментарий, только для
  прошедших неотменённых классов (FR-20).
- **Лист ожидания, программа лояльности, сбор данных об аллергиях, онлайн-оплата, резервный
  канал уведомлений — вне скоупа** (см. `business-requirements.md` → «Ограничения верхнего
  уровня»). В экранах этих функций нет.

Подробнее о продукте, аудитории, дизайн-принципах и контексте использования —
[00-foundations.md §1–2](00-foundations.md#1-продукт-и-аудитория).

## Роли и их задачи

| Роль | Что делает в приложении |
| :-- | :-- |
| **Клиент** | Просматривает и фильтрует слоты, записывается, выбирает экипировку, видит цену, отменяет записи, видит свои бронирования, получает напоминания, оценивает шефа после класса. |
| Шеф / Владелец-администратор | **Не входят в приложение** — работают через существующую инфраструктуру/админку (вне скоупа). |

## Сквозные правила (foundations)

Единые для всех экранов требования — в [00-foundations.md](00-foundations.md): дизайн-принципы,
структурные токены, каркас и навигация (таб-бар, bottom sheet), паттерн состояний, tone of
voice и микрокопия, доступность (с оговоркой об отсутствии формального НФТ), сквозные функции
(уведомления, безопасность данных).

**Почему не 1-в-1 как в примере (`primer`):** в этом проекте нет отдельного экрана «Профиль» —
функциональные требования не описывают просмотр/редактирование профиля клиента, только вход по
email/паролю (FR-1, FR-2). Выход из аккаунта вынесен в компактную шторку BS-004 вместо полного
экрана, чтобы не добавлять функциональность сверх того, что зафиксировано в требованиях. Также
нет экрана/шторки карты маршрута — в этом домене нет геопривязанного маршрута (в отличие от
примера с SUP-прогулками).

## Открытые вопросы, влияющие на дизайн

- **Групповая запись (несколько мест в одной брони)** — не разрешено требованиями
  (`02-domain.md` → «Открытые вопросы», п.1). SCR-004 спроектирован из допущения «одна запись —
  одно место»; при уточнении заказчика потребует пересмотра.
- **Соотношение «12 столов ↔ лимит 8 человек»** (`02-domain.md`, открытый вопрос №2) не влияет
  на дизайн напрямую — клиент показывает только присланное значение вместимости (FR-12), но
  стоит держать в уме при финальной проверке макета с реальными данными API.
