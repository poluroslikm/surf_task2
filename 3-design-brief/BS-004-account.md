# Требования на дизайн · BS-004 · Аккаунт / выход

**ID:** BS-004 · **Тип:** Bottom Sheet · **Зона:** АЗ · **Приоритет:** не хардкожен в MoSCoW функциональных требований — минимальная необходимость (см. §1) · **Статус:** Черновик · **Версия:** 0.1 · **Дата:** 2026-07-02

> Самая простая шторка в реестре: единственное действие — **выход из аккаунта**, с одним
> read-only полем (email) для ориентира «под кем я вошёл». Это **не экран «Профиль»** — просмотра
> или редактирования данных клиента здесь нет и не будет (см. §1). Сквозные правила (каркас
> шторок, состояния, микрокопия, доступность) — в [`00-foundations.md`](00-foundations.md); здесь
> **не дублируются**, а уточняются под кейс.

**Источники:**
[Foundations §4.2, §4.3, §6.1](00-foundations.md) ·
[design-brief.md — реестр экранов, строка BS-004](design-brief.md#реестр-экранов-и-шторок) ·
[Функциональные требования — FR-1, FR-2](../2-requirements/functional-requirements.md#авторизация-клиент)

## 1. Назначение и контекст

**Оговорка по источнику (важно).** В функциональных требованиях нет отдельного FR на просмотр/
редактирование профиля клиента и нет отдельного FR на выход из аккаунта. Авторизация описана
только FR-1 (регистрация по email/паролю) и FR-2 (вход по email/паролю). Возможность выйти из
аккаунта — это **минимальная необходимость**, логически вытекающая из самого факта наличия
входа (нельзя авторизовать вход и никогда не дать возможности выйти), а **не подтверждённое
заказчиком требование** с собственным ID. Это прямо зафиксировано в
[`00-foundations.md` §4.2](00-foundations.md#42-таб-бар-авторизованная-зона):

> «Профиля как отдельного раздела нет... Единственная связанная необходимость — выход из
> аккаунта, минимально вытекающая из самого факта авторизации (FR-2), но отдельным FR не
> описанная.»

Следствия для этой шторки:
- Она заменяет собой полноценный экран «Профиль» — но **не является** им и не должна
  разрастаться до него.
- Единственное действие внутри — **выход из аккаунта**.
- Единственная показываемая информация о пользователе — **email текущего клиента, read-only**
  (минимально нужен, чтобы клиент понимал, под каким аккаунтом он вошёл, — допущение дизайнера,
  не отдельное требование).

**Что сознательно не входит** (нет опоры в требованиях, добавлять нельзя):
- Редактирование имени, телефона, пароля.
- Смена email.
- Удаление аккаунта.
- Любые другие настройки (уведомления, язык и т.п.) — управления уведомлениями в MVP нет
  ([00-foundations.md §8.1](00-foundations.md#81-напоминания-и-уведомления-fr-21-fr-22)).

Контекст использования — mobile-first, в свободную минуту с телефона
([§1 foundations](00-foundations.md#1-продукт-и-аудитория)): шторка открывается редко, задача —
не мешать основному сценарию (просмотр слотов / своих записей), поэтому она короткая и не
требует отдельного экрана.

## 2. Пользователь и цель

- **Роль:** Клиент (единственная роль приложения).
- **Цель пользователя:** убедиться, под каким аккаунтом он вошёл, и при необходимости выйти
  (например, чтобы уступить телефон другому человеку или войти под другим email).
- **Цель продукта:** дать минимально необходимую точку выхода из сессии, не создавая иллюзию
  полноценного раздела «Профиль», которого нет в требованиях.

## 3. Навигация

### Входящие
- Иконка аккаунта в хедере [SCR-002 «Список слотов»](SCR-002-slot-list.md).
- Иконка аккаунта в хедере [SCR-005 «Мои бронирования»](SCR-005-my-bookings.md).

(См. карту навигации — [00-foundations.md §4.4](00-foundations.md#44-карта-навигации-сквозная).)

### Исходящие
- **Подтверждение выхода** → выполнение выхода (сброс сессии) → закрытие шторки → переход на
  [SCR-001 «Регистрация/Вход»](SCR-001-auth.md). Снек **не показывается** — обратная связь сам
  переход на экран входа ([00-foundations.md §6.1](00-foundations.md#61-каталог-снеков-успеха)).
- **Отмена / закрытие без действия** (кнопка «Отмена», крестик) → закрытие шторки → возврат на
  текущий экран (SCR-002 или SCR-005) **без изменений**, сессия сохраняется.

### Свойства шторки
Общие правила bottom sheet — [§4.3 foundations](00-foundations.md#43-bottom-sheet-шторки-bs-001bs-004).
Уточнение для BS-004 как **критичного подтверждения**: закрытие — **только явной кнопкой**
(«Отмена» / крестик). Закрытие по тапу вне шторки и по swipe-вниз **недоступно** — выход из
аккаунта достаточно значим (обрывает сессию), чтобы не позволять закрыть/подтвердить его
случайным жестом.

## 4. Контент-инвентарь

| Блок | Содержимое | Источник |
|------|------------|----------|
| Заголовок (шаг 1) | «Аккаунт» | — |
| Email | Email текущего клиента, **read-only**, без иконки редактирования | Допущение дизайнера — минимально нужно для ориентира «под кем я вошёл»; не отдельное требование |
| Кнопка-триггер | «Выйти из аккаунта» | Минимальная необходимость (см. §1) |

## 5. Структура и иерархия

Шторка состоит из **двух шагов внутри одной шторки** (без перехода на отдельный экран):

**Шаг 1 — по умолчанию:**
1. Грабер.
2. Заголовок — «Аккаунт».
3. Строка email (read-only).
4. Кнопка «Выйти из аккаунта» (внизу).

```
┌─────────────────────────────┐
│            ▔▔▔▔              │  ← грабер
│                              │
│  Аккаунт                     │  ← заголовок
│                              │
│  ┌─────────────────────────┐ │
│  │ client@example.com      │ │  ← email, read-only
│  └─────────────────────────┘ │
│                              │
│  [  Выйти из аккаунта  ]     │  ← кнопка-триггер
└─────────────────────────────┘
```

**Шаг 2 — подтверждение** (заменяет содержимое шторки после тапа по «Выйти из аккаунта», высота
шторки не меняет масштаб взаимодействия — тот же bottom sheet):
1. Грабер.
2. Заголовок — «Выйти из аккаунта?».
3. Действия (низ шторки): деструктивная «Выйти» и безопасная «Отмена».

```
┌─────────────────────────────┐
│            ▔▔▔▔              │  ← грабер
│                              │
│  Выйти из аккаунта?          │  ← заголовок подтверждения
│                              │
│  [       Выйти        ]      │  ← деструктивная (менее акцентная)
│  [       Отмена        ]     │  ← безопасная (акцентная/основная)
└─────────────────────────────┘
```

> Иерархия следует [§3.3 foundations](00-foundations.md#33-типографическая-иерархия-уровни-не-шрифты):
> заголовок → основной контент (email или вопрос) → кнопки.

## 6. Компоненты и поведение

### 6.1 Email (read-only)
- Отображается как обычный текст/строка, без обводки под ввод и без иконки карандаша — это не
  поле формы, редактирование недоступно (нет опоры в FR).
- Если email длинный — переносится/обрезается по правилам платформы, не ломает разметку.

### 6.2 Кнопка-триггер «Выйти из аккаунта» (шаг 1)
- Одна кнопка, крупная тач-зона ([§3.1 foundations](00-foundations.md#31-тач-зоны-и-размеры)).
- Тап переключает содержимое шторки на шаг 2 (подтверждение), сама шторка не закрывается и не
  открывается заново — плавный переход контента.

### 6.3 Кнопки подтверждения (шаг 2)
- **«Выйти»** — деструктивное действие (завершает сессию). Визуально отлично и **менее
  акцентно** относительно безопасной кнопки (стиль `destructive` — цвет задаёт дизайнер, без
  хардкода); смысл дублируется текстом, не только цветом.
- **«Отмена»** — безопасное действие, основное/акцентное, не «ловушка» (не маскируется под
  деструктивное).
- Обе кнопки — крупные тач-зоны, с отступом, исключающим случайный тап не по цели
  ([§3.1 foundations](00-foundations.md#31-тач-зоны-и-размеры)).

### 6.4 Состояние выполнения (loading)
После тапа по «Выйти» — кнопка переходит в состояние загрузки/блокировки, повторные тапы
исключены (защита от повторного запроса, аналогично BS-003 §6.3). Остальные элементы шторки
заблокированы до ответа (сброс сессии — операция клиента, не обязательно требует сетевого
ответа, но трактуется по тому же паттерну, что и остальные действия приложения).

## 7. Состояния экрана

Применяется сквозной паттерн [§5 foundations](00-foundations.md#5-сквозной-паттерн-состояний-экрана);
специфика BS-004 (шторка простая, полноценной загрузки контента с сервера не требует — email уже
известен из текущей сессии):

| Состояние | Содержимое / Поведение |
|-----------|------------------------|
| **Шаг 1 — по умолчанию** | Email + кнопка «Выйти из аккаунта»; кнопка активна. |
| **Шаг 2 — подтверждение** | Вопрос «Выйти из аккаунта?» + кнопки «Выйти» / «Отмена»; обе активны. |
| **Выполнение выхода** | Кнопка «Выйти» в состоянии загрузки; элементы заблокированы. |
| **Ошибка выхода** | Общий сетевой текст из [foundations §6](00-foundations.md#6-tone-of-voice-и-общая-микрокопия) + возможность повтора; шторка не закрывается, сессия не сбрасывается. |

## 8. Валидации и микрокопия

Точные тексты:

- **Заголовок (шаг 1):** «Аккаунт»
- **Кнопка-триггер:** «Выйти из аккаунта»
- **Заголовок (шаг 2):** «Выйти из аккаунта?»
- **Деструктивная кнопка:** «Выйти»
- **Безопасная кнопка:** «Отмена»
- **Ошибка выполнения выхода** (общий сетевой текст, [§6 foundations](00-foundations.md#6-tone-of-voice-и-общая-микрокопия)):
  > «Не удалось выполнить. Проверьте соединение и повторите.»

> Снек об успешном выходе **не предусмотрен** — обратная связь сам переход на SCR-001
> ([00-foundations.md §6.1](00-foundations.md#61-каталог-снеков-успеха)).

## 9. Доступность и mobile-first

Базовые требования — [§7 foundations](00-foundations.md#7-доступность) и
[§3.1](00-foundations.md#31-тач-зоны-и-размеры). Специфика BS-004:

- **Защита от случайного нажатия:** выход из аккаунта требует двух явных тапов (триггер на шаге
  1 → подтверждение на шаге 2); закрытие по бэкдропу/swipe недоступно на обоих шагах.
- Безопасная кнопка «Отмена» **не маскируется** под деструктивную, имеет понятную акцентную
  форму и доступное имя для screen reader.
- Различие деструктивного/безопасного действия передаётся текстом + формой, **не только цветом**
  ([§3.2 foundations](00-foundations.md#32-контраст-и-читаемость-nfr-1)).
- Email как read-only текст имеет доступное имя для screen reader (например, «Текущий аккаунт:
  client@example.com»).

## 10. Критерии приёмки дизайна

```gherkin
Функция: Аккаунт / выход (BS-004)

  Сценарий: AC-001 Открытие шторки и просмотр email
    Дано клиент авторизован и находится на SCR-002 или SCR-005
    Когда клиент тапает иконку аккаунта в хедере
    Тогда открывается шторка BS-004 с текущим email клиента (read-only)
    И доступна одна кнопка «Выйти из аккаунта»

  Сценарий: AC-002 Переход к подтверждению выхода
    Дано шторка BS-004 открыта на шаге "Аккаунт"
    Когда клиент нажимает «Выйти из аккаунта»
    Тогда содержимое шторки переключается на шаг подтверждения «Выйти из аккаунта?»
    И доступны кнопки «Выйти» и «Отмена»

  Сценарий: AC-003 Отмена выхода
    Дано шторка BS-004 открыта на шаге подтверждения
    Когда клиент нажимает «Отмена» (или закрывает шторку явной кнопкой)
    Тогда шторка закрывается
    И клиент остаётся на текущем экране (SCR-002 или SCR-005) без изменений сессии

  Сценарий: AC-004 Подтверждение выхода
    Дано шторка BS-004 открыта на шаге подтверждения
    Когда клиент нажимает «Выйти»
    Тогда показывается состояние выполнения без повторных тапов
    И при успехе сессия завершается, шторка закрывается, клиент попадает на SCR-001
    И снек не показывается

  Сценарий: AC-005 Защита от случайного нажатия
    Дано шторка BS-004 открыта (любой шаг)
    Когда клиент тапает вне шторки или свайпает вниз
    Тогда шторка не закрывается и действие не выполняется
    И закрытие возможно только явной кнопкой

  Сценарий: AC-006 Ошибка выполнения выхода
    Дано клиент нажал «Выйти»
    Когда операция завершается ошибкой
    Тогда показано нейтральное сообщение об ошибке с возможностью повтора
    И сессия не сбрасывается, шторка остаётся открытой
```

## 11. Решения

| # | Решение |
|---|---------|
| 1 | **Нет отдельного экрана «Профиль»** — вместо него компактная шторка с одним действием, чтобы не изобретать функциональность сверх FR-1/FR-2 (см. [00-foundations.md §4.2](00-foundations.md#42-таб-бар-авторизованная-зона)). |
| 2 | **Email показан как read-only**, без формы редактирования — минимально необходимая информация, не отдельное требование (допущение дизайнера). |
| 3 | **Подтверждение реализовано как второй шаг внутри той же шторки**, а не отдельная вложенная шторка — экономит переход, но сохраняет обязательный явный тап для деструктивного действия. |
| 4 | **Тап вне шторки и swipe-вниз отключены на обоих шагах** — выход из аккаунта трактуется как критичное подтверждение наравне с отменой записи (BS-003), см. [00-foundations.md §4.3](00-foundations.md#43-bottom-sheet-шторки-bs-001bs-004). |
| 5 | **Снек после выхода не показывается** — обратная связь сам переход на SCR-001, зафиксировано в [00-foundations.md §6.1](00-foundations.md#61-каталог-снеков-успеха). |
