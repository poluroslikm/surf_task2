# Use cases / сценарии

> Этап 3 (Спецификация). Сценарии использования. Обязательно прорабатываем **альтернативные
> потоки** и **исключения** — ИИ любит их забывать.
>
> **Скоуп: клиентское веб-приложение и API для него.** Слоты, программы и шефы поступают из
> существующего бэкенда через API (read-only).

> **Допущение по составу записи.** `1-elicitation/02-domain.md` → «Открытые вопросы по домену»,
> п. 1, оставляет открытым, может ли одна запись включать несколько мест. Сценарии ниже написаны
> из допущения **«одна запись — одно место»** как единственно однозначно выводимого варианта; при
> уточнении заказчика (групповая запись) UC-1 потребует пересмотра.
>
> **Допущение по видимости заполненных слотов.** Ни бриф, ни ответы заказчика явно не описывают,
> скрываются ли полностью заполненные слоты из списка. Ниже принято допущение, что такие слоты
> **остаются видимыми** с пометкой «мест нет» — см. также `functional-requirements.md`.

## UC-1: Запись на класс

- **Актёр:** Клиент
- **Предусловия:** клиент авторизован; в расписании есть хотя бы один слот со свободным местом.
- **Триггер:** клиент хочет забронировать место на кулинарном классе.

### Основной поток
1. Клиент открывает список слотов (см. UC-3).
2. Открывает карточку выбранного слота и просматривает параметры: дата/время, программа (фото,
   ингредиенты/аллергены), уровень сложности, шеф, всего/свободно мест.
3. Указывает вариант экипировки: своя или рабочий набор напрокат.
4. Подтверждает запись.
5. Система проверяет наличие свободного места на сервере (актуальные данные; гарантия «0 двойных
   записей» обеспечивается бэкендом, R-004) и создаёт запись, уменьшая число свободных мест
   слота.
6. Система показывает подтверждение записи с ценой класса (оплата офлайн, FR-23) и добавляет
   запись в «Мои бронирования».

### Альтернативные потоки
- **A1. Экипировка — рабочий набор напрокат.** На шаге 3 клиент выбирает прокат; прокатный фонд
  не ограничен (Q-001), доступность экипировки не влияет на исход записи (FR-9).
- **A2. У клиента уже есть активная запись на это же или пересекающееся время.** Запись всё равно
  проходит — приложение не ограничивает число одновременных активных бронирований клиента
  (Q-013, FR-11).

### Исключения / ошибки
- **E1. Места закончились к моменту подтверждения (гонка бронирований).** Система отклоняет
  запись и показывает сообщение «мест больше нет», клиент возвращается к списку слотов; листа
  ожидания нет (Q-012, FR-10).
- **E2. Слот отменён студией к моменту подтверждения.** Запись отклоняется, клиенту сообщается,
  что класс отменён (R-008).
- **E3. Сетевой сбой при подтверждении записи.** Приложение показывает сообщение об ошибке и
  предлагает повторить попытку; неопределённый результат не должен приводить к двойной записи —
  гарантия обеспечивается бэкендом (R-004, NFR-6).

### Постусловия
- При успехе: создана запись клиента; свободные места слота уменьшены; запись видна в «Мои
  бронирования».
- При ошибке: состояние слота не изменилось.

## UC-2: Отмена записи

- **Актёр:** Клиент
- **Предусловия:** у клиента есть предстоящая запись; класс ещё не начался.
- **Триггер:** клиент решает отменить участие, либо студия отменяет класс целиком (срыв поставки
  продуктов, поломка оборудования или иной форс-мажор) — само действие отмены выполняется вне
  клиентского приложения существующей инфраструктурой, клиент видит только результат (см. A2).

### Основной поток (отмена клиентом)
1. Клиент открывает «Мои бронирования» и выбирает предстоящую запись.
2. Нажимает «Отменить».
3. Система определяет время до начала класса (источник истины — сервер).
4. **Ранняя отмена (≥24 часов до начала):** запись отменяется без предупреждений, место
   освобождается обратно в слот.
5. Система показывает клиенту подтверждение отмены.

### Альтернативные потоки
- **A1. Поздняя отмена (менее 24 часов до начала).** На шаге 4 запись всё равно отменяется (не
  блокируется), но клиенту показывается предупреждение о том, что продукты на класс уже
  закуплены; денежного штрафа нет (Q-007, FR-16).
- **A2. Класс отменён студией (форс-мажор).** Бронь клиента не удаляется, а переходит в статус
  «Отменён студией» с указанием причины; клиенту приходит push-уведомление. При недоступности
  push клиент увидит статус при следующем открытии приложения — резервного канала нет (Q-014,
  FR-17, FR-22). Повторная запись на этот слот запрещена (R-008, FR-18).

### Исключения / ошибки
- **E1. Класс уже начался или прошёл.** Кнопка отмены недоступна.
- **E2. Запись уже отменена (повторная попытка).** Повторная отмена не выполняется, показывается
  текущий статус записи.
- **E3. Сетевой сбой при отмене.** Приложение показывает сообщение об ошибке и предлагает
  повторить попытку; финальный статус запись получает по ответу сервера.

### Постусловия
- Статус записи обновлён («отменена» / «отменена, поздно» / «Отменён студией»).
- При ранней отмене клиентом место снова доступно другим клиентам.
- При отмене студией — повторная запись на этот слот запрещена для всех клиентов.

## UC-3: Просмотр и фильтрация списка слотов

- **Актёр:** Клиент
- **Предусловия:** клиент авторизован.
- **Триггер:** клиент хочет найти подходящий кулинарный класс.

### Основной поток
1. Клиент открывает список слотов. По умолчанию показываются слоты на **ближайшие 7 дней** от
   текущего момента (R-027), отсортированные по времени начала (ближайшие выше).
2. Заполненные слоты отображаются в списке с пометкой «мест нет» (FR-6a; см. допущение о
   видимости заполненных слотов выше) без возможности записи.
3. (Опц.) Клиент задаёт более длинный период через фильтр дат.
4. Система показывает список с ключевыми параметрами каждого слота (дата/время, программа,
   уровень сложности, шеф, всего/свободно мест), сохраняя сортировку по времени начала.

### Альтернативные потоки
- **A1. Более длинный период.** Клиент расширяет диапазон дат фильтром — список обновляется по
  заданному диапазону (R-027, FR-4).
- **A2. Сброс фильтра дат.** Клиент убирает фильтр — список возвращается к дефолтной выдаче
  (ближайшие 7 дней).

### Исключения / ошибки
- **E1. Нет слотов в выбранном периоде.** Показывается empty state «Пока нет доступных классов»
  (R-027, FR-5).
- **E2. Сетевой сбой при загрузке списка.** Приложение показывает сообщение об ошибке и позволяет
  повторить загрузку.

### Постусловия
- Клиент видит актуальный список предстоящих слотов; данные расписания не изменяются.

## UC-4: Оценка шефа после посещения класса

- **Актёр:** Клиент
- **Предусловия:** у клиента есть запись на слот, время начала которого уже прошло; запись не
  была отменена (ни клиентом, ни студией).
- **Триггер:** клиент хочет поделиться впечатлением о прошедшем классе.

### Основной поток
1. Клиент открывает прошедшую (завершённую) запись в «Мои бронирования».
2. Система предлагает клиенту оценить шефа этого класса.
3. Клиент указывает оценку по шкале 1–5 звёзд.
4. (Опц.) Клиент добавляет текстовый комментарий.
5. Клиент подтверждает оценку.
6. Система сохраняет оценку, связанную с записью и шефом.

### Альтернативные потоки
- **A1. Клиент не оставляет оценку.** Оценка не обязательна (Q-002, FR-19) — клиент может
  закрыть экран, не оценивая класс; запись остаётся без оценки.
- **A2. Оценка без комментария.** Клиент указывает только звёзды, оставляя текстовое поле
  пустым — оценка сохраняется.

### Исключения / ошибки
- **E1. Запись была отменена (клиентом или студией) или класс ещё не начался.** Возможность
  оценки недоступна для этой записи (FR-20).
- **E2. Оценка для этой записи уже поставлена.** Повторная отправка оценки недоступна;
  возможность редактирования ранее поставленной оценки заказчиком не оговаривалась — это
  допущение по умолчанию, подлежит уточнению.
- **E3. Сетевой сбой при сохранении оценки.** Приложение показывает сообщение об ошибке и
  предлагает повторить попытку.

### Постусловия
- При успехе: оценка сохранена и связана с записью и шефом.
- При отказе от оценки: запись остаётся без оценки, доступность повторной попытки оценить
  сохраняется, пока это не сделано.
