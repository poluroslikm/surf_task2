# Функциональные требования

> Этап 3 (Спецификация). Что система должна делать (функции, поведение).
> Не путать с нефункциональными ([non-functional-requirements.md](non-functional-requirements.md)).

Приоритеты по MoSCoW: **Must** — обязательно к запуску; **Should** — желательно при наличии
времени; **Could** — приятное дополнение; **Won't** — вне границ текущего скоупа.

## Границы скоупа (клиентское приложение + API)

Скоуп — **клиентское веб-приложение (PWA) и API для него**. Функции Шефа и
Владельца/Администратора обеспечиваются **существующей инфраструктурой** и здесь не описываются
(`1-elicitation/02-domain.md` → «Границы скоупа», ответ Q-019).

Следствия для скоупа:

- **Слоты (классы), программы и шефы приходят из существующего бэкенда через API.** Экранов
  создания и редактирования расписания в клиентском приложении нет.
- **Вместимость слота (правило «не более 8 человек на классы со сложным оборудованием при 12
  столах») — логика бэкенда.** Клиент только отображает значение вместимости из API, не
  дублирует правило (Q-018).
- **Прокатный фонд рабочих наборов не ограничен** — недоступность экипировки не может быть
  причиной отказа в записи (Q-001).
- Явно **вне скоупа**: онлайн-оплата, лист ожидания (waitlist), программа лояльности, сбор
  данных об аллергиях, резервный канал уведомлений (SMS/email) — см.
  `business-requirements.md` → «Ограничения верхнего уровня».

> **Открытый вопрос, влияющий на этот раздел.** `1-elicitation/02-domain.md` → «Открытые вопросы
> по домену», п. 1: не уточнено, может ли одна запись включать несколько мест (например, клиент +
> гость). Ни бриф, ни снятые вопросы это не разрешают. Ниже функциональные требования
> сформулированы из допущения **«одна запись — одно место»** как единственно однозначно
> выводимого варианта; групповая запись (несколько мест в одной брони) в это допущение не входит
> и требует отдельного уточнения у заказчика перед реализацией.

## Просмотр и фильтрация слотов (клиент)

| ID | Требование | Приоритет | Источник |
| :-- | :-- | :-- | :-- |
| FR-3 | Система должна показывать клиенту список предстоящих слотов на **ближайшие 7 дней** от текущего момента по умолчанию, отсортированный по времени начала (ближайшие выше). Состав данных по каждому слоту: дата/время начала, программа, уровень сложности, шеф, всего мест, свободно мест. | Must | R-027 (brief-cooking.md); 02-domain.md → «Ограничения» |
| FR-4 | Система должна позволять клиенту просмотреть более длинный период через фильтр дат (за пределами дефолтных 7 дней). | Must | R-027 |
| FR-5 | Система должна показывать понятный empty state «Пока нет доступных классов», если в выбранном периоде нет ни одного слота. | Must | R-027; 02-domain.md → «Ограничения» |
| FR-6 | Система должна показывать клиенту карточку слота с полным составом параметров: дата/время начала, программа (название, фото блюда, список ингредиентов/аллергенов блюда), уровень сложности программы (явное поле, а не только текст в названии), шеф, всего/свободно мест. | Must | Q-003; Q-004; 02-domain.md → «Ключевые объекты домена» |
| FR-6a | Система должна показывать в списке полностью заполненные слоты (свободных мест нет) с пометкой «мест нет», не скрывая их из списка, и без возможности записи на них. **Требование основано на допущении аналитика, не подтверждено заказчиком** — см. пояснение ниже. | Should | Допущение аналитика (см. блок-цитату ниже); ни бриф, ни `01-questions.md` этого не разрешают |

> **Видимость заполненных слотов — допущение.** Ни бриф, ни ответы заказчика не уточняют, должны
> ли полностью заполненные слоты скрываться из списка или оставаться видимыми с пометкой «мест
> нет». FR-6a и UC-3 приняты из допущения, что заполненные слоты **остаются видимыми** в
> списке — это следует из общей задачи «клиент сам видит расписание и не путается», но точка
> явно не подтверждена заказчиком и подлежит уточнению перед реализацией.

## Запись на класс (клиент)

| ID | Требование | Приоритет | Источник |
| :-- | :-- | :-- | :-- |
| FR-7 | Система должна позволять клиенту записаться на выбранный слот при наличии свободного места. Одна запись резервирует одно место (см. допущение в разделе «Границы скоупа»). | Must | Бриф; 02-domain.md → «Процесс» |
| FR-8 | Система должна позволять клиенту при записи указать вариант экипировки: своя (фартук и набор ножей) или рабочий набор напрокат. | Must | Бриф (абзац «со своим фартуком... или ему дать напрокат»); Q-001 |
| FR-9 | Система не должна ограничивать запись доступностью рабочих наборов напрокат: прокатный фонд не лимитирован и не может быть самостоятельной причиной отказа в записи. | Must | Q-001 |
| FR-10 | Система должна отклонять попытку записи, если к моменту подтверждения свободных мест не осталось (гонка бронирований), и показывать клиенту понятное сообщение «мест больше нет» без предложения листа ожидания. | Must | Q-012; 02-domain.md → «Граница серверной интеграции» (R-004) |
| FR-11 | Система должна позволять клиенту иметь несколько активных записей одновременно, в том числе на пересекающееся по времени слоты, без ограничений со стороны клиента. | Must | Q-013 |
| FR-12 | Система должна отображать вместимость слота (всего/свободно мест) значением, полученным из API, не проверяя и не дублируя на клиенте бизнес-правило «не более 8 человек на классы со сложным оборудованием». | Must | Q-018; 02-domain.md → «Ограничение вместимости (8/12) — логика бэкенда» |

> **Граница со серверной интеграцией (R-004).** Гарантия «0 двойных записей» обеспечивается на
> стороне существующего бэкенда атомарной проверкой свободных мест при бронировании. Детали
> серверной реализации (транзакции, маппинг сущностей) — вне скоупа клиентского приложения;
> клиент полагается на ответ API и корректно обрабатывает отказ (FR-10). (`brief-cooking.md`,
> `1-elicitation/02-domain.md` → «Технические / платформенные»)
