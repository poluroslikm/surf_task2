# Нефункциональные требования

> Этап 3 (Спецификация). Как система должна работать: производительность, надёжность,
> безопасность, доступность, удобство. Частая ошибка ИИ — путать ФТ и НФТ: если формулировка про
> то, что система умеет — это ФТ ([functional-requirements.md](functional-requirements.md)); если
> про качество исполнения существующей функции — это НФТ.

> **Скоуп — клиентское веб-приложение (PWA) и API для него.** НФТ относятся к клиентскому
> приложению; функции Шефа/Владельца и админка обеспечиваются существующей инфраструктурой.

| ID | Категория | Требование | Приоритет | Источник |
| :-- | :-- | :-- | :-- | :-- |
| NFR-1 | Удобство использования | Интерфейс клиента — mobile-first, рассчитан на использование с телефона (в т.ч. на кухне). | Высокий | Бриф (абзац «пользоваться будут в основном с телефона») |
| NFR-2 | Удобство использования | Запись на слот выполняется за минимальное число шагов (ориентир ≤ 3 экранов до подтверждения — предположение, точный порог заказчиком не назван), чтобы клиент мог записаться быстро и без обучения. | Высокий | Бриф (общая задача «люди сами записываются, а я только смотрел») |
| NFR-3 | Платформа / Совместимость | Клиент — веб-приложение (PWA), открываемое в браузере на телефоне, без публикации в App Store / Google Play. Целевые браузеры — актуальные версии мобильных браузеров (предположение; конкретный список заказчиком не задан). | Высокий | Q-010 |
| NFR-4 | Локализация | Интерфейс поддерживает только русский язык в MVP; мультиязычность не требуется. | Высокий | Q-011 |
| NFR-5 | Производительность | Список слотов и подтверждение записи откликаются быстро — ориентир: время отклика < 2–3 с при нормальной сети (предположение, точный порог заказчиком не назван). | Средний | Бриф (общая задача снятия хаоса записи) |
| NFR-6 | Надёжность / Целостность данных | Гарантия «0 двойных записей» и корректная обработка отказа при исчерпании мест (FR-10) обеспечиваются на стороне существующего бэкенда; клиент не создаёт условий для двойной записи (не отправляет повторные подтверждения без обработки предыдущего ответа сервера). | Высокий | 02-domain.md → «Граница серверной интеграции» (R-004) |
| NFR-7 | Безопасность / Приватность | Учётные данные клиента (email, пароль) и персональные данные передаются и хранятся защищённым способом, исключающим их раскрытие третьим лицам. Конкретный протокол передачи и механизм хранения (сессии, токены, пароли) заказчиком не задан и остаётся на усмотрение реализации API — вне скоупа анализа требований. | Высокий | Q-008 (авторизация по email/паролю подразумевает защиту передаваемых данных — предположение по уровню, не по конкретному механизму) |
| NFR-8 | Безопасность / Разграничение доступа | Клиент видит и управляет только своими записями и своими оценками; доступ к чужим бронированиям и к данным управления (расписание, шефы, программы) через клиентский API запрещён. | Высокий | 02-domain.md → «Пользователи и роли», «Границы скоупа» |
| NFR-9 | Доступность (Availability) | Сервис доступен в часы работы студии в течение сезона; целевой уровень доступности — порядка 99% в рабочее время (предположение, точный порог заказчиком не назван). | Средний | 02-domain.md → «Проект» |
| NFR-10 | Ограничения проекта | Решение должно быть запущено к началу нового сезона — срок ~2 месяца от текущего момента; точная дата запуска — открытый вопрос. | Высокий (Must) | Бриф; Q-009 |
| NFR-11 | Ограничения проекта | Бюджет ограничен: применяется MVP-подход, приоритет — простота реализации; онлайн-оплата, лист ожидания, программа лояльности и сбор данных об аллергиях откладываются на развитие после MVP. | Высокий | Бриф; 02-domain.md → «Границы скоупа» |
