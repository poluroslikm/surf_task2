# API «Шеф-стол» — как читать эту спецификацию

> Клиентский API кулинарной студии «Шеф-стол» (роль «Клиент», см.
> [`1-elicitation/02-domain.md`](../1-elicitation/02-domain.md) → «Границы скоупа»). Спецификация
> разбита по доменам вместо одного файла — так проще поддерживать и просматривать по частям.

## Структура

```
api/
├── auth/       {api.yaml, models.yaml}   — регистрация, вход, выход
├── slots/      {api.yaml, models.yaml}   — список и карточка классов (read-only)
├── bookings/   {api.yaml, models.yaml}   — запись, отмена, оценка шефа
├── push/       {api.yaml, models.yaml}   — регистрация Web Push подписок устройств
└── common/     {models.yaml}             — общая схема ошибки, сквозные параметры
```

Каждый домен — самостоятельный OpenAPI-фрагмент (`api.yaml` — пути и операции, `models.yaml` —
схемы данных), связанный с остальными через относительные `$ref` (например,
`../common/models.yaml#/components/schemas/Error`).

**Почему 5 доменов, а не 4 или 6+.** В отличие от похожих проектов, здесь нет отдельного домена
«профиль» (в требованиях нет просмотра/редактирования профиля клиента — только вход, FR-1/FR-2)
и нет отдельного домена «программы»/«шефы» (это read-only данные, вложенные в `Slot`, не
самостоятельно просматриваемые ресурсы, `1-elicitation/02-domain.md`). Оценка шефа — не отдельный
домен, а под-путь `bookings/{bookingId}/rating`, так как оценка всегда привязана к конкретной
записи и не имеет собственного списка/поиска. `push` — пятый домен, добавленный отдельным
решением (`6-development/FEATURES_IMPLEMENTATION_PLAN.md` → FEAT-01): подписка на push — это
самостоятельный ресурс с собственным жизненным циклом (создаётся клиентом, привязана к
устройству, не к конкретной записи), а не read-only проекция — тот же принцип, по которому
`bookings` не стал подпутём `slots`.

## С чего начать разработчику бэкенда

1. `common/models.yaml` — сквозная схема `Error` и переиспользуемые параметры (`date_from`,
   `date_to`, `slotId`, `bookingId`). Остальные домены на неё ссылаются.
2. `auth/api.yaml` — без токена работают только `register` и `login`; всё остальное требует
   `Authorization: Bearer <token>` (см. `securitySchemes/bearerAuth` в `auth/models.yaml`).
3. `slots/api.yaml` — read-only список и карточка классов; поля `total_seats`/`free_seats`
   приходят готовыми, бизнес-правило вместимости (8/12) реализуется на бэкенде, в контракте не
   выражается отдельным полем-формулой.
4. `bookings/api.yaml` — самый крупный домен: создание (`createBooking`), список
   (`listBookings`), детали (`getBooking`), отмена (`cancelBooking`, порог 24 часа — сервер сам
   определяет `cancelled`/`late_cancel` по `slot.start_at`) и оценка (`submitRating`).
5. `push/api.yaml` — одна операция (`registerPushSubscription`, upsert по `endpoint`); сама
   доставка push (напоминание за 24ч, отмена студией) — не часть контракта, это
   backend-инфраструктура (`6-development/BE_IMPLEMENTATION_PLAN.md` → BE-10).

## Валидация спецификации

Каждый `api.yaml` — валидный самостоятельный документ OpenAPI 3.0.3 с относительными `$ref` на
соседние файлы. Для проверки подойдёт любой линтер/рендерер OpenAPI, поддерживающий
многофайловые спецификации (например Redocly CLI: `redocly lint api/bookings/api.yaml`).
Конфигурационный файл линтера — инфраструктурный артефакт тулинга, не часть аналитической
поставки, и в этом репозитории сознательно не создаётся.

## Открытые вопросы, влияющие на контракт

- **Тариф проката экипировки.** `02-domain.md` (R-015) упоминает «прокатный тариф на экипировку»
  как возможное поле бэкенда, но ни один FR/US не определяет формулу его влияния на цену — в
  `Slot`/`Booking` такого поля нет. Если тариф окажется ненулевым, потребуется добавить
  `rental_price` в `slots/models.yaml` и пересмотреть формулу `price_total` в `bookings/models.yaml`
  (см. `3-design-brief/SCR-004-booking.md` §11 п.4).
- **Точный механизм хранения токена/пароля** — вне скоупа анализа требований (NFR-7), оставлен
  на усмотрение реализации; в `auth/models.yaml` зафиксирован только контракт использования
  (`Authorization: Bearer <token>`).

**Решено:** регистрация push-подписки (ранее числилась здесь как открытый вопрос) — закрыта
доменом `push/` (`registerPushSubscription`), см. `6-development/FEATURES_IMPLEMENTATION_PLAN.md`
→ FEAT-01 и `5-mobile-app-spec/09_Логики/LOGIC-004_...md` → «API-запросы».
