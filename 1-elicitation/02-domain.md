# Описание домена и ограничения

> Этап 1. Артефакт 2 проекта: согласованное описание домена + список ограничений.
> Заполняется после ответов заказчика ([01-questions.md](01-questions.md)).

## Описание домена

**Бизнес.** «Шеф-стол» — кулинарная студия в лофте на бывшем заводе, работающая третий сезон.
Студия проводит **групповые кулинарные классы**. Класс идёт около **3 часов** и проходит по
одной из **программ**: простая итальянская кухня для новичков или более сложная программа для
опытных. Каждый класс ведёт **шеф** (сейчас их пятеро, в сезон нанимаются приглашённые —
для домена это те же шефы, без отдельного статуса).

**Процесс.** Владелец заранее формирует **расписание на неделю вперёд**: для каждого класса
задаются дата и время старта, программа, назначенный шеф и число свободных мест. Студия
располагает **12 рабочими столами**; для классов с духовкой и сложной техникой берут не более
**8 человек** (иначе не хватает плит и шеф не успевает за всеми). Клиент видит свободные
**слоты** классов и **записывается**, указывая, придёт он со своим фартуком и набором ножей или
возьмёт **рабочий набор напрокат** (по условиям заказчика прокатный фонд не ограничен — набора
хватает всегда). До класса клиент может **отменить** запись; после посещения — **оценить шефа**.
Сейчас запись клиентов ведётся вручную (WhatsApp + Google-таблица), что приводит к двойным
записям, путанице по выходным и потерям продуктов при поздних неявках.

**Цель клиентского приложения.** Дать клиентам самостоятельную онлайн-запись на классы
(просмотр слотов, бронирование, выбор экипировки, отмена, свои бронирования, оценка шефа) и
тем самым **сократить время, которое владелец тратит на ручную обработку записи**. Управление
расписанием и работу шефов/владельца обеспечивает **существующая инфраструктура**; приложение
потребляет её данные через API.

**Ключевые объекты домена:**

- **Класс / слот** — конкретное занятие: дата/время старта, программа, шеф, всего и свободно мест. Создаётся в существующей инфраструктуре; клиенту доступен только для чтения.
- **Программа (меню класса)** — вариант кулинарной программы (новичковая / для опытных), включает уровень сложности, фото и список ингредиентов/аллергенов блюда; справочные данные, read-only.
- **Шеф** — ведёт класс; для клиента — справочные данные слота (read-only).
- **Рабочий набор** — комплект фартука и ножа для проката; выдаётся клиенту без своей экипировки.
- **Запись (бронь)** — резерв клиентом места на конкретный слот: своя/прокатная экипировка, статус, оценка шефа после посещения.
- **Клиент** — записывается на классы, отменяет запись, оценивает шефа.

## Пользователи и роли

> **Скоуп текущей работы — только клиентское мобильное (веб-) приложение и API для него.**
> Роли Шеф и Владелец/Администратор обслуживаются **существующей инфраструктурой** (бэкенд,
> расписание, админка) и в этот скоуп не входят.

| Роль | Описание | Цели | В скоупе |
| :-- | :-- | :-- | :-- |
| **Клиент** | Гость студии, ходит на кулинарные классы. Пользуется с телефона (PWA). | Увидеть свободные слоты на ближайшие 7 дней, записаться, выбрать свою/прокатную экипировку, видеть свои бронирования, отменить запись, получать напоминания, оценить шефа после класса. | **Да** |

> Шеф и Владелец/Администратор (Артём) взаимодействуют с системой через **уже реализованные**
> интерфейсы существующей инфраструктуры. Клиентское приложение лишь потребляет подготовленные
> ими данные (слоты, программы, шефы) через API.

## Ограничения

**Бизнес-ограничения**

- Всего **12 рабочих столов**; для классов с духовкой/сложной техникой — не более **8 человек**.
- **Политика отмен:** отмена доступна в любой момент; отмена **не позднее чем за 24 часа** до
  старта — без предупреждений; отмена **менее чем за 24 часа** — запись всё равно отменяется, но
  клиенту показывается предупреждение (продукты уже закуплены), без штрафа и без блокировки.
- **Оплата офлайн** (наличные / перевод на карту при посещении) — приложение не принимает оплату,
  только фиксирует запись и показывает цену.
- Один клиент может иметь **несколько активных бронирований одновременно**, в том числе на
  пересекающееся время — система этого не ограничивает.
- **Период списка слотов по умолчанию — 7 дней (R-027).** По умолчанию клиенту показываются
  слоты на **ближайшие 7 дней** от текущего момента — совпадает с горизонтом планирования
  расписания («на неделю вперёд»). Более длинный период — через **фильтр дат**. При отсутствии
  расписания на ближайшие дни — **empty state** («Пока нет доступных классов»).
- **Отмена класса студией / по форс-мажору (R-008).** Если класс отменяет студия (организатор
  или из-за срыва поставки продуктов/поломки оборудования), бронь на стороне клиента **не
  удаляется**, а получает статус **«Отменён студией»** с указанием причины; клиенту приходит
  **push-уведомление** (резервный канал не предусмотрен — при недоступности push клиент увидит
  статус при следующем открытии приложения). **Повторная запись на отменённый слот запрещена.**
  Само действие отмены слота выполняется в существующей инфраструктуре.
- **Оценка шефа** — после посещения класса клиент может (необязательно) оставить оценку 1–5
  звёзд и опциональный текстовый комментарий.

**Технические / платформенные**

- **Mobile-first, веб-приложение (PWA)**: не публикуется в сторах, открывается в браузере на
  телефоне. Только русский язык интерфейса.
- Аутентификация клиента — по **email и паролю**.
- Клиентское приложение взаимодействует с **существующим бэкендом через API**; слоты, программы
  и шефы доступны только для чтения.
- **Граница серверной интеграции (R-004).** Детали серверной реализации — транзакционность,
  SLA, маппинг сущностей на внутренние модели бэкенда — **вне скоупа клиентского приложения**.
  Существующий бэкенд — **black-box источник истины**: гарантия «**0 двойных броней**»
  обеспечивается на его стороне (атомарная проверка свободных мест при бронировании). Со стороны
  клиента граница закрывается **контрактом API**: при отказе (мест уже нет) приложение
  показывает клиенту сообщение об ошибке; отдельного листа ожидания нет.
- **Ограничение вместимости (8/12) — логика бэкенда, не клиента.** Слоты, программы и шефы
  поступают из существующего бэкенда через API — приложение их потребляет, но не создаёт и не
  редактирует. Значит и ограничение по вместимости — часть данных API (поле вместимости слота),
  клиент не дублирует это бизнес-правило.
- **Данные существующей инфраструктуры (R-015).** Это **учебный/тестовый проект**, исторических
  данных и легаси-схемы нет. **Канонической схемой считается контракт API**: бэкенд отдаёт все
  поля контракта, включая прокатный тариф на экипировку, программу/меню класса, адрес студии.
  Миграция/backfill исторических данных — вне скоупа.

**Проект**

- **Срок** — к началу нового сезона, ориентировочно ~2 месяца от текущего момента; точная дата
  запуска пока не названа заказчиком (открытый пункт, см. `01-questions.md`, Q-009).
- **Бюджет** ограничен → начинаем с самого нужного, «навороты» позже.

**Границы скоупа**

Скоуп — **клиентское (веб-) приложение и API для него**. Управление расписанием, роль шефа и
владельца, админка — это **существующая инфраструктура**, вне скоупа.

| В скоупе клиентского приложения | Вне скоупа (существующая инфраструктура / отложено) |
| :-- | :-- |
| Регистрация/вход клиента по email и паролю | Роли Шефа и Владельца/Администратора, админ-панель |
| Просмотр слотов на ближайшие 7 дней + фильтр по датам | Создание и редактирование расписания, программ, шефов (приходят из API) |
| Просмотр деталей класса: программа/меню (фото, ингредиенты/аллергены), уровень сложности, шеф, свободные места | Сбор данных об аллергиях клиента |
| Самостоятельная запись, без ограничения числа активных броней | Онлайн-оплата в приложении |
| Выбор «своя / прокатная экипировка» | Резервный канал уведомлений (SMS/email) при недоступности push |
| Список своих бронирований, статус «Отменён студией» с причиной | Лист ожидания (waitlist) на заполненные слоты |
| Отмена записи с предупреждением при отмене <24 часов | Программа лояльности для постоянных клиентов |
| Push-напоминание о записи за 24 часа до класса | Массовая рассылка (кроме push при отмене студией — выполняется инфраструктурой) |
| Push-уведомление и статус при отмене класса студией | — |
| Оценка шефа после класса (звёзды 1–5 + комментарий) | — |
| Цена и фиксация записи (оплата офлайн) | — |

## Глоссарий

| Термин | Определение |
| :-- | :-- |
| **Класс / слот** | Запланированное групповое занятие: дата/время старта, программа, шеф, всего и свободно мест. |
| **Программа (меню класса)** | Вариант кулинарной программы: новичковая (простая итальянская кухня) или для опытных (сложнее); включает уровень сложности, фото и список ингредиентов/аллергенов. |
| **Шеф** | Ведёт класс; включает постоянных и приглашённых (сезонных) шефов — в домене без различий. |
| **Рабочий набор** | Комплект фартука и ножа, выдаваемый напрокат клиенту без своей экипировки; по условиям заказчика фонд не ограничен. |
| **Своя экипировка** | Клиент приходит со своим фартуком и набором ножей. |
| **Запись (бронь)** | Резерв клиентом места на конкретный слот: своя/прокатная экипировка, статус, оценка шефа после посещения. |
| **Отмена** | Снятие записи клиентом. Не позднее чем за 24 часа до старта — без предупреждений; менее чем за 24 часа — запись отменяется, но клиенту показывается предупреждение (продукты уже закуплены), без штрафа. |
| **Отмена студией** | Отмена класса организатором или из-за форс-мажора (срыв поставки продуктов, поломка оборудования). Бронь переходит в статус «Отменён студией» с причиной, клиент получает push, повторная запись на этот слот запрещена. |
| **Оценка шефа** | Отзыв клиента после посещения класса: 1–5 звёзд + опциональный комментарий. |
| **Владелец/администратор** | Артём — управляет расписанием и шефами через существующую инфраструктуру (вне скоупа клиентского приложения). |
| **MVP** | Минимально жизнеспособный продукт: набор функций к старту нового сезона. |

## Открытые вопросы по домену

1. **Число мест в одной записи.** Одна бронь всегда на одно место, или клиент может забронировать сразу несколько мест в одной записи (например, на себя и друга)? Ни бриф, ни снятые вопросы это не затрагивают — влияет на структуру сущности «Запись».
2. **Соотношение «рабочие столы (12 шт.) ↔ вместимость класса».** Ограничение «не больше 8 человек на классы со сложным оборудованием» при 12 столах в студии означает, что часть столов в такие дни не используется, или один стол может вмещать более одного человека? От ответа зависит, как трактовать поле вместимости слота, приходящее из API.
