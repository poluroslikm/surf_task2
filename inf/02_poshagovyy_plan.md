# Пошаговый план реализации: от брифа до ТЗ с помощью ИИ-агента

> Реконструкция workflow, который спикер выполнил вживую на примере кейса: Игорь, владелец саббординг-клуба «Волна» на городской набережной, хочет приложение для записи клиентов на прогулки. Инструмент — Клод в роли агента (с суб-агентами).
>
> **Важное примечание к этой версии файла**: базовая логика шагов и промптов взята из транскрипта вебинара (см. предыдущую версию). Точная **структура папок и файлов** ниже реконструирована по итоговому архиву результата автора и добавлена как практический шаблон организации проекта — чтобы твой результат совпадал с его архивом 1-в-1. Шаги, которых не было в самом транскрипте дословно (например `00-foundations.md`, `design-review.md`, декомпозиция API по доменам), помечены отдельно как "восстановлено по структуре результата, промпт — логическое развитие показанного в видео подхода".

## Исходный бриф заказчика (дан "как есть", сырой)

Ключевые тезисы из брифа Игоря:
- Клуб на набережной, саббординг, 3-й сезон, записи "зашились".
- Хочет приложение, чтобы клиенты сами записывались, а он только смотрел.
- Возит людей группами по реке, прогулка 1.5–2 часа, разные маршруты (для новичков — попроще, для опытных — подлиннее).
- Каждую прогулку ведёт инструктор, расписание на неделю вперёд.
- Клиент должен видеть свободные слоты и записываться.
- Хочет, чтобы клиент выбирал: со своей доской или взять напрокат.
- Иногда отмены: если заранее — ок, если за 10 минут до старта — проблема.
- После прогулки — клиент может оценить инструктора.
- Хочет уведомления, оплату онлайн (пока принимает наличкой), учёт погоды, сезон, постоянных клиентов.
- Пользоваться будут в основном с телефона: клиенты и инструкторы — прямо на берегу.
- Владельцу нужен отдельный экран (админский).
- Хочет успеть к началу следующего сезона.

**Важный принцип с самого начала**: нельзя просто отдать агенту весь бриф и сказать "напиши приложение". Причины:
- Задачи комплексные — их надо делить на этапы.
- Без контроля процесса на понятных, контролируемых шагах нет гарантии, что результат будет соответствовать ожиданиям.
- Разбиение на этапы даёт: поддерживаемость, понятную привязку кода к требованиям, возможность согласовывать документацию с заказчиком.

## Общая схема workflow

```
0-customer-brief → 1-elicitation → 2-requirements → 3-design-brief
     → 4-design (данные + API-flow) → api/ (OpenAPI по доменам)
     → 5-mobile-app-spec (финальное ТЗ)
```

Как был получен этот флоу: спикер заранее сам прошёл его целиком, нашёл сложности и ошибки, доработал, и попросил Клода зафиксировать результат в виде фреймворка (готового набора шагов/промптов), чтобы не изобретать заново каждый раз и не тратить время. Рекомендация: то же самое можно сделать самостоятельно — либо надиктовать (наговорить) агенту последовательность своих шагов и попросить оформить как Workflow, либо, набив шишек в первый раз вручную, зафиксировать результат как готовый фреймворк для повторного использования.

## Шаг 1. Вопросы к заказчику + описание доменной модели → `1-elicitation/`

Промпт агенту (два отдельных запроса подряд):
```
Изучи бриф (01-analysis/0-customer-brief/customer-brief.md).
Сформируй вопросы для заказчика.
Сохрани результат в 01-analysis/1-elicitation/customer-questions.md
```
```
Опиши доменную модель на основе брифа и ответов на вопросы.
Сохрани результат в 01-analysis/1-elicitation/domain-description.md
```

Что делает агент:
- Формирует список открытых вопросов, снимающих неопределённость брифа → `customer-questions.md`.
- Формирует описание домена: действующие лица, ключевые сущности (в кейсе: клуб, инструктор, маршрут, слот, прогулка, доска, бронь, оценка, уведомление клиента), бизнес-правила из брифа, жизненный цикл ключевой сущности (модель состояний брони), границы предложения → `domain-description.md`.

### Пример вопросов, которые сгенерировал агент, и как аналитик на них отвечал (режим вопрос-ответ, роль заказчика на себя, если реального заказчика нет)

| Вопрос агента | Варианты | Решение аналитика |
|---|---|---|
| Окно бесплатной отмены клиентом | за 2ч / за 24ч / без ограничений (лимит неявок) | За 2 часа (простой путь; в проде — лучше не фиксировать число, а вынести как абстрактный параметр N, настраиваемый в админке) |
| Как считается совместимость слота (места vs доски) | единый лимит мест / место + отдельный прокатный фонд | Место + отдельный прокатный счётчик досок (по рекомендации агента) |
| Одна бронь = один человек или можно компанией | один человек / компания одной бронью | Один человек — одна бронь (в брифе про запись компанией ничего нет → это хороший вопрос заказчику, но для скорости выбрано простое решение) |
| Оплата | оффлайн без цен / показываем цену оффлайн / онлайн-оплата сразу | Оффлайн (для простоты; в реальности можно держать оба варианта требований, оценить и решить, критична ли онлайн-оплата для MVP) |
| Как клиент входит | телефон+SMS / другое | Телефон + SMS |
| Как устроена оценка инструктора | только звёзды / текст+звёзды | Только звёзды |
| Какие push-уведомления нужны на старте | напоминание, отмена клубом, программа лояльности | Напоминание + отмена клубом; лояльность — отложить (в первую поставку не берём) |
| Как показывать маршруты | текст / карта | Только текст |
| Ограничивать ли запись новичка на "опытный" маршрут | да / нет | Нет, клиент выбирает сам |
| Когда присылать напоминание о прогулке | за сутки / за 2 часа | Оба (по рекомендации агента) |
| Нужен ли экран "Мои записи" (история + предстоящее) | да / нет | Да, обязательно |

**Принцип принятия решений на этом шаге**: там, где в брифе есть прямой ответ — берём его; там, где брифа недостаточно — либо помечаем как открытый вопрос к реальному заказчику, либо (если экономим время/тестируем процесс) сам аналитик принимает решение в пользу более простого варианта для MVP, и в идеале избегает фиксации "магических чисел" — выносит их как параметр.

**Важно**: после ответов на вопросы — попросить агента обновить `domain-description.md`, чтобы зафиксировать принятые решения.

## Шаг 2. Требования → `2-requirements/` (пять отдельных файлов)

Промпт агенту:
```
На основе 01-analysis/1-elicitation/domain-description.md выведи:
- бизнес-требования → сохрани в 01-analysis/2-requirements/business-requirements.md
- функциональные требования → 01-analysis/2-requirements/functional-requirements.md
- нефункциональные требования → 01-analysis/2-requirements/non-functional-requirements.md
- user stories → 01-analysis/2-requirements/user-stories.md
- use cases → 01-analysis/2-requirements/use-cases.md

Каждую категорию сохраняй отдельным файлом, не смешивай их в одном документе.
Ограничение: не добавляй ничего, чего нет в domain-description.md.
```

### Рекомендуемый (правильный) порядок работы аналитика:
1. Сначала бизнес-требования отдельно.
2. Затем User Story (пользовательский уровень).
3. Параллельно — функциональные требования и Use Case.
4. Нефункциональные требования — прорабатывать отдельно и внимательно, не полагаясь на то, что агент сделает их качественно "заодно".

На практике в демонстрации агенту дали задачу "скопом" (всё сразу) — это допустимо на небольших/простых проектах, но на объёмных задачах приводит к тому, что ИИ что-то упускает из-за слишком раздутого контекста. По возможности лучше идти по частям функциональности (например, отдельно спроектировать одну "прогулку" целиком), а не пытаться закрыть всё сразу.

### Ключевое правило файла — декомпозиция с самого начала
**Обязательно проси агента сразу писать в пять отдельных файлов** (как в целевой структуре выше), а не в один общий документ. Именно это было главной найденной ошибкой в демонстрации на вебинаре: агент сначала запихнул всё в один файл — это неудобно поддерживать, и суб-агенты дальше тратят лишние токены на чтение лишнего контекста.

### Найденные проблемы, которые нужно проверять при ревью на своём проекте
1. **Нефункциональные требования получаются слабыми/общими**, если не прорабатывать их отдельным явным запросом.
2. **Бизнес-правила могут зацепиться за конкретные числа из брифа** (например "не более 8 человек, всего досок 12") — это плохая практика: такие правила должны быть параметризованы, а не жёстко зашиты.
3. Use Case от агента часто описывает **только основной положительный сценарий**, упуская альтернативные потоки и исключения — явно проси добавить их.

### Обязательный шаг ревью
После получения требований — прогнать ревью: либо явно попросить переделать конкретные найденные ошибки в этом же чате, либо открыть **новый чат**, дать туда все пять файлов и попросить провалидировать: указать, что было выдумано агентом сверх исходного `domain-description.md` — такие места нужно вынести на обсуждение с заказчиком.

## Шаг 3. Дизайн-бриф: фундамент + черновые требования по экранам → `3-design-brief/`

### 3.1. Фундамент дизайна → `00-foundations.md`
*(восстановлено по структуре результата; промпт — логическое развитие показанного в видео подхода к формированию дизайн-требований)*

Промпт агенту:
```
На основе всех требований (01-analysis/2-requirements/) опиши общий фундамент
дизайна приложения: платформа, ключевые принципы UI, общие паттерны
взаимодействия (навигация, состояния загрузки/ошибки, тон коммуникации),
которые будут одинаковы для всех экранов.
Сохрани в 01-analysis/3-design-brief/00-foundations.md
```

Смысл файла — зафиксировать сквозные решения один раз, чтобы не повторять их в каждом экране отдельно (аналогично идее с логиками — не дублировать общее в каждом документе).

### 3.2. Реестр экранов и общий дизайн-бриф → `design-brief.md`

Промпт агенту (передаём весь набор требований):
```
[Передать весь набор требований из 2-requirements/]
Сгенерируй реестр экранов приложения и карту навигации.
Сохрани в 01-analysis/3-design-brief/design-brief.md
```

**Важное предупреждение из опыта демонстрации**: если заранее не задать агенту ограничение "не использовать готовые шаблоны полного ТЗ на этом шаге, а сначала создать отдельный, более лёгкий шаблон описания требований именно для дизайнера" — агент может самовольно пойти писать полноценные ТЗ по шаблону экрана раньше времени, не по делу. **Явно ограничивай агента на этом шаге.**

Что должен вернуть агент в `design-brief.md`:
- **Реестр экранов** — список всех экранов приложения. В кейсе: Вход по номеру телефона / Подтверждение SMS-кода (→ `SCR-001-registration`); Расписание прогулок (→ `SCR-002-slot-list`); Детали слота и маршрута (→ `SCR-003-slot-card`); Бронирование (→ `SCR-004-booking`); Мои записи (→ `SCR-005-my-bookings`); Детали брони (→ `SCR-006-booking-details`); Профиль клиента (→ `SCR-007-profile`).
- **Карта навигации** — граф переходов между экранами (в демонстрации использовалась нотация Mermaid).
- **Границы вне скоупа** — что явно не делаем (в кейсе: карта маршрута — только текст без визуальной карты; онлайн-оплата — не в этой поставке).

**Про папку `_export/`**: если карта навигации (или другие диаграммы) описана в Mermaid прямо внутри `design-brief.md`, некоторые редакторы/плагины (например расширения VS Code или онлайн-рендереры Mermaid) при экспорте диаграммы в картинку (SVG/PNG) автоматически создают рядом папку `_export/` с сохранённым файлом изображения. Это, как и `redocly.yaml` для OpenAPI (см. Шаг 5), — **побочный артефакт инструмента**, а не отдельный шаг, требующий промпта агенту. Отдельно создавать эту папку не нужно; она появится сама, если ты (или агент через MCP/инструмент) explicitly экспортируешь диаграмму в файл. Если хочешь получить такой экспорт сознательно, можно попросить агента: `Сгенерируй экспорт карты навигации в SVG и сохрани в 01-analysis/3-design-brief/_export/` — но это опционально и не влияет на дальнейшие шаги.

### 3.3. Требования по экранам (черновая версия) → `SCR-00X-*.md`

Промпт агенту:
```
Для каждого экрана из реестра (design-brief.md) сгенерируй отдельный файл
требований для дизайнера по шаблону: пользовательская цель, контент/данные
на экране, привязка к соответствующим User Story/Use Case (со ссылками).
Каждый экран — отдельный файл SCR-0XX-<название>.md в
01-analysis/3-design-brief/
```

Пример имён файлов из кейса: `SCR-001-registration.md`, `SCR-002-slot-list.md`, `SCR-003-slot-card.md`, `SCR-004-booking.md`, `SCR-005-my-bookings.md`, `SCR-006-booking-details.md`, `SCR-007-profile.md`.

### 3.4. Требования по "шторкам" (bottom sheets) → `BS-00X-*.md`
*(восстановлено по структуре результата — логичное развитие идеи "экран делится на блоки", когда часть UI выносится не отдельным экраном, а всплывающей панелью)*

Промпт агенту:
```
Выдели отдельные bottom sheet (шторки) — модальные панели, которые не являются
полноценными экранами (например: фильтры, подтверждение успешного действия,
подтверждение отмены, карта маршрута). Опиши каждую отдельным файлом
BS-0XX-<название>.md в 01-analysis/3-design-brief/, по аналогии с шаблоном
для экранов.
```

Пример имён файлов из кейса: `BS-001-filters.md`, `BS-002-booking-success.md`, `BS-003-cancel-confirm.md`, `BS-004-route-map.md`.

Рекомендация: если планируете передавать это дизайнеру-человеку, попросите агента "причесать" техническое описание — сделать более читаемым и приятным форматом, так как сырой технический вывод агента может выглядеть избыточно/непонятно для не-технического дизайнера.

### 3.5. Ревью дизайн-брифа → `design-review.md`
*(восстановлено по структуре результата — прямое применение принципа "после каждого шага — ревью, лучше в отдельном чате", описанного в видео)*

## Шаг 5. API-контракт по доменам → `api/`

### 5.1. Генерация OpenAPI

Промпт агенту (передаём проектирование дизайна — экраны и требования, явно указывая путь):
```
На основе 01-analysis/3-design-brief/ (SCR-*.md, BS-*.md) и
01-analysis/4-design/data-model.md сгенерируй OpenAPI-контракт.
```
Важно: **явно указывать агенту папку/файлы**, откуда брать контекст — иначе он может ошибиться с источником (из опыта демонстрации).

### 5.2. Декомпозиция API по доменам
*(восстановлено по структуре результата — прямое применение рекомендации из видео: "для более крупных проектов описывать каждый запрос отдельным документом, декомпозировать спецификацию")*

**Найденная в демонстрации проблема**: если попросить агента сгенерировать OpenAPI одним запросом без уточнений — получится **один большой файл**, что неудобно поддерживать в будущем. Правильный следующий шаг — явно попросить декомпозицию:

```
Раздели единый OpenAPI-контракт на отдельные модули по доменным сущностям:
auth, bookings, instructors, slots, profile, а также общий common для
переиспользуемых схем.
Для каждого модуля создай папку в api/<домен>/ с двумя файлами:
- api.yaml — эндпоинты этого домена
- models.yaml — схемы данных этого домена
Общие переиспользуемые схемы (ошибки, пагинация и т.п.) вынеси в
api/common/models.yaml
```

Результат — структура:
```
api/
├── auth/{api.yaml, models.yaml}
├── bookings/{api.yaml, models.yaml}
├── instructors/{api.yaml, models.yaml}
├── slots/{api.yaml, models.yaml}
├── profile/{api.yaml, models.yaml}
└── common/{models.yaml}
```

Что должно получиться по содержанию: набор эндпоинтов, покрывающих сценарии экранов (в кейсе: `auth` — получение токена по SMS; `bookings` — создание/отмена брони; `slots` — список слотов/маршрутов; `instructors` — данные инструкторов, рейтинг; `profile` — профиль клиента, список уведомлений).

*Примечание: инструмент валидации/рендера полученной спеки (например Redocly) и его конфигурационный файл — это уже инфраструктурный, а не аналитический артефакт, отдельно его создавать не нужно, он появляется автоматически при использовании тулинга для проверки YAML-файлов.*

### 5.3. README для папки `api/`
*(восстановлено по структуре результата — отдельный от корневого README, описывающий конкретно API-контракт)*

В отличие от корневого `README.md` (общий обзор всего проекта, см. Шаг 6.4), этот файл — прицельно про то, как читать и использовать API-спецификацию.

Промпт агенту:
```
Составь README.md для папки api/: как устроена структура (папки по доменам,
в каждой — api.yaml и models.yaml, общие схемы — в common/models.yaml),
с какого домена начать разработчику бэкенда, как валидировать спецификацию
(например через Redocly), где искать сквозные схемы (ошибки, пагинация).
Сохрани в api/README.md
```

### 6.1. Подготовка шаблонов

```
история изменений, краткое описание, бизнес-ценность, навигация входящая/
исходящая, локальные модели данных, блоки и компоненты, состояния,
применяемая логика, инициализация экрана, макет, требования, критерии
```

```
требования, критерии приёмки, обработка ошибок).
Сохрани в 01-analysis/5-mobile-app-spec/_LOGIC_TEMPLATE.md
```

(См. подробное содержание обоих шаблонов в файле теории, раздел 10.)

### 6.2. Генерация финальных ТЗ по экранам и шторкам

Промпт агенту (собираем воедино все артефакты):
```
Используя _SCREEN_TEMPLATE.md, _LOGIC_TEMPLATE.md, требования для дизайна
из 3-design-brief/ (включая design-review.md) и готовый API-контракт из api/ —
сгенерируй финальную спецификацию (ТЗ) на каждый экран и каждую шторку
отдельным файлом по шаблону.
Экраны сохраняй как SCR-0XX-<название>.md, шторки — как BS-0XX-<название>.md,
в 01-analysis/5-mobile-app-spec/
```

Агент на этом шаге делегирует работу суб-агентам — по одному на каждый экран/шторку (в демонстрации получилось 9 итоговых документов: 7 экранов + шторки/логики). Каждый суб-агент получает свой промпт: "сначала прочитай шаблон и нужный контекст, затем напиши ТЗ по шаблону для конкретного экрана X".

**Важное замечание об эффективности**: поскольку исходные требования уже разложены по отдельным файлам (Шаг 2) и черновые дизайн-требования тоже (Шаг 3.3–3.4), суб-агентам не придётся вычитывать весь массив данных целиком — только нужные соседние файлы. Это экономит токены и снижает риск ошибок.

### Пример готового результата (экран "Вход" / `SCR-001-registration.md`)
Финальное ТЗ на экран входа содержит:
- User Story: "Как клиент клуба, я хочу войти по номеру телефона и коду из SMS, чтобы попасть [в приложение]"
- Бизнес-ценность: быстрый вход, отсутствие пароля, снижение барьера входа, снижение числа обращений в поддержку, единая точка контроля доступа (без успешного входа приложение недоступно)
- Навигация: старт приложения / первый запуск; после выхода из аккаунта — разлогин; по deep-link предлагает зайти. Исходящая навигация — подтверждение входа либо выход из приложения
- Входные данные: телефон
- Логика: проверка валидности телефона, применяемая логика "Вход по SMS-коду"
- Экран загрузки: не требуется отдельного состояния
- Запрос на получение SMS-кода
- Макет экрана (сгенерированный агентом эскиз; в идеале — ссылаться на реальный макет из Figma)
- Описание компонентов: логотип, заголовок и т.д.

### 6.3. Логики → `09_Логики/`

Промпт агенту:
```
Вынеси сложные цепочки действий/запросов, которые не укладываются в один
экран или переиспользуются в нескольких местах (например: процесс
авторизации, отправка напоминаний, обработка отмены с учётом окна N часов),
в отдельные документы по _LOGIC_TEMPLATE.md.
Сохрани в 01-analysis/5-mobile-app-spec/09_Логики/
```

Используется, когда сложную последовательность запросов/действий нельзя уложить в один экран, либо она переиспользуется в нескольких местах — чтобы не описывать одно и то же в каждом экране заново, а только ссылаться на единый документ логики.

## Сквозные принципы, которые нужно соблюдать на каждом шаге

1. **Не отдавать агенту весь бриф целиком с запросом "напиши приложение"** — всегда идти по этапам, описанным выше.
2. **Явно ограничивать агента** на каждом шаге — что можно использовать (шаблоны, файлы), а что нельзя, иначе агент "додумает" не туда.
3. **Явно указывать путь к файлам/папкам**, из которых агент должен брать контекст (см. точную структуру папок выше).
4. **Ставить жёсткое ограничение "не добавляй того, чего нет в домене/требованиях"**, чтобы агент не выдумывал лишнего.
5. **После каждого крупного шага — ревью** (самостоятельно и/или агентом в отдельном чате с передачей контекста) — как это сделано с `design-review.md`.
6. **Открывать новый чат при смене несвязанной задачи**, чтобы не забивать контекст и не провоцировать "забывчивость"/галлюцинации модели.
7. **Декомпозировать документы с самого начала** (по требованиям, по экранам, по Use Case, по доменам API) — не складывать всё в один файл. Именно так получена итоговая структура: пять файлов в `2-requirements/`, отдельный файл на каждый экран/шторку, API разбит по доменам вместо одного гигантского файла.
8. **Не фиксировать "магические числа"** в требованиях и бизнес-правилах напрямую — выносить их как параметры, если они могут измениться.
9. **На небольших/простых проектах допустимо** идти "скопом" (всё сразу), но на объёмных — идти по частям функциональности, чтобы не раздувать контекст и не терять детали.
10. **Двухпроходный подход к экранам**: сначала лёгкие требования для дизайна (`3-design-brief/`), потом — полноценное ТЗ по шаблону, уже с готовым API и моделью данных (`5-mobile-app-spec/`). Не пытайся сразу написать финальное ТЗ, минуя дизайн-бриф.
11. **Финальный результат всегда проверяется человеком** перед передачей в разработку.
12. **Отличай артефакты аналитика от побочных файлов инструментов.** Папки вроде `_export/` (экспорт диаграмм) или файлы вроде `redocly.yaml` появляются автоматически при использовании конкретных редакторов/тулинга — это не отдельные шаги твоего анализа, и не нужно просить агента "создать" их напрямую, если только ты сознательно не хочешь получить такой экспорт.
