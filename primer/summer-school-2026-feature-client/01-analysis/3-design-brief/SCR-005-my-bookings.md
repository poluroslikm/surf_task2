## 1. Назначение и контекст

Экран даёт клиенту **контроль над своими прогулками** — отдельно **предстоящими** и
**прошедшими**. Это домашняя точка для проверки «куда и когда я записан», входа в детали
записи и (через детали) запуска отмены. Экран — корневой во вкладке «Мои записи» таб-бара
(см. [foundations §4.2](00-foundations.md#42-таб-бар-авторизованная-зона)).

Контекст использования — берег, солнце, спешка перед выходом на воду
([foundations §1](00-foundations.md#1-продукт-и-аудитория)): клиенту нужно **с одного взгляда**
понять ближайшую запись (когда старт, какой маршрут, сколько мест) и её статус.

---

## 2. Пользователь и цель

- **Роль:** Клиент (единственная роль приложения).
- **Цель пользователя:** быстро увидеть свои предстоящие прогулки, убедиться в деталях
  ближайшей записи, при необходимости перейти к деталям/отмене; при отсутствии записей —
  понять, как записаться.
- **Покрываемая история:** [US-9](../2-requirements/user-stories.md) — «Как клиент, я хочу
  видеть список своих записей (предстоящих и прошедших), чтобы контролировать свои прогулки».

---

## 3. Навигация

### Входящие
- **Таб «Мои записи»** в таб-баре с любого корневого экрана АЗ
  ([foundations §4.2](00-foundations.md#42-таб-бар-авторизованная-зона)).
- Из шторки [BS-002 «Подтверждение записи»](BS-002-booking-success.md) по действию
  **«Мои бронирования»** (сразу после успешной записи).

### Исходящие
- **Тап по карточке записи** → [SCR-006 «Детали брони + отмена»](SCR-006-booking-details.md).
- Переключение вкладок таб-бара → [SCR-002 «Список слотов»](SCR-002-slot-list.md),
  [SCR-007 «Профиль»](SCR-007-profile.md).
- Из **Empty-состояния** по действию «Записаться на прогулку» → переход к списку слотов
  [SCR-002](SCR-002-slot-list.md).

### Каркас и таб-бар
Это **корневой** экран вкладки — таб-бар виден; на дочерних экранах (детали брони) скрывается
по правилу [foundations §4.2](00-foundations.md#42-таб-бар-авторизованная-зона). Базовый каркас —
[foundations §4.1](00-foundations.md#41-базовый-каркас). Собственного нижнего CTA у экрана нет.

## 4. Контент-инвентарь

Данные — записи (брони) **текущего клиента**, см. [модель данных](../4-design/data-model.md).
Числовые лимиты не зашиваются в UI — берутся из данных слота/маршрута
([foundations §6](00-foundations.md#6-tone-of-voice-и-общая-микрокопия)).

### 4.1 Карточка записи (booking)

| Поле | Источник | Назначение на карточке |
|------|----------|------------------------|
| `status` | booking | Бейдж статуса: `active` → «активна», `cancelled` → «отменена», `late_cancel` → «поздняя отмена». Текст + форма, не только цвет. |
| `start_at` | slot | Дата и время старта прогулки — крупно, основной ориентир. |
| `route.name` | slot → route | Название маршрута (и его тип, если приходит). |
| `instructor.name` | slot → instructor | Имя инструктора. |
| `price` | slot / booking | Цена (итоговая по записи). Оплата офлайн — текст см. [foundations §6](00-foundations.md#6-tone-of-voice-и-общая-микрокопия). |
| `seats_count` | booking | Число мест в записи (себя + гости). |
| Вариант доски / `rental_count` | booking | Сводка по доскам: своя / прокатная (сколько прокатных). Лейблы — [foundations §6](00-foundations.md#6-tone-of-voice-и-общая-микрокопия). |

> Карточка — **сводка**, достаточная для опознания записи; полная детализация — на
> [SCR-006](SCR-006-booking-details.md).

### 4.2 Разделение списка
- **Предстоящие** — **только активные** записи (`status = active`), у которых `start_at` в будущем
  (ориентир и приоритетная секция).
- **Прошедшие** — записи с `start_at` в прошлом **И все отменённые** (`cancelled` / `late_cancel`)
  независимо от даты старта. То есть отменённая запись с будущим стартом тоже попадает в «Прошедшие».

> Правило отнесения: запись активна и старт в будущем → «Предстоящие»; иначе (старт в прошлом
> **или** запись отменена) → «Прошедшие». Активность и время старта вычисляются из данных, без
> отдельного флага. Прошедшие сортируются **по убыванию даты старта** (свежие сверху).

## 5. Структура и иерархия

Сверху вниз: **хедер** (заголовок + переключатель/секции) → **скролл-список карточек** →
**таб-бар**.

```
┌──────────────────────────────┐
│  Мои записи                  │  ← заголовок экрана (хедер)
│  [ Предстоящие | Прошедшие ] │  ← сегмент-переключатель (или секции)
├──────────────────────────────┤
│ ┌──────────────────────────┐ │  ← скролл-список карточек
│ │ сб, 21 июн · 09:00       │ │     (карточка записи)
│ │ Новичковый маршрут       │ │
│ │ Инструктор: Анна         │ │
│ │ 2 места · 1 прокатная    │ │
│ │ 3000 ₽          [активна]│ │  ← бейдж статуса (текст+форма)
│ └──────────────────────────┘ │
│ ┌──────────────────────────┐ │
│ │ вс, 22 июн · 18:00       │ │
│ │ Опытный маршрут          │ │
│ │ Инструктор: Игорь        │ │
│ │ 1 место · своя доска     │ │
│ │ 1500 ₽          [активна]│ │
│ └──────────────────────────┘ │
├──────────────────────────────┤
│  Прогулки · Мои записи · Профиль │  ← таб-бар
└──────────────────────────────┘
```

Группировка при варианте с секциями (вместо сегмента):

```
┌──────────────────────────────┐
│  Мои записи                  │
├──────────────────────────────┤
│  ПРЕДСТОЯЩИЕ                  │  ← заголовок секции (sticky)
│  [ карточка ]                │
│  [ карточка ]                │
│                              │
│  ПРОШЕДШИЕ                    │  ← заголовок секции (sticky)
│  [ карточка ]                │
│  [ карточка ]                │
└──────────────────────────────┘
```

Типографическая иерархия — [foundations §3.3](00-foundations.md#33-типографическая-иерархия-уровни-не-шрифты):
дата/время старта и цена — наиболее контрастные элементы карточки.

## 6. Компоненты и поведение

| Компонент | Поведение |
|-----------|-----------|
| **Переключатель «Предстоящие / Прошедшие»** | Сегмент-контрол вверху **или** разделение списка на sticky-секции (выбор паттерна — за дизайнером). При сегменте показывается одна группа за раз; активная вкладка визуально выделена (текст + состояние, не только цвет). По умолчанию открыты **Предстоящие**. |
| **Карточка записи** | Кликабельна целиком (тач-зона ≥ [foundations §3.1](00-foundations.md#31-тач-зоны-и-размеры)); тап → [SCR-006](SCR-006-booking-details.md). Крупная, в одну колонку, явно отделена отступами/границей. |
| **Бейдж статуса** | Текст + форма (контур/заливка/иконка), не только цвет — [foundations §3.2](00-foundations.md#32-контраст-и-читаемость-nfr-1), [§3.5](00-foundations.md#35-иконки-и-индикаторы). Тексты — см. §8. |
| **Pull-to-refresh** | Жест «потянуть вниз» обновляет список (актуальные статусы/данные) — паттерн [foundations §5](00-foundations.md#5-сквозной-паттерн-состояний-экрана). |
| **Скролл-список** | Вертикальный, одна колонка; при варианте секций — sticky-заголовки групп. |

## 7. Состояния экрана

Применяется сквозной паттерн [foundations §5](00-foundations.md#5-сквозной-паттерн-состояний-экрана).
Ниже — экранная специфика.

| Состояние | Содержимое / Поведение |
|-----------|------------------------|
| **Loading** | Скелетоны карточек записи (в форме будущего контента), без блокирующего спиннера. |
| **Content** | Список карточек, разделённый на Предстоящие (только активные) / Прошедшие (прошлые + все отменённые). |
| **Empty (нет записей вообще)** | Заглушка «У вас пока нет записей» + действие **«Записаться на прогулку»** → [SCR-002](SCR-002-slot-list.md). |
| **Empty в секции «Предстоящие»** (есть только прошедшие) | В секции/вкладке «Предстоящие» — короткая подсказка «Нет предстоящих записей» + «Записаться на прогулку»; секция «Прошедшие» при этом доступна. |
| **Error** | Заглушка ошибки нейтральным тоном + кнопка **«Обновить»**. Текст сетевой ошибки — общий из [foundations §6](00-foundations.md#6-tone-of-voice-и-общая-микрокопия). |

## 8. Валидации и микрокопия

### Доступ к данным
- Клиент видит **только свои записи** ([NFR-12](../2-requirements/non-functional-requirements.md),
  [foundations §8.2](00-foundations.md#82-безопасность-данных-в-ui-nfr-11-nfr-12)). Чужих
  броней, контактов или административных данных на экране нет.

### Тексты бейджей статуса

| `status` | Текст бейджа |
|----------|--------------|
| `active` | «активна» |
| `cancelled` | «отменена» |
| `late_cancel` | «поздняя отмена» |

> Подробное пояснение поздней отмены (правило 2 часов, отсутствие штрафа) — на
> [SCR-006](SCR-006-booking-details.md) / [BS-003](BS-003-cancel-confirm.md); на карточке —
> только короткий бейдж.

### Empty-состояние
- Заголовок: «У вас пока нет записей».
- Подсказка/действие: «Запишитесь на прогулку» (ведёт на список слотов).
- Тон — спокойный, без давления ([foundations §6](00-foundations.md#6-tone-of-voice-и-общая-микрокопия)).

### Лейблы доски и оплаты
- «Своя доска» / «Прокатная доска», текст об офлайн-оплате — переиспользуются из
  [foundations §6](00-foundations.md#6-tone-of-voice-и-общая-микрокопия), не вводятся заново.

## 9. Доступность и mobile-first

Базовые правила — [foundations §7](00-foundations.md#7-доступность-nfr-1) и
[§3](00-foundations.md#3-структурные-токены-без-бренда). Экранная специфика:

- **Крупные карточки** с большой тач-зоной — удобно тапать мокрым/холодным пальцем на берегу.
- **Читаемые статусы**: бейдж дублирует смысл текстом и формой, считывается на ярком солнце и
  при дальтонизме.
- **Читаемые даты/время**: формат старта понятен с первого взгляда (день недели + дата +
  время), без двусмысленности.
- Переключатель/секции и карточки имеют доступные имена для screen reader; фокус-состояния
  обязательны.

## 10. Критерии приёмки дизайна

```gherkin
Сценарий: AC-001 Разделение на предстоящие и прошедшие
  Дано у клиента есть записи с разными датами старта и статусами
  Когда он открывает экран «Мои бронирования»
  Тогда записи разделены на группы «Предстоящие» и «Прошедшие»
  И по умолчанию отображаются предстоящие записи
  И в «Предстоящие» попадают только активные записи с будущим стартом

Сценарий: AC-001a Отменённая запись с будущим стартом — в «Прошедшие»
  Дано запись отменена (cancelled или late_cancel), но её старт ещё в будущем
  Когда клиент открывает экран
  Тогда эта запись показана в группе «Прошедшие», а не в «Предстоящие»
  И прошедшие записи отсортированы по убыванию даты старта

Сценарий: AC-002 Отображение статуса записи
  Дано запись имеет статус active, cancelled или late_cancel
  Когда клиент видит её карточку
  Тогда на карточке показан бейдж со статусом «активна», «отменена» или «поздняя отмена»
  И статус передан текстом и формой, а не только цветом

Сценарий: AC-003 Пустой список
  Дано у клиента нет записей
  Когда он открывает экран
  Тогда показана заглушка «У вас пока нет записей»
  И доступно действие «Записаться на прогулку», ведущее к списку слотов

Сценарий: AC-004 Только свои записи
  Дано в системе есть записи разных клиентов
  Когда клиент открывает «Мои бронирования»
  Тогда он видит исключительно собственные записи
  И на экране нет чужих или административных данных

Сценарий: AC-005 Переход к деталям брони
  Дано на экране показана карточка записи
  Когда клиент тапает по карточке
  Тогда открывается экран деталей брони (SCR-006)

Сценарий: AC-006 Обновление списка
  Дано экран в состоянии Content
  Когда клиент выполняет жест pull-to-refresh
  Тогда список перезапрашивается и отображает актуальные данные и статусы
```

## 11. Решения

1. **Отменённые записи — всегда в «Прошедшие»** (в т.ч. с будущим стартом); «Предстоящие» — только активные (§4.2).
2. **Сортировка прошедших** — по убыванию даты старта (свежие сверху) (§4.2).
3. **Empty по секциям** — отдельная подсказка «Нет предстоящих записей» в секции «Предстоящие», если есть только прошедшие; общий empty «У вас пока нет записей» — когда нет ни одной (§7).
4. **При отсутствии прошедших** — секция/вкладка «Прошедшие» показывается пустой (не скрывается), чтобы навигация по секциям была предсказуемой.
5. *(Параметр конфигурации, не блокер)* **Глубина истории прошедших** — параметр данных/конфига; в MVP грузится доступная история целиком, лимит не хардкодим.
