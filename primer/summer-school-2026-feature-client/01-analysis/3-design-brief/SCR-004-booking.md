## 1. Назначение и контекст

Экран собирает параметры брони (число мест, вариант доски на каждое место, итоговую цену) и
фиксирует запись. Это **самый сложный экран** клиентского сценария: на нём пересекаются два
независимых лимита (места группы и прокатный фонд) и обрабатываются конкурентные записи.

Экран — **последний шаг перед подтверждением** в потоке записи: список → карточка →
оформление → подтверждение. Соблюдается правило **≤ 3 экранов до подтверждения** (NFR-2): от
[SCR-002](SCR-002-slot-list.md) до [BS-002](BS-002-booking-success.md) клиент проходит карточку
([SCR-003](SCR-003-slot-card.md)) и этот экран, без лишних шагов.

> **Без хардкода.** Все числа (`free_seats`, `free_rental_boards`, потолок маршрута, `price`,
> `rental_price`) приходят из данных слота/маршрута и **не зашиваются** в макет или тексты
> (см. [foundations §6](00-foundations.md)). Правила двух лимитов и формула цены собраны в одном
> месте — **§6.5**; остальные разделы ссылаются на него, не повторяя условия.

## 2. Пользователь и цель

- **Роль:** Клиент (единственная роль приложения, см. [foundations §1](00-foundations.md)).
- **Контекст:** клиент стоит на берегу, спешит, руки могут быть мокрыми — нужны крупные
  элементы и минимум действий (P1, P2).
- **Цель пользователя:** быстро записать себя (и при желании 1–2 гостей), выбрать доску на
  каждого, увидеть итоговую цену и подтвердить бронь.
- **Цель системы:** собрать корректную бронь в рамках двух лимитов, не допустить овербукинга и
  двойной брони, зафиксировать запись.

## 4. Контент-инвентарь

### 4.1 Поля брони (вводимые / производные)

| Поле | Тип | Источник / правило | Примечание |
|------|-----|--------------------|------------|
| `seats_count` | Ввод | Число мест: себя + 0–2 гостя, итого **1–3** (без ввода имён гостей) | Минимум 1 (сам клиент), максимум — см. §6.5 |
| `board_choice[i]` | Ввод | Для каждого места `i`: `own` / `rental` | Лейблы: «Своя» / «Прокатная» |
| `rental_count` | Производное | Количество мест с `board_choice = rental` | Не вводится напрямую, считается из переключателей |
| `own_count` | Производное | `seats_count − rental_count` | — |
| Итоговая цена | Производное | `price × seats_count + rental_price × rental_count` | Пересчёт в реальном времени; формула — §6.5 |

> **Имена гостей не вводятся** — достаточно числа мест (NFR-2/NFR-3, минимум полей). Цена места
> (`price`) и **отдельный тариф проката** (`rental_price`) приходят из данных слота; итог зависит
> и от числа прокатных мест (`rental_count`). Полная формула — §6.5.

### 4.2 Контекст слота (read-only, для расчётов и подписей)

| Поле | Назначение в UI |
|------|------------------|
| Дата / время старта, маршрут, инструктор | Краткая шапка-сводка слота (что бронируем) |
| `free_seats` | Свободно мест в группе — верхняя граница `seats_count` |
| `free_rental_boards` | Свободно прокатных досок — верхняя граница `rental_count` |
| Потолок маршрута | Максимум мест маршрута (новичковый/опытный); участвует в `min(...)` |
| `price` | Цена за место |
| `rental_price` | Тариф проката за одну прокатную доску (отдельно от `price`) |

## 5. Структура и иерархия

Каркас — по [foundations §4.1](00-foundations.md): фикс. хедер → скролл-контент → фикс.
нижний CTA. Сверху вниз:

1. **Хедер** — «назад» + заголовок «Оформление записи».
2. **Сводка слота** (компактная, read-only) — дата/время, маршрут, инструктор; подпись о
   свободных местах и прокатных досках.
3. **Счётчик мест** (`seats_count`) — степпер 1–3 с подписью текущего лимита.
4. **Блок выбора доски по каждому месту** — список из `seats_count` строк: «Место 1 (вы)»,
   «Место 2 (гость)»… с переключателем «Своя / Прокатная».
5. **Блок цены** — цена за место × число мест + тариф проката × число прокатных = итог;
   напоминание об офлайн-оплате (формула — §6.5).
6. **Фикс. нижний CTA «Записаться»** — всегда виден, не перекрыт клавиатурой.

## 6. Компоненты и поведение

### 6.1 Степпер «Число мест» (`seats_count`)

- Крупный степпер с кнопками **«−» / «+»** и числом по центру (тач-зоны ≥ 44–48 pt,
  [foundations §3.1](00-foundations.md)).
- **Минимум — 1** (клиент всегда бронирует себя): на минимуме «−» disabled.
- **Максимум** — по правилу лимита мест из **§6.5** (`min(free_seats, потолок маршрута, 3)`).
  На максимуме «+» disabled (ограничение сверху, без скрытой блокировки) — клиент видит границу.
- Подпись под степпером поясняет текущий лимит словами (число — из данных, без хардкода).
- При **уменьшении** `seats_count` лишние строки выбора доски (с конца) убираются; при
  увеличении — добавляются новые строки со значением по умолчанию (см. §6.2).

### 6.2 Переключатели «Своя / Прокатная» (на каждое место)

- На каждое место — сегментный переключатель из двух опций: **«Своя»** / **«Прокатная»**
  (бинарный, без третьего состояния). Состояние дублируется не только цветом — формой/текстом
  ([foundations §3.2](00-foundations.md)).
- **Значение по умолчанию для нового места:** «Прокатная», если `rental_count < free_rental_boards`;
  иначе «Своя» (чтобы не создавать заведомо невалидную бронь).
- **Ограничение сверху по прокату:** суммарно нельзя выбрать «Прокатная» больше, чем
  `free_rental_boards`. При достижении границы опция «Прокатная» на ещё не-прокатных местах
  становится **недоступной** с поясняющей подписью (см. §8, E2), а не молча игнорируется.
- Подпись-счётчик «Прокатных выбрано: `rental_count` из `free_rental_boards`» обновляется
  в реальном времени.

### 6.3 Блок итоговой цены

- Показывает две строки — **места** (`price × seats_count`) и, если есть прокатные места,
  **прокат** (`rental_price × rental_count`) — и **итог** крупно и контрастно (важное число,
  [foundations §3.2](00-foundations.md)). Формула — §6.5.
- **Пересчёт в реальном времени** при изменении `seats_count` **и** варианта доски (прокат влияет
  на итог). Строка «Прокат» скрыта при `rental_count = 0`. Итог дублируется суммой на CTA.
- Под ценой — напоминание об офлайн-оплате (текст из [foundations §6](00-foundations.md):
  «Оплата на месте: наличные или перевод на карту.»).

### 6.4 Кнопка «Записаться» (нижний CTA)

- Во всю ширину контентной области, всегда видна над клавиатурой ([foundations §4.1](00-foundations.md)).
- **Enabled**, когда бронь валидна по правилам лимитов из **§6.5**.
- **Disabled** при невалидной комбинации (поясняющая микрокопия рядом, не «глухая» кнопка).
- **Состояние отправки (loading):** при тапе CTA переходит в индикатор загрузки и
  **блокируется от повторного тапа** (защита от двойной брони / овербукинга, NFR-8 — см. §7).

### 6.5 Правила лимитов и цены (единый источник, FR-13–FR-15, FR-30)

**Лимиты (два независимых):**

| Лимит | Граница | Правило |
|-------|---------|---------|
| Число мест `seats_count` | `1 ≤ seats_count ≤ min(free_seats, потолок маршрута, 3)` | Минимум 1 (сам клиент); максимум — меньшее из свободных мест, потолка маршрута и 3 (себя + 2 гостя, FR-12). |
| Прокатные места `rental_count` | `rental_count ≤ free_rental_boards` | Считается из переключателей «Прокатная»; не больше свободных прокатных досок. |

- **«Своя доска»** занимает место в группе, но **НЕ** уменьшает прокатный фонд (FR-14, UC-1 A1).
- **«Прокатная»** занимает место в группе **И** уменьшает прокатный фонд (FR-14, UC-1 A2).
- Следствие: при `free_rental_boards = 0` запись возможна **только из собственных досок**.

**Формула итоговой цены:**

```
Итого = price × seats_count + rental_price × rental_count
```

- `price` — цена за место, `rental_price` — отдельный тариф за одну прокатную доску (оба из данных слота).
- Прокат тарифицируется **отдельно**, поэтому итог зависит и от числа прокатных мест.

Эти правила — **единственное место** их описания на экране; §6.1, §6.2, §6.3, §8 ссылаются сюда.

## 7. Состояния экрана

Базовый паттерн — [foundations §5](00-foundations.md). Специфика SCR-004:

| Состояние | Содержимое / Поведение |
|-----------|------------------------|
| **Content** | Основной режим: форма брони готова к вводу. Данные слота переданы с предыдущего экрана / актуализированы. |
| **Отправка (submitting)** | После тапа «Записаться» — индикатор загрузки **на CTA**, элементы формы блокируются. Повторный тап невозможен. |
| **Ошибки** | Специфичные ошибки записи E1–E4 — см. [§8](#8-валидации-и-микрокопия). Показываются inline / в виде нотиса, без потери введённых данных, где это возможно. |

**Защита от двойной брони и овербукинга (NFR-8):**

- Повторный тап «Записаться» во время отправки **заблокирован** (§6.4) — UI не отправляет
  второй запрос на ту же бронь.
- Финальную проверку лимитов делает бэкенд в момент создания записи; UI-ограничения (§6) —
  предупреждающие, не заменяют серверную проверку. При проигрыше гонки — обработка E3 (§8).

## 8. Валидации и микрокопия

> Тон — спокойный, без вины ([foundations §6](00-foundations.md)). В текстах `[N]`, `[M]` —
> **подстановка из актуальных данных**, не константы.

| Код | Условие | Точный текст | Поведение UI |
|-----|---------|--------------|--------------|
| **E1** | Запрошено мест больше, чем свободно (`seats_count > free_seats`) | **«Недостаточно мест. Свободно: [N]. Уменьшите бронь до [N] мест.»** | Запись отклоняется; показать актуальное `free_seats`; предложить/выполнить откат счётчика до `[N]`. |
| **E2** | Прокатных запрошено больше, чем свободно досок (`rental_count > free_rental_boards`) | **«Недостаточно прокатных досок. Свободно: [M]. Выберите меньше прокатных или свою доску.»** | Предложить уменьшить число прокатных мест **или** переключить место(а) на «Свою». Опция «Прокатная» сверх `[M]` недоступна (§6.2). |
| **E3** | Гонка запросов: слот заполнили параллельно (NFR-8) | **«Слот заполнен другим пользователем. Список обновлён.»** | Запись не создана; показать **обновлённые** `free_seats` / `free_rental_boards`; пересчитать доступные лимиты и состояние CTA; при нуле мест — увести/подсказать вернуться к слотам. |
| **E4** | Сетевой сбой при подтверждении | **«Не удалось загрузить. Проверьте соединение и попробуйте снова.»** ([foundations §6](00-foundations.md)) + кнопка **«Обновить»**/«Повторить» | **Идемпотентность в UI:** повтор не создаёт дублирующую бронь. **Клиент генерирует UUID `Idempotency-Key`** один раз на попытку отправки формы и **переиспользует тот же ключ** при повторе той же операции (например, после сетевого сбоя). Backend ключ **не генерирует**. Отдельного UI для ключа нет. До получения ответа CTA остаётся в состоянии загрузки (§6.4). |

**Запреты:** UI-ограничения на `seats_count` и `rental_count` заданы правилами лимитов **§6.5**
(степпер и переключатели ограничены сверху). UI предупреждает заранее; сервер — финальный
арбитр (E1–E3).

**Лейблы и сквозные тексты:**
- Доски: **«Своя доска»** / **«Прокатная доска»** ([foundations §6](00-foundations.md)).
- Офлайн-оплата (в блоке цены): **«Оплата на месте: наличные или перевод на карту.»**
  ([foundations §6](00-foundations.md)).

## 9. Доступность и mobile-first

База — [foundations §3, §7](00-foundations.md). Специфика экрана:

- **Крупный степпер** и **крупные переключатели** «Своя/Прокатная» — тач-зоны ≥ 44–48 pt,
  достаточный отступ между ними («мокрый палец», P1).
- Сценарий укладывается в **≤ 3 шага/экрана** до подтверждения (NFR-2): счётчик → доски →
  «Записаться».
- **Итоговая цена** — крупная, контрастная, читаемая на солнце; продублирована на CTA.
- Состояния переключателей и disabled-элементов передаются не только цветом, но текстом/формой.
- Все элементы имеют доступные имена (screen reader): «Место 2, гость, доска прокатная»,
  «Увеличить число мест» и т. п.
- Layout не «ломается» при системном увеличении шрифта; CTA не перекрывается клавиатурой.

## 10. Критерии приёмки дизайна (Gherkin)

### Позитивные

```gherkin
AC-001: Запись только себя со своей доской
  Дано клиент на экране оформления слота со свободными местами
  Когда он оставляет число мест = 1 и выбирает «Свою» доску
  Тогда итоговая цена равна price × 1 (строка «Прокат» скрыта)
  И кнопка «Записаться» активна
  И по подтверждению происходит переход на BS-002

AC-002: Запись с гостями (свои доски)
  Дано free_seats и потолок маршрута позволяют 3 места
  Когда клиент выставляет число мест = 3 и все доски — «Свои»
  Тогда появляются 3 строки выбора доски (вы + 2 гостя)
  И итоговая цена равна price × 3
  И кнопка «Записаться» активна

AC-003: Смешанные доски — прокат тарифицируется отдельно
  Дано свободно прокатных досок не меньше 1
  Когда клиент выбирает для части мест «Прокатную», для остальных «Свою»
  Тогда счётчик «Прокатных выбрано» отражает только прокатные места
  И итоговая цена равна price × seats_count + rental_price × rental_count (§6.5)
  И строка «Прокат» в блоке цены отражает rental_price × rental_count
  И кнопка «Записаться» активна
```

### Негативные

```gherkin
AC-004: E1 — нехватка мест
  Дано клиент подтверждает бронь
  Когда запрошено мест больше, чем свободно
  Тогда показывается «Недостаточно мест. Свободно: [N]. Уменьшите бронь до [N] мест.»
  И счётчик предлагается/откатывается к [N]
  И запись не создаётся

AC-005: E2 — нехватка прокатных досок
  Дано клиент выбрал больше прокатных мест, чем свободно досок
  Тогда показывается «Недостаточно прокатных досок. Свободно: [M]. Выберите меньше прокатных или свою доску.»
  И предлагается уменьшить число прокатных или переключить место на «Свою»
  И запись не создаётся

AC-006: E3 — слот заняли параллельно
  Дано клиент тапнул «Записаться»
  Когда сервер сообщает, что мест уже нет (гонка запросов)
  Тогда показывается «Слот заполнен другим пользователем. Список обновлён.»
  И на экране отображаются обновлённые свободные места и прокатные доски
  И двойная бронь/овербукинг не возникают

AC-007: E4 — сетевой сбой без дубля
  Дано сеть прервалась при подтверждении
  Тогда показывается «Не удалось загрузить. Проверьте соединение и попробуйте снова.» с повтором
  И повторная попытка не создаёт вторую бронь (идемпотентность в UI)

AC-008: Защита от двойного тапа
  Дано CTA в состоянии отправки
  Когда клиент повторно тапает «Записаться»
  Тогда второй запрос не отправляется
```

### Граничные

```gherkin
AC-009: Максимум мест = min(free_seats, потолок маршрута)
  Дано число мест достигло min(free_seats, потолок маршрута)
  Тогда кнопка «+» степпера неактивна
  И нельзя выбрать больше мест

AC-010: Ноль свободных прокатных досок
  Дано free_rental_boards = 0
  Тогда опция «Прокатная» недоступна на всех местах
  И доступна запись только из собственных досок
  И при выборе только «Своих» кнопка «Записаться» активна
```

## 11. Решения

1. **Имена гостей — не вводятся.** Достаточно числа мест (степпер 1–3), без полей имён (NFR-2/NFR-3). §4.1, §6.5.
2. **Цена проката — отдельный тариф** `rental_price`: итог = `price × seats_count + rental_price × rental_count`. §6.5.
3. **Поведение при E3 с нулём мест** — показываем обновлённые данные и **явную кнопку** возврата к списку слотов (без авто-перехода, чтобы клиент успел прочитать сообщение). §8 (E3).
4. *(Параметр API, не блокер)* **Идемпотентный ключ записи генерирует клиент** — UUID `Idempotency-Key` создаётся один раз на попытку отправки формы и переиспользуется при повторе той же операции (E4); backend ключ не генерирует, отдельного UI для ключа нет. §8 (E4). Согласовано с US-14 и sequence-диаграммой (клиент генерирует ключ).
