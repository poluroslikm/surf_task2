## 1. Назначение и контекст

Экран — единственная точка входа в приложение для неавторизованного пользователя. Его задача
(job-to-be-done): **дать клиенту попасть внутрь приложения максимально быстро, без обучения и
без пароля** — подтвердив номер телефона, чтобы тут же перейти к списку слотов и записаться на
прогулку.

Экран реализует принцип **минимального порога входа** (см. [`00-foundations.md`](00-foundations.md), §2, P3)
и обслуживает как **первичную регистрацию**, так и **повторный вход** по номеру телефона. Вход —
**трёхшаговый поток** на единой точке входа (без отдельного выбора «регистрация / вход»):

1. **Шаг 1 — Телефон.** Клиент вводит номер телефона.
2. **Шаг 2 — Код из SMS (OTP).** Клиент вводит код, отправленный на номер; есть таймер
   повторной отправки.
3. **Шаг 3 — Имя (только для нового пользователя).** Если номер ещё не зарегистрирован — клиент
   указывает имя. Известный номер этот шаг **пропускает** и сразу попадает в приложение.

## 2. Пользователь и цель

**Кто:** клиент SUP-клуба. Часто — **новый пользователь без опыта работы с приложением**, который
открывает его впервые прямо перед прогулкой.

**Контекст использования** (см. [`00-foundations.md`](00-foundations.md), §1):
- стоит на **берегу**, нередко на ярком **солнце** (блики на экране, нужен высокий контраст);
- руки могут быть **мокрыми / холодными** (промахи по мелким элементам);
- **спешит** перед выходом на воду — нет терпения на длинные формы и обучение.

**Цель пользователя:** «Хочу записаться на прогулку — быстро войти и не вспоминать пароль».
**Цель экрана:** провести вход за минимально возможное число действий (номер → код → имя для
новых) и сразу отдать пользователя в [SCR-002 «Список слотов»](SCR-002-slot-list.md).

## 5. Структура и иерархия

Каркас — по [`00-foundations.md`](00-foundations.md), §4.1. Таб-бар в НЗ отсутствует. Поток —
три последовательных шага; на шагах 2 и 3 в хедере есть кнопка «Назад» к предыдущему шагу.
Нижний CTA на каждом шаге — фиксированный, всегда видимый, **не перекрытый клавиатурой**.

**Шаг 1 — Телефон:**

├───────────────────────────────┤

│  на прогулку                   │
│                               │
│  Телефон                      │
│  [ +7 ___ ___-__-__ ]         │  ← поле «Телефон» (числовая клавиатура)
│                               │
│  Без пароля — вход по номеру   │  ← подсказка (caption)
│                               │
├───────────────────────────────┤
│  [     Получить код     ]     │  ← фикс. нижний CTA
└───────────────────────────────┘
```

**Шаг 2 — Код из SMS:**

```
┌───────────────────────────────┐
│  ‹ Назад   Подтверждение       │  ← хедер: назад к шагу 1
├───────────────────────────────┤

│  +7 999 123-45-67              │  ← номер из шага 1
│                               │
│  Код из SMS                    │
│  [ _ _ _ _ ]                  │  ← поле кода (числовая клавиатура)
│                               │
│  Отправить код повторно (00:30)│  ← таймер → ссылка по истечении
│                               │
├───────────────────────────────┤
│  [      Подтвердить      ]    │  ← фикс. нижний CTA
└───────────────────────────────┘
```

**Шаг 3 — Имя (только для нового пользователя):**

```
┌───────────────────────────────┐
│  ‹ Назад   Как вас зовут?       │  ← хедер
├───────────────────────────────┤
│                               │
│  Имя                          │
│  [____________________]       │  ← поле «Имя» (автофокус)
│                               │
│  Так к вам будут обращаться    │  ← подсказка (caption)
│                               │
├───────────────────────────────┤

└───────────────────────────────┘

> на шаге 2 сессия устанавливается и сразу открывается [SCR-002](SCR-002-slot-list.md).

## 6. Компоненты и поведение

### Шаг 1 · Поле «Телефон» (`phone`)
- Поле с **числовой / телефонной клавиатурой** и **маской ввода** национального формата.
- Маска подсказывает структуру номера и отсекает посторонние символы; пользователь не вводит разделители вручную.
- Автофокус при открытии экрана.
- Конкретный формат и допустимые префиксы — параметр конфигурации, не зашивается в макет.
- По одному и тому же номеру выполняется и регистрация, и повторный вход (уникальное поле).

### Шаг 1 · Кнопка «Получить код» (CTA)
- Во всю ширину контентной области, размер тач-зоны — по [`00-foundations.md`](00-foundations.md), §3.1.
- **Disabled:** пока телефон не заполнен/не прошёл формат-валидацию (состояние дублируется не только цветом, §3.2).
- **Enabled:** телефон валиден по формату.
- **Loading:** после нажатия — индикатор, повторные нажатия блокируются; по ответу бэкенда — переход на шаг 2.

### Шаг 2 · Поле «Код из SMS» (OTP)
- Поле с **числовой клавиатурой** на 4–6 знаков (точная длина — параметр конфигурации, не хардкод). Автофокус при переходе на шаг.
- Над полем — текст «Мы отправили код на `<номер>`» с номером из шага 1.
- **Таймер повторной отправки:** ссылка «Отправить код повторно» неактивна и показывает обратный отсчёт; по его истечении становится активной и сбрасывает таймер заново при повторе.
- **Кнопка «Подтвердить»:** disabled, пока код не введён полностью; loading при проверке; повторные тапы блокируются. По успеху: известный номер → [SCR-002](SCR-002-slot-list.md); новый номер → шаг 3.
- **Назад** в хедере возвращает на шаг 1 (исправить номер) без потери набранного номера.

### Шаг 3 · Поле «Имя» (`name`) — только для нового пользователя
- Показывается, **только если номер не был зарегистрирован ранее**. Известный номер этот шаг не видит.
- Однострочное текстовое поле с лейблом и плейсхолдером-примером, автофокус.
- Капитализация первой буквы — на усмотрение системной клавиатуры; без принудительных ограничений длины в UI (лимит — из правил валидации, не хардкод).
- **Кнопка «Продолжить»:** disabled, пока имя не заполнено/не прошло формат-валидацию; loading при отправке; по успеху — регистрация завершена, открывается [SCR-002](SCR-002-slot-list.md).

### Повторный вход
- При вводе ранее зарегистрированного телефона после подтверждения кода (шаг 2) пользователь входит в существующий аккаунт **без шага 3** (имя уже известно). Отдельного экрана/режима «вход» нет.

### Запрос разрешения на push
- На этом экране **не запрашивается**. По [`00-foundations.md` §8.1](00-foundations.md#81-напоминания-и-уведомления-fr-33-nfr-13) разрешение запрашивается после первой успешной записи (в [BS-002](BS-002-booking-success.md)).

## 7. Состояния экрана

Базовый паттерн — [`00-foundations.md`](00-foundations.md), §5. Специфика SCR-001:

| Состояние | Содержимое / Поведение | Правило |
|-----------|------------------------|---------|
| **Loading (старт)** | Минимальный — экран статичен, тяжёлых запросов нет | Сетевая загрузка не блокирует ввод. |
| **Content (шаг 1/2/3)** | Поля текущего шага + CTA: телефон → код → имя (для нового) | Основной сценарий. Шаги 2–3 имеют кнопку «Назад». |
| **Валидация полей** | Подсветка невалидного поля + текст ошибки под полем | Ошибка появляется не раньше, чем пользователь покинул поле / нажал CTA; не «ругается» во время набора. |
| **Отправка / ожидание (шаг 1, 2, 3)** | CTA в состоянии Loading, поля заблокированы | Защита от повторной отправки; ощущается < 2–3 с (P4). |
| **Ожидание повторной отправки кода (шаг 2)** | Ссылка «Отправить код повторно» неактивна + обратный отсчёт | По истечении таймера ссылка активируется. |
| **Неверный код (шаг 2)** | Сообщение об ошибке + возможность ввести заново / запросить новый код | Нейтральный тон, без вины; номер из шага 1 не теряется. |
| **Error (сеть)** | Заглушка/сообщение + повтор | Текст сетевой ошибки — сквозной (§6 foundations). |

## 8. Валидации и микрокопия

Тон — по [`00-foundations.md`](00-foundations.md), §6 (на «вы», без вины и давления).
Числовые лимиты не зашиваются в тексты — задаются правилами валидации/конфигурацией.

| Правило | Текст (микрокопия) |
|---------|--------------------|
| Телефон обязателен (шаг 1) | «Введите номер телефона» |
| Телефон в неполном/неверном формате (шаг 1) | «Похоже, номер введён не полностью» |
| Подсказка под формой (отсутствие пароля, NFR-3) | «Без пароля — входим по номеру телефона» |
| Лейбл/плейсхолдер телефона | «Телефон» / пример в национальном формате |
| Над полем кода (шаг 2) | «Мы отправили код на `<номер>`» |
| Неверный код (шаг 2) | «Неверный код. Проверьте и введите ещё раз» |
| Таймер повтора (шаг 2) | «Отправить код повторно (00:30)» → по истечении: «Отправить код повторно» |
| Имя обязательно (шаг 3) | «Укажите, как к вам обращаться» |
| Имя в недопустимом формате (шаг 3) | «Проверьте имя — кажется, тут лишние символы» |
| Подсказка под именем (шаг 3) | «Так к вам будут обращаться» |
| Ошибка отправки/подтверждения | «Не удалось войти. Попробуйте ещё раз» |

**Принципы валидации:**
- **Пароля нет** (NFR-3) — поля пароля на экране отсутствуют принципиально; подтверждение — кодом из SMS.
- **Минимум полей** (NFR-2 / NFR-3) — только `phone`, код и (для новых) `name`; ничего сверх необходимого.
- Валидируется **формат** телефона, **длина/формат** кода и формат имени; уникальность телефона и валидность кода проверяет бэкенд.

## 9. Доступность и mobile-first

Базовые правила — [`00-foundations.md`](00-foundations.md), §3 и §7. Специфика SCR-001:

- **Крупные поля и CTA** — рассчитаны на ввод «мокрым пальцем» на солнце (тач-зоны §3.1, контраст §3.2).
- **Числовая / телефонная клавиатура** для полей «Телефон» (шаг 1) и «Код из SMS» (шаг 2) — без переключения раскладок.
- **Автофокус** на активном поле каждого шага (телефон → код → имя); логичный переход к CTA.
- Лейблы и доступные имена у всех полей и кнопок (screen reader, §7 foundations).
- Нижний CTA **не перекрывается** экранной клавиатурой; форма скроллится при необходимости.
- Поддержка системного увеличения шрифта без поломки раскладки.

## 10. Критерии приёмки дизайна

**AC-001 — Шаг 1: минимальный ввод телефона**
- **Дано:** пользователь не авторизован и открыл приложение
- **Когда:** загрузился экран SCR-001 (шаг 1)
- **Тогда:** видно только поле «Телефон» и кнопка «Получить код»; поля пароля отсутствуют

**AC-002 — Шаг 1: CTA активируется при валидном телефоне**
- **Дано:** поле «Телефон» пусто
- **Когда:** пользователь ввёл корректный номер
- **Тогда:** кнопка «Получить код» переходит из disabled в enabled

**AC-003 — Шаг 2: отправка кода и его подтверждение**
- **Дано:** введён валидный телефон
- **Когда:** пользователь нажал «Получить код»
- **Тогда:** открывается шаг 2 с полем кода и таймером повторной отправки; код приходит в SMS

**AC-004 — Повторный вход известного номера без шага «Имя»**
- **Дано:** телефон уже зарегистрирован ранее
- **Когда:** пользователь подтвердил код на шаге 2
- **Тогда:** устанавливается сессия и сразу открывается [SCR-002 Список слотов](SCR-002-slot-list.md), шаг 3 пропускается

**AC-005 — Новый номер: шаг 3 «Имя» и завершение регистрации**
- **Дано:** номер не был зарегистрирован ранее и код подтверждён
- **Когда:** открывается шаг 3, пользователь вводит имя и нажимает «Продолжить»
- **Тогда:** регистрация завершается и открывается [SCR-002 Список слотов](SCR-002-slot-list.md)

**AC-006 (негативный) — Невалидный телефон блокирует шаг 1**
- **Дано:** пользователь ввёл телефон в неполном/неверном формате
- **Когда:** он пытается получить код
- **Тогда:** CTA остаётся неактивным либо показывается ошибка формата под полем, код не отправляется

**AC-007 (негативный) — Неверный код блокирует вход**
- **Дано:** пользователь ввёл неверный код на шаге 2
- **Когда:** он нажимает «Подтвердить»
- **Тогда:** показывается «Неверный код…», вход не выполняется; можно ввести заново или запросить новый код

**AC-008 (негативный) — Пустое имя блокирует шаг 3**
- **Дано:** на шаге 3 поле «Имя» пусто
- **Когда:** пользователь нажимает «Продолжить»
- **Тогда:** показывается подсказка заполнить имя, переход не происходит

**AC-009 (граничный) — Таймер и повторная отправка кода**
- **Дано:** пользователь на шаге 2, код только что отправлен
- **Когда:** идёт обратный отсчёт
- **Тогда:** ссылка «Отправить код повторно» неактивна до конца отсчёта, затем активируется

**AC-010 (граничный) — Защита от двойной отправки**
- **Дано:** пользователь нажал CTA любого шага
- **Когда:** идёт ожидание ответа и он нажимает кнопку повторно
- **Тогда:** повторная отправка не выполняется; CTA в состоянии Loading

**AC-011 (граничный) — Сохранение ввода при ошибке и «Назад»**
- **Дано:** на шаге 2 показана ошибка или пользователь нажал «Назад»
- **Когда:** он возвращается/повторяет
- **Тогда:** введённый номер сохранён, повтор без перезаполнения

## 11. Решения и параметры конфигурации

Открытые вопросы закрыты следующими решениями (см. также [`00-foundations.md` §8.1](00-foundations.md#81-напоминания-и-уведомления-fr-33-nfr-13)):

1. **Способ подтверждения входа — SMS-код (OTP).** Трёхшаговый поток: телефон → код → имя (для нового). Шаг ввода кода обязателен (§5, §6).
2. **Запрос разрешения на push — не на этом экране**, а после первой записи (в [BS-002](BS-002-booking-success.md)).
3. *(Параметр конфигурации, не блокер)* **Формат и допустимые префиксы телефона** — задаются конфигурацией; макет использует маску национального формата, конкретные страны/маску уточняет бэкенд.
4. *(Параметр конфигурации, не блокер)* **Длина кода и длительность таймера повторной отправки** — параметры конфигурации; в макете показаны как 4–6 знаков и обратный отсчёт.
