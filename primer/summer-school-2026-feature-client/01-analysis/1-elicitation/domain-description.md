# Описание домена и ограничения

> Этап 1. Артефакт 1 проекта: согласованное описание домена + список ограничений.
> Заполняется после ответов заказчика ([customer-questions.md](customer-questions.md)).

## Описание домена

**Бизнес.** «Волна» — небольшой клуб сапбординга (SUP) на городской набережной, работающий
третий сезон. Клуб организует **групповые прогулки** на сапах по реке. Прогулка длится
1.5–2 часа и проходит по одному из **маршрутов**: спокойный — для новичков, подлиннее — для
опытных. Каждую прогулку ведёт **инструктор** (сейчас их пятеро, летом нанимаются
подработчики).

**Процесс.** Владелец заранее формирует **расписание на неделю вперёд**: для каждой прогулки
задаются дата и время старта, маршрут, назначенный инструктор и число мест. Клуб располагает
**12 досками**; для новичковых групп берут не более 8 человек (тяжелее уследить), для
опытных — до 12 (по числу досок). Клиент видит свободные **слоты** прогулок и **записывается**,
указывая, поедет он со своей доской или возьмёт доску **напрокат**. До прогулки клиент может
**отменить** запись — при ранней отмене место освобождается, поздняя отмена оставляет
доску простаивать. Сейчас запись клиентов ведётся вручную (WhatsApp + тетрадь), что приводит к
двойным записям и путанице, особенно в выходные.

**Цель клиентского приложения.** Дать клиентам самостоятельную онлайн-запись на прогулки
(просмотр и фильтрация слотов, бронирование, выбор доски, отмена, свои бронирования), исключив
ручную путаницу и двойные брони. Управление расписанием и работу инструкторов/владельца
обеспечивает **существующая инфраструктура**; приложение потребляет её данные через API.

**Ключевые объекты домена:**

- **Прогулка / слот** — конкретный выезд группы: дата/время старта, маршрут, инструктор, всего и свободно мест. Создаётся в существующей инфраструктуре; клиенту доступен только для чтения.
- **Маршрут** — вариант прогулки (новичковый / опытный), определяет длительность и потолок мест (справочные данные, read-only).
- **Инструктор** — ведёт прогулку; для клиента — справочные данные слота (read-only).
- **Доска (SUP)** — инвентарь (всего 12); может выдаваться напрокат.
- **Запись (бронь)** — резерв места клиентом на конкретный слот: число мест, своя/прокатная доска, статус.
- **Клиент** — записывается на прогулки, отменяет запись.

## Пользователи и роли

> **Скоуп текущей работы — только клиентское мобильное приложение и API для него.**
> Роли Инструктор и Владелец/Администратор обслуживаются **существующей инфраструктурой**
> (бэкенд, расписание, админка) и в этот скоуп не входят.

| Роль | Описание | Цели | В скоупе |
| :-- | :-- | :-- | :-- |
| **Клиент** | Гость клуба, катается на сапах. Пользуется с телефона. | Увидеть и отфильтровать свободные слоты, записаться (себя и гостей), выбрать свою/прокатную доску, видеть свои бронирования, отменить запись, получать напоминания. | **Да** |

> Инструктор и Владелец/Администратор взаимодействуют с системой через **уже реализованные**
> интерфейсы существующей инфраструктуры. Клиентское приложение лишь потребляет подготовленные
> ими данные (слоты, маршруты, инструкторы) через API.

## Ограничения

**Бизнес-ограничения**

- Всего **12 досок** — общий потолок инвентаря и прокатного фонда.
- Новичковая группа — **не более 8 человек**; опытная — до 12 (по числу досок).
- Лимит мест слота = `min(свободные доски, потолок маршрута)`.
- **Политика отмен:** бесплатная отмена не позднее чем за **2 часа** до старта; поздняя
  отмена (<2 ч) фиксируется без освобождения места и без денежных штрафов.
- **Оплата офлайн** (наличные / перевод на карту) — приложение лишь фиксирует запись и
  показывает цену.
- **Период списка слотов по умолчанию — 7 дней (R-027).** По умолчанию клиенту показываются
  слоты на **ближайшие 7 дней** от текущего момента — это совпадает с горизонтом планирования
  расписания («на неделю вперёд»). Более длинный период доступен через **фильтр дат**. При
  отсутствии расписания на ближайшие дни показывается **empty state** («Пока нет доступных
  прогулок»).
- **Отмена прогулки клубом / по погоде (R-008).** Если прогулку отменяет клуб (организатор или
  из-за погоды), бронь на стороне клиента **не удаляется**, а получает статус **«Отменена
  клубом»** с указанием причины; клиенту приходит **push-уведомление**. **Повторная запись на
  отменённый слот запрещена.** Само действие отмены слота выполняется в существующей
  инфраструктуре (вне скоупа клиентского приложения).

**Технические / платформенные**

- **Mobile-first**: клиентское приложение рассчитано на телефон (использование на берегу).
- Клиентское приложение взаимодействует с **существующим бэкендом через API**; слоты, маршруты и
  инструкторы доступны только для чтения.
- **Граница серверной интеграции (R-004).** Детали серверной реализации — транзакционность,
  SLA, маппинг сущностей на внутренние модели бэкенда — **вне скоупа клиентского приложения**.
  Существующий бэкенд — **black-box источник истины**: гарантия «**0 двойных броней**»
  обеспечивается на его стороне (атомарная проверка свободных мест при бронировании). Со стороны
  клиента граница закрывается **контрактом API**: приложение полагается на ответы бэкенда и
  корректно обрабатывает отказ, если мест уже нет.
- **Данные существующей инфраструктуры (R-015).** Это **учебный/тестовый проект**, исторических
  данных и легаси-схемы нет. **Канонической схемой считается контракт API** (OpenAPI в
  `01-analysis/api/`): бэкенд по условию отдаёт ровно те поля, что описаны в контракте, включая
  `rental_price`, `geometry`, `meeting_point` и `free_rental_boards`. Поэтому **миграция/backfill
  и стратегия дозаполнения исторических слотов вне скоупа** — отдельных требований к ним не
  формулируем.

**Проект**

- **Срок** — к началу следующего сезона, ~2 месяца (жёсткая цель).
- **Бюджет** ограничен → начинаем с самого нужного, «навороты» позже.

**Границы скоупа**

Скоуп — **клиентское мобильное приложение и API для него**. Управление расписанием, роли
инструктора и владельца, оценки, админка — это **существующая инфраструктура**, вне скоупа.

| В скоупе клиентского приложения | Вне скоупа (существующая инфраструктура / не делаем сейчас) |
| :-- | :-- |
| Регистрация/вход клиента по имени и телефону | Роли Инструктора и Владельца/Администратора, админ-экран |
| Просмотр и **фильтрация** списка слотов (дата, тип маршрута, наличие мест, инструктор) | Создание и редактирование расписания (слоты приходят из существующего бэкенда через API) |
| Самостоятельная запись, в т.ч. несколько мест за раз | Оценка инструктора и рейтинги — **Phase 2** (R-001; заказчик ранее называл функцию критичной, перенос в Phase 2 согласован с заказчиком 2026-06-25) |
| Выбор «своя / прокатная доска», контроль лимита мест и прокатного фонда | Отметка явки/неявки, отмена слота инструктором + массовое уведомление |
| Список своих бронирований | Онлайн-оплата; ручная отметка «оплачено» |
| Отмена записи с правилом 2 часов | Автоматический учёт погоды по прогнозу |
| Цена и фиксация записи (оплата офлайн) | Программа лояльности для постоянных клиентов |
| Напоминания о записи — по возможности в срок | Публичный показ рейтинга клиентам — **Phase 2** (R-001) |

## Глоссарий

| Термин | Определение |
| :-- | :-- |
| **Прогулка / слот** | Запланированный групповой выезд: дата/время старта, маршрут, инструктор, всего и свободно мест. |
| **Маршрут** | Вариант прогулки по реке: новичковый (спокойный, потолок 8) или опытный (подлиннее, до 12). |
| **Группа** | Состав клиентов, записанных на один слот. |
| **Доска (SUP)** | Сапборд из инвентаря клуба; всего 12; может выдаваться напрокат. |
| **Аренда / прокат** | Выдача клубной доски клиенту, у которого нет своей. |
| **Своя доска** | Клиент приезжает с собственной доской; занимает место в группе, но не прокатный фонд. |
| **Запись (бронь)** | Резерв одного или нескольких мест на конкретный слот: число мест, своя/прокатная доска, статус. |
| **Отмена** | Снятие записи. Клиентом: ранняя (≥2 ч до старта) освобождает место; поздняя (<2 ч) фиксируется без освобождения места. Клубом (организатором / по погоде): бронь переходит в статус «Отменена клубом» с причиной, клиент получает push, повторная запись на этот слот запрещена. |
| **Отмена клубом** | Отмена прогулки организатором или из-за погоды (выполняется в существующей инфраструктуре). На стороне клиента бронь сохраняется со статусом «Отменена клубом» + причина; отправляется push-уведомление; повторная запись на отменённый слот недоступна. |
| **Администратор / владелец** | Игорь — управляет расписанием и инструкторами через существующую инфраструктуру (вне скоупа клиентского приложения). |
| **MVP** | Минимально жизнеспособный продукт: набор функций к старту сезона. |
