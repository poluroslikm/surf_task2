# Бизнес-требования

> Этап 2. Зачем бизнесу продукт: цели, метрики успеха, ограничения верхнего уровня.

> **Скоуп — клиентское мобильное приложение и API для него.** Бизнес-цели, обслуживаемые
> существующей инфраструктурой (картина для владельца, интерфейс инструктора, оценки), в этот
> список не входят.

> **О нумерации требований.** ID требований (BR-NN, M-NN) сохраняются стабильными для
> трассировки. В нумерации есть пропуски — это следствие осознанного сужения скоупа (часть ранее
> заведённых требований вынесена в развитие/Phase 2 или относится к существующей инфраструктуре).
> Нумерация намеренно не нормализуется, чтобы не ломать связи в таблице-источнике и смежных
> документах.

> **Роли в текущей поставке.** В текущую поставку входит только роль **Клиент** (самостоятельная
> запись на прогулки с мобильного приложения). Интерфейсы инструктора и администратора/владельца
> — **вне скоупа текущей поставки**: они обеспечиваются существующей инфраструктурой. Это
> осознанное решение в рамках MVP-подхода и жёсткого срока: продукт фокусируется на клиентском
> сценарии, который снимает основную боль (хаос записи и двойные брони), а серверные/ролевые
> функции остаются на стороне существующего бэкенда.

## Метрики успеха

> Целевые значения — разумные ориентиры для MVP (как в исходниках: точные пороги фиксируются как допущение, при наличии данных уточняются).

| ID | Метрика | Целевой ориентир | Связь с требованиями |
| :-- | :-- | :-- | :-- |
| M-1 | Доля записей, сделанных клиентами онлайн самостоятельно (от общего числа записей за сезон). | ≥ 70% к концу первого сезона | BR-2 |
| M-2 | Число двойных броней / конфликтов по местам. | 0 (полное устранение) | BR-1 |
| M-4 | Простой досок из-за поздних отмен (доля поздних отмен от записанных). | Снижение относительно текущего уровня (целевой ориентир ≤ 10%) | BR-5 |
| M-6 | Готовность к старту сезона. | Запуск в срок ~2 месяцев с критичным набором функций (просмотр слотов, запись, отмены) | BR-7 |

## Ограничения верхнего уровня

- **Срок и бюджет.** Жёсткая цель — начало следующего сезона (~2 месяца); бюджет ограничен → строго MVP-подход, «навороты» откладываются. (Бриф; Ответы заказчика № 20)
- **Оплата офлайн на старте.** Онлайн-оплата вне границ скоупа. (Ответы заказчика № 9, 11)
- **Mobile-first.** Клиентское приложение — с телефона; данные получает из существующего бэкенда через API. (Ответы заказчика № 19)
- **Вне скоупа.** Функции инструктора/владельца, расписание-CRUD и админка обеспечиваются существующей инфраструктурой. Онлайн-оплата, автоматический учёт погоды, программа лояльности, публичный рейтинг — направления развития. (Описание домена → «Границы скоупа»)
- **Оценки инструкторов — Phase 2 (подтверждено заказчиком).** Оценки инструкторов осознанно вынесены в Phase 2 и не входят в MVP. Заказчик ранее называл функцию критичной, поэтому она не отбрасывается, а планируется к реализации после запуска критичного набора MVP (BR-7) — это сознательное сужение скоупа ради попадания в жёсткий срок начала сезона. **Перенос согласован с заказчиком** (2026-06-25). (Бриф; Описание домена → «Границы скоупа»; решение R-001)
