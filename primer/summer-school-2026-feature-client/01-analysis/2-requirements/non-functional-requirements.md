# Нефункциональные требования

> Этап 2. Как система должна работать: производительность, надёжность, безопасность,
> доступность, удобство. Частая ошибка ИИ — путать ФТ и НФТ.

> **Скоуп — клиентское мобильное приложение и API для него.** НФТ относятся к клиентскому
> приложению; функции инструктора/владельца/админки обеспечиваются существующей инфраструктурой.

> **О нумерации ID (R-030).** Разрывы в нумерации NFR-NN (напр. отсутствуют NFR-5, NFR-14)
> — следствие сужения скоупа до клиентского приложения и удаления ранее существовавших
> требований. Существующие ID сохранены, чтобы не ломать внешние ссылки; новые требования
> нумеруются продолжением схемы. Сквозная нормализация нумерации не выполняется намеренно.

> **Целевая платформа (R-002).** Продукт — нативное/гибридное мобильное приложение для iOS и
> Android, распространяемое через App Store и Google Play (см. NFR-4). Требования к push,
> хранению токена, сессии и a11y сформулированы исходя из этого решения.

| NFR-17 | Уведомления | Канал уведомлений MVP — системный push (согласуется с native app, NFR-4). Доставка обеспечивается push-сервисами платформ (APNs/FCM). Push-токен регистрируется на бэкенде после получения системного разрешения; при отзыве разрешения или невалидном токене отправка прекращается без ошибки для пользователя. Напоминание о записи отправляется заблаговременно до старта прогулки (см. NFR-13, `reminder_hours`). **Тайминг напоминаний в MVP — за 24 ч и за 2 ч до старта (`reminder_hours = [24, 2]`)** (согласовано с заказчиком 2026-06-25); конкретные часы клиент берёт из ответа `createBooking`, не хардкодит. SMS/email и внутриканальный inbox — вне скоупа MVP (Phase 2). | Средний | R-006; mobile-app-spec (LOGIC-007); foundations |
| NFR-18 | Безопасность / Сессия | Сессия — JWT с парой токенов: короткоживущий **access** (срок жизни ~15 мин) и долгоживущий **refresh** (срок жизни ~30 дней). Access автоматически обновляется по refresh без участия пользователя; перелогин (OTP-вход) требуется только при истечении/отзыве refresh. Оба токена хранятся исключительно в защищённом хранилище платформы (Keychain/Keystore — см. NFR-4), не в незащищённом storage/логах. **401-flow:** при `401` от API клиент пытается обновить access по refresh; при неуспехе — очищает токены из хранилища и переводит пользователя на экран авторизации. Весь обмен с API — только по TLS (HTTPS, TLS 1.2+). | Высокий | R-016; mobile-app-spec (LOGIC-001); auth/models.yaml |
| NFR-19 | Безопасность / Антифрод OTP | Отправка OTP ограничена rate-limit'ами: не более ~1 запроса кода в 30–60 с на номер и не более ~5 запросов в час на номер/устройство; ограничено число попыток ввода кода (напр. ≤5) до временной блокировки; код имеет конечный TTL. При исчерпании лимитов пользователю показывается понятное сообщение с указанием времени до повторной попытки. | Высокий | R-016; auth/api.yaml; common/models.yaml |
| NFR-20 | Безопасность / Приватность (ПДн) | Базовый набор требований к ПДн для учебного проекта (имя, телефон): (1) явное согласие пользователя и доступная политика конфиденциальности до отправки данных; (2) удаление/анонимизация данных пользователя по запросу на удаление аккаунта, включая анонимизацию связанных броней (связать с R-009); (3) передача всех ПДн только по TLS (см. NFR-18); (4) маскирование номера телефона в UI (напр. `+7 *** *** ** 67`) и в логах (запрет логирования полного номера и OTP); (5) базовый retention: прошедшие/отменённые брони хранятся **12 месяцев** после события (завершения/отмены прогулки), далее анонимизируются. **Вне скоупа MVP (Phase 2):** полный комплаенс 152-ФЗ — экспорт данных субъекта, журнал/аудит доступа администраторов, формализованные юридические основания обработки. | Высокий | R-017; profile/api.yaml |
| NFR-21 | Производительность (измеримая) | Целевые метрики при «нормальной сети» (4G/Wi-Fi, ≥5 Мбит/с): получение списка слотов и открытие карточки слота — **p95 < 2.5 с**; `createBooking` и `cancelBooking` — **p95 < 1.5 с**. Метрики измеряются на стороне клиента от инициации запроса до готовности данных к отображению. Уточняет и заменяет качественную формулировку NFR-6. | Высокий | R-018; NFR-6 |
| NFR-22 | Производительность / Нагрузка | Расчётный пик — выходные в сезон, оценочно до ~50 одновременных активных пользователей. Под пиком сохраняются метрики NFR-21. **Сценарий нагрузочного теста на гонку бронирований:** N (≈20–50) клиентов одновременно вызывают `createBooking` на один слот с числом мест, заведомо превышающим остаток; проверяется, что подтверждается ровно столько броней, сколько было свободных мест/прокатных досок, остальным возвращается корректная ошибка (`slot_full`), двойных броней и овербукинга нет (см. NFR-8), p95 успешного ответа удерживается в пределах NFR-21. Уточняет NFR-7. | Высокий | R-018; NFR-7; NFR-8 |
| NFR-23 | Доступность (Availability) | Целевая доступность — **99%** в рабочие часы клуба в течение сезона. Окно измерения и все временные правила трактуются в **локальной таймзоне клуба** (по умолчанию — единая локальная зона клуба). Конкретные рабочие часы клуба, границы сезона и таймзона фиксируются как **обязательный pre-launch config** с владельцем (владелец продукта / эксплуатация); **критерий приёмки:** значения заданы и проверены до запуска. Из расчёта доступности **исключаются**: плановые работы (с предварительным уведомлением) и недоступность внешней инфраструктуры (картографический сервис, SMS/OTP-провайдер, push-сервисы APNs/FCM). **Деградация:** при отказе картографического сервиса — fallback-режим без карты (адрес/координаты текстом), бронь работает; при отказе push — бронирование и отмена работают без напоминаний. Уточняет NFR-10. | Средний | R-019; NFR-10 |
| NFR-24 | Надёжность / Offline | Просмотр уже загруженных данных (списка слотов, своих броней) доступен офлайн из кэша с явной пометкой «данные могли устареть». Мутации (создание/отмена брони) офлайн **запрещены** — показывается понятное сообщение о необходимости подключения. Единый паттерн Error/Retry для всех сетевых ошибок. Таймаут сетевого запроса — ~10 с. При неопределённом результате создания брони (таймаут/обрыв) клиент повторяет запрос с тем же `Idempotency-Key` (см. R-022), что исключает дубли. | Средний | R-020; R-022; foundations |
| NFR-25 | Доступность (A11y) | Приёмочные критерии доступности (проверяемые при приёмке): контраст текста и ключевых элементов соответствует WCAG 2.1 уровня AA; минимальный размер интерактивных touch-target ≥ 44pt; поддержка системного увеличения шрифта (Dynamic Type / шкала шрифта системы) без потери функциональности; все интерактивные элементы и изображения имеют labels для screen reader (VoiceOver/TalkBack); смысл (статусы, ошибки, доступность мест) не передаётся одним только цветом; интерфейс остаётся работоспособным и читаемым на малых экранах. Уточняет NFR-1. | Средний | R-029; NFR-1; foundations |
| NFR-26 | Интеграции / Картография | Отображение точки/маршрута через **Яндекс.Карты**: геометрия слота **обязательна** (R-011). API-ключ Яндекс.Карт хранится как **параметр конфигурации окружения** (не в коде, не в модели данных, не в репозитории), ротация ключа не требует пересборки клиента. Учитываются квоты/лимиты запросов провайдера; при превышении квоты или недоступности картографического сервиса — **fallback-режим без карты**: адрес/координаты показываются текстом (см. FR-44, NFR-23), бронирование продолжает работать. | Средний | R-011; FR-44; NFR-23 |
