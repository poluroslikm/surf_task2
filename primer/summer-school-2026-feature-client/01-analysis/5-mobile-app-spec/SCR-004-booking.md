## Обзор

Экран собирает параметры брони (число мест, вариант доски на каждое место, итоговую цену) и
фиксирует запись на слот. Это **самый сложный экран** клиентского сценария: на нём пересекаются
два независимых лимита — места группы (`free_seats`, потолок маршрута) и прокатный фонд
(`free_rental_boards`) — и обрабатываются конкурентные записи (гонка / овербукинг).

Экран — **последний шаг перед подтверждением** в потоке записи: список → карточка слота →
оформление → подтверждение. Соблюдается правило **≤ 3 экранов до подтверждения** (NFR-2):
от [SCR-002](SCR-002-slot-list.md) до [BS-002](BS-002-booking-success.md) клиент проходит
карточку ([SCR-003](SCR-003-slot-card.md)) и этот экран, без лишних шагов. Таб-бар скрыт — это
вложенный экран потока.

Оплата офлайн: экран показывает итоговую цену и фиксирует запись, онлайн-оплаты нет.

### User Story

> Как **Клиент**, я хочу записаться на выбранный слот, указать число мест (себя и до 2 гостей)
> и выбрать для каждого места свою или прокатную доску, увидеть итоговую цену и подтвердить
> бронь, чтобы гарантированно занять место на прогулке.

### Бизнес-ценность

- Замена записи через WhatsApp и бумажную тетрадь — самостоятельная запись клиента.
- Исключение двойных броней и овербукинга (главная боль заказчика, NFR-8) за счёт серверного
  арбитража лимитов при атомарном создании брони.
- Прозрачность стоимости до подтверждения (цена места + отдельный тариф проката), офлайн-оплата.

## Входные данные

| Название | Тип | Возможные значения | Описание |
|----------|-----|-------------------|----------|
| `slotId` | Параметр навигации | UUID | Идентификатор слота, переданный с [SCR-003](SCR-003-slot-card.md). |
| `slot` | Кэш / Состояние | объект `Slot` | Данные слота, переданные/закэшированные с предыдущего экрана: `start_at`, `route` (с потолком маршрута `capacity_cap`), `instructor`, `free_seats`, `free_rental_boards`, `price`, `rental_price`, `status`. Источник всех чисел экрана — **без хардкода**. |
| `seats_count` | Состояние формы | `1…max_seats` | Число мест. По умолчанию `1`. Верхняя граница — см. [LOGIC-002](09_Логики/LOGIC-002_Расчёт-доступности.md). |
| `board_choice[i]` | Состояние формы | `own` / `rental` | Для каждого места `i` (`1…seats_count`) — вариант доски. |
| `rental_count` | Состояние (производное) | `0…seats_count` | Число мест с `board_choice = rental`. Считается из переключателей. |
| `idempotencyKey` | Состояние (генерируется на клиенте) | UUID | Ключ идемпотентности, **обязателен** при `createBooking` (R-022). Генерируется один раз на попытку отправки формы и переиспользуется при повторе после сетевого сбоя (E4, NFR-9): повтор той же операции (тот же ключ + то же тело) → идентичный ответ, дубль не создаётся. |

> Числовые лимиты и цены (`free_seats`, `free_rental_boards`, потолок маршрута `route.capacity_cap`,
> `price`, `rental_price`) приходят из данных слота и **не зашиваются** в макет или тексты.
> Единственная константа — групповой лимит **3** (себя + до 2 гостей, FR-12).

**Инициализация формы.** При открытии экрана форма строится из переданного/закэшированного `slot`:
- `seats_count = 1` (по умолчанию — сам клиент).
- `board_choice[1]` — вариант доски по умолчанию по [LOGIC-002](09_Логики/LOGIC-002_Расчёт-доступности.md): «Прокатная», если `free_rental_boards ≥ 1`, иначе «Своя».
- Данные слота (дата/время, маршрут, инструктор, лимиты, цены) — из объекта `slot`.
- Полей профиля на экране **нет** — форма профилем (имя, телефон) **не предзаполняется**.

## Инициализация

Экран открывается с уже полученными данными слота (`slot`), переданными с [SCR-003](SCR-003-slot-card.md).
При открытии **сетевые запросы не отправляются** — форма строится из переданного `slot`. Создание
брони (`createBooking`) — это запрос **по действию** (тап CTA), а не при инициализации.

Опционально, при возврате к экрану или для актуализации лимитов перед показом, может выполняться
повторный `getSlot(slotId)` — но это не блокирующая инициализация; базовый сценарий использует
переданные данные (P4 — воспринимаемая скорость, без лишних ожиданий).

### Диаграмма загрузки

```mermaid
flowchart LR
    Start([onEnter]) --> Have{slot передан с SCR-003?}
    Have -->|Да| Content([Content: форма из slot])
    Have -->|Нет / актуализация| GS[getSlot slotId]
    GS --> Content
```

### Запросы при открытии

| № | Запрос | Критичный | Зависит от | Условие |
|---|--------|-----------|------------|---------|
| 1 | — (данные из `slot`, переданного с SCR-003) | Да | — | Всегда (базовый сценарий) |
| 2 | [getSlot](#getslot) | Нет | — | `slot` не передан / требуется актуализация лимитов |

> Создание брони — запрос по действию, см. [createBooking](#createbooking).

### getSlot

**Тип:** REST  
**Метод:** GET  
**Спецификация:** [../api/slots/api.yaml](../api/slots/api.yaml) → `getSlot`

**Триггер:** Инициализация (только если `slot` не передан или нужна актуализация лимитов)

**Параметры:**

| Параметр | Тип | Обязательность | Источник | Описание |
|----------|-----|----------------|----------|----------|
| `slotId` | string (uuid) | Да | Параметр навигации | Идентификатор слота. |

**Обработка ответа:**

| Результат | Условие | UI-реакция |
|-----------|---------|------------|
| Загрузка | — | Скелетон-шиммер формы |
| Успех 200 | `status = scheduled`, `free_seats > 0` | Построить форму; пересчитать лимиты ([LOGIC-002](09_Логики/LOGIC-002_Расчёт-доступности.md)) |
| Успех 200 | `free_seats = 0` или `status = cancelled` | Состояние «нет мест / слот недоступен», CTA скрыт, кнопка «Вернуться к слотам» |
| 4xx с `message` | тело `Error` с непустым `message` | Снек с текстом из `message` (00-foundations §6 / [LOGIC-008](09_Логики/LOGIC-008_Паттерн-состояний-экрана.md) Шаг 6) |
| 5xx | `internal_error` | Error-state первичной загрузки: заглушка + кнопка «Обновить» (повтор `getSlot`). Это **не** снек действия — показывать нечего (00-foundations §6.3). |
| Сеть | Нет соединения / таймаут | Error-state первичной загрузки: заглушка «Не удалось загрузить. Проверьте соединение и попробуйте снова.» + кнопка «Обновить» (00-foundations §6 / §6.3) |

### createBooking

**Тип:** REST  
**Метод:** POST  
**Спецификация:** [../api/bookings/api.yaml](../api/bookings/api.yaml) → `createBooking`

**Триггер:** Тап на кнопку «Записаться» (CTA)

**Заголовки:**

| Заголовок | Тип | Обязательность | Источник | Описание |
|-----------|-----|----------------|----------|----------|
| `Idempotency-Key` | string (uuid) | **Да (обязателен, R-022)** | `idempotencyKey` из состояния | Обязательный ключ идемпотентности (NFR-9, R-022). Один и тот же ключ переиспользуется при повторе после сетевого сбоя (E4): повтор с тем же ключом и **тем же телом** → идентичный ответ ранее созданной брони (тот же 201), дубль не создаётся; тот же ключ с **другим телом** → 409 `idempotency_key_conflict`. Ключ хранится на сервере ≥ 24 ч. |

**Тело запроса** (`CreateBookingRequest`):

| Параметр | Тип | Обязательность | Источник | Описание |
|----------|-----|----------------|----------|----------|
| `slot_id` | string (uuid) | Да | `slotId` | Идентификатор слота. |
| `seats_count` | integer `1…3` | Да | Состояние формы | Число мест. |
| `rental_count` | integer `0…seats_count` | Да | Производное из переключателей | Число прокатных досок. |

**Обработка ответа:**

| Результат | Условие (`code` в теле `Error`) | UI-реакция |
|-----------|--------------------------------|------------|
| Загрузка | — | CTA → индикатор загрузки; форма блокируется; повторный тап игнорируется (NFR-8) |
| Успех **201** | тело `CreateBookingResponse` (`Booking` + `price_total` + `is_first_booking` + `reminder_hours`) | Переход на [BS-002](BS-002-booking-success.md) с объектом `booking` (итог берётся из серверного `price_total`, R-005) |
| **409 Conflict** | `slot_full` | **E1/E3** — нехватка мест / гонка. Запись не создана. Прочитать `details.available_seats` → обновить `free_seats`, пересчитать лимиты, при необходимости откатить `seats_count`. Снек/нотис с текстом. CTA снова активен (если бронь снова валидна). |
| **409 Conflict** | `double_booking` | **E3** — двойная бронь (та же бронь уже создана). Запись не дублируется. Показать сообщение и увести на [BS-002](BS-002-booking-success.md)/«Мои записи» либо на [SCR-003](SCR-003-slot-card.md). |
| **409 Conflict** | нехватка прокатных досок (`details.available_rental_boards` < `rental_count`) | **E2** — обновить `free_rental_boards`, предложить уменьшить число прокатных мест или переключить место(а) на «Свою». |
| **409 Conflict** | `idempotency_key_conflict` | Тот же `Idempotency-Key` повторно отправлен с **другим телом** (R-022). Защитный сценарий: сгенерировать новый `idempotencyKey` для изменённой формы; снек «Не удалось оформить запись. Попробуйте ещё раз.» CTA снова активен. |
| **410 Gone** | `slot_cancelled` | Слот отменён организатором. Показать «Слот отменён и больше недоступен.»; CTA скрыт; кнопка «Вернуться к слотам» → [SCR-002](SCR-002-slot-list.md). |
| **422 Unprocessable** | `slot_started` | Слот уже стартовал. Показать «Слот уже стартовал — запись недоступна.»; CTA скрыт; «Вернуться к слотам». |
| **400 Bad Request** | `bad_request` | Параметры вне диапазона (защитный сценарий — UI не должен такое отправлять). Снек «Не удалось оформить запись. Проверьте параметры.» |
| **401** | `unauthorized` | Снек «Сессия истекла, войдите снова», затем перевод на повторную авторизацию ([SCR-001](SCR-001-registration.md)). |
| **Прочие 4xx с `message`** | любой `code`, тело `Error` с непустым `message` | Снек с текстом из `message` (общее правило 00-foundations §6 / [LOGIC-008](09_Логики/LOGIC-008_Паттерн-состояний-экрана.md) Шаг 6). CTA возвращается в активное состояние. |
| **4xx без `message`** (дефолт) | тело `Error` без `message` | Снек «Не удалось выполнить. Попробуйте ещё раз.» (00-foundations §6). CTA возвращается в активное состояние. |
| **5xx / default** | `internal_error` | Снек «Что-то пошло не так. Попробуйте ещё раз позже.» (00-foundations §6). CTA возвращается в активное состояние. |
| **Сеть** | Нет соединения / таймаут | **E4** — «Не удалось выполнить. Проверьте соединение и повторите.» + кнопка «Повторить». Повтор отправляется с **тем же** `Idempotency-Key` (дубль не создаётся, NFR-9). До получения ответа CTA остаётся в загрузке. |

## Макет экрана

### Структура

```
┌─────────────────────────────────────┐
│ ←  Оформление записи                 │  хедер (фикс., без таб-бара)
├─────────────────────────────────────┤
│ Сб, 21 июн · 10:00                    │  сводка слота
│ Маршрут «Новичковый» · Анна          │  (read-only)
│ Свободно мест: N · прокат: M         │
│ ─────────────────────────────────── │
│ Число мест                           │  счётчик
│            [ − ]   2   [ + ]          │  (макс = min(free_seats, потолок, 3))
│ Можно записать до K мест             │  подпись лимита (из данных)
│ ─────────────────────────────────── │
│ Доска для каждого места              │  блок выбора доски
│ Место 1 (вы)     [Своя |•Прокатная] │
│ Место 2 (гость)  [•Своя | Прокатная]│
│ Прокатных выбрано: 1 из M            │  подпись фонда (из данных)
│ ─────────────────────────────────── │
│ Места: 1 200 ₽ × 2        2 400 ₽    │  блок цены: места
│ Прокат: 500 ₽ × 1           500 ₽    │  прокат (только если rental_count>0)
│ Итого                     2 900 ₽    │  крупно, контрастно
│ Оплата на месте: наличные/перевод    │  текст (foundations §6)
├─────────────────────────────────────┤
│ [        Записаться · 2 900 ₽      ] │  фикс. нижний CTA
└─────────────────────────────────────┘
```

### Компоненты

| Компонент | Описание | Обязательность |
|-----------|----------|----------------|
| Хедер | «Назад» + заголовок «Оформление записи». Таб-бар скрыт. | Да |

### 2. Счётчик мест (`seats_count`)

| Элемент | Описание | Источник данных | Валидация | Действие |
|---------|----------|-----------------|-----------|----------|
| Кнопка «−» | Уменьшить число мест | состояние `seats_count` | — | `seats_count -= 1`, убрать последнюю строку доски |
| Число | Текущее `seats_count` | состояние | — | — |
| Кнопка «+» | Увеличить число мест | состояние `seats_count` | — | `seats_count += 1`, добавить строку доски со значением по умолчанию |
| Подпись лимита | «Можно записать до K мест» (`K = max_seats`) | [LOGIC-002](09_Логики/LOGIC-002_Расчёт-доступности.md) | — | — |

**Логика:**
- Степпер: [LOGIC-002](09_Логики/LOGIC-002_Расчёт-доступности.md) — `max_seats = min(free_seats, route.capacity_cap, 3)`.
- При уменьшении `seats_count` лишние строки доски (с конца) убираются; при увеличении — добавляется строка со значением по умолчанию (см. блок 3).

**Условия доступности:**
- «−» disabled при `seats_count = 1` (минимум — сам клиент).
- «+» disabled при `seats_count = max_seats` (граница видна, без скрытой блокировки).

### 3. Блок выбора доски (на каждое место)

| Элемент | Описание | Источник данных | Валидация | Действие |
|---------|----------|-----------------|-----------|----------|
| Строка «Место i» | «Место 1 (вы)», «Место 2 (гость)»… | состояние формы | — | — |
| Переключатель «Своя/Прокатная» | Сегментный, 2 опции | `board_choice[i]` | — | Переключить значение → пересчитать `rental_count` и цену |
| Счётчик проката | «Прокатных выбрано: `rental_count` из `free_rental_boards`» | [LOGIC-002](09_Логики/LOGIC-002_Расчёт-доступности.md) | — | — |

**Логика:**
- Переключатели: [LOGIC-002](09_Логики/LOGIC-002_Расчёт-доступности.md) — `rental_count ≤ free_rental_boards`.
- **Значение по умолчанию нового места:** «Прокатная», если `rental_count < free_rental_boards`; иначе «Своя».
- «Своя» занимает место в группе, но **не** уменьшает прокатный фонд; «Прокатная» уменьшает фонд (FR-14, UC-1 A1/A2).

**Условия доступности:**
- Опция «Прокатная» на ещё не-прокатных местах недоступна, когда `rental_count = free_rental_boards` (поясняющая подпись, E2). При `free_rental_boards = 0` «Прокатная» недоступна на всех местах.

### 4. Блок цены

| Элемент | Описание | Источник данных | Валидация | Действие |
|---------|----------|-----------------|-----------|----------|
| Строка «Места» | `slot.price × seats_count` | `slot.price`, `seats_count` | — | — |
| Строка «Прокат» | `slot.rental_price × rental_count` (скрыта при `rental_count = 0`) | `slot.rental_price`, `rental_count` | — | — |
| «Итого» (превью) | `slot.price × seats_count + slot.rental_price × rental_count` | [LOGIC-003](09_Логики/LOGIC-003_Расчёт-цены-брони.md) (R-010; итог брони — серверный `price_total`, R-005) | — | — |
| Напоминание об оплате | «Оплата на месте: наличные или перевод на карту.» | foundations §6 | — | — |

**Логика:**
- Цена: [LOGIC-003](09_Логики/LOGIC-003_Расчёт-цены-брони.md). Пересчёт в реальном времени при изменении `seats_count` **и** варианта доски. Итог дублируется на CTA.

**Источник истины по цене (канон):**
- **На SCR-004 брони ещё нет** — итог в форме это **превью**, клиентский расчёт из актуальных
  тарифов слота: `slot.price × seats_count + slot.rental_price × rental_count` (R-010; «своя»
  доска бесплатна).
- Актуальные `price` и `rental_price` берутся из текущего объекта `slot` (переданного с
  [SCR-003](SCR-003-slot-card.md) либо обновлённого после [getSlot](#getslot)).
- **После создания брони источник истины — сервер:** ответ `createBooking` (201) содержит
  серверное поле `booking.price_total` (R-005), которое и показывается на [BS-002](BS-002-booking-success.md)
  и далее (SCR-005/SCR-006). Клиент `price_total` **не пересчитывает**.
- **Изменение цены между шагами.** Если при актуализации `getSlot` между открытием экрана и
  [createBooking](#createbooking) изменились `price`/`rental_price`, UI пересчитывает **превью**
  из свежего `slot` и обновляет сумму в блоке цены и на CTA; окончательный итог брони фиксирует
  сервер в `price_total` ответа `createBooking`.

### 5. CTA «Записаться»

| Элемент | Описание | Источник данных | Валидация | Действие |
|---------|----------|-----------------|-----------|----------|
| Кнопка «Записаться · `Итого` ₽» | Фикс. нижняя, во всю ширину | итог из [LOGIC-003](09_Логики/LOGIC-003_Расчёт-цены-брони.md) | — | Отправить [createBooking](#createbooking) |

**Момент валидации:** UI-ограничения применяются непрерывно (степпер/переключатели). Финальный арбитраж лимитов — на сервере при создании брони.

**Логика:**
- При тапе → перевести CTA в загрузку, заблокировать форму, сгенерировать (если ещё нет) `idempotencyKey`, отправить [createBooking](#createbooking) с заголовком `Idempotency-Key`.

**Условия доступности:**
- CTA enabled, когда бронь валидна по [LOGIC-002](09_Логики/LOGIC-002_Расчёт-доступности.md): `1 ≤ seats_count ≤ max_seats` и `rental_count ≤ free_rental_boards`.
- CTA заблокирован от повторного тапа во время отправки (NFR-8).

### Диаграмма переходов

```mermaid
stateDiagram-v2
    [*] --> Content : onEnter (slot передан)

    Content --> Submitting : тап «Записаться» (форма валидна)
    Submitting --> Submitting : повторный тап игнорируется (NFR-8)

    Submitting --> Success : 201 Created
    Success --> [*] : переход на BS-002

    Submitting --> E1 : 409 slot_full + available_seats >= 1
    Submitting --> E2 : 409 + available_rental_boards < rental_count
    Submitting --> E3 : 409 slot_full (мест нет) / double_booking
    Submitting --> SlotGone : 410 slot_cancelled
    Submitting --> SlotStarted : 422 slot_started
    Submitting --> E4 : нет сети / таймаут
    Submitting --> Content : 5xx / прочая ошибка (снек)

    E1 --> Content : откат seats_count, форма разблокирована
    E2 --> Content : уменьшить прокат / «Своя»
    E3 --> Content : лимиты обновлены (мест >= 1)
    E3 --> [*] : мест нет → «Вернуться к слотам» (SCR-002)
    E4 --> Submitting : «Повторить» (тот же Idempotency-Key)
    SlotGone --> [*] : «Вернуться к слотам» (SCR-002)
    SlotStarted --> [*] : «Вернуться к слотам» (SCR-002)

    Content --> [*] : «назад» → SCR-003 (без сохранения формы)
```

**Защита от двойной брони и овербукинга (NFR-8):**
- Повторный тап «Записаться» во время отправки заблокирован — второй запрос не уходит.
- Финальную проверку лимитов делает бэкенд при атомарном создании записи; UI-ограничения (§степпер/переключатели) — предупреждающие, не заменяют серверную проверку. При проигрыше гонки — E3.

**Идемпотентность повтора (NFR-9):**
- `Idempotency-Key` (UUID) генерируется один раз на попытку отправки и переиспользуется при повторе после сетевого сбоя — сервер по тому же ключу не создаёт дублирующую бронь.
