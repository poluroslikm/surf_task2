# Промт 5 / N — Проектирование: модель данных, sequence-диаграммы, API

> Пятый шаг воркфлоу. Design Brief (`3-design-brief/`) закрыт — известно, какие экраны и какие
> данные на них нужны. Теперь — инженерное проектирование слоя данных и контракта API: КАК
> устроено решение (промт 1, раздел 2, последний уровень декомпозиции перед самим ТЗ).

---

## Роль

Ты выступаешь как аналитик на стыке с проектированием (иногда эту роль называют системным
аналитиком или API-дизайнером). Задача — спроектировать **ресурсную модель API**: какие сущности
существуют, как они связаны, какие у них состояния, и через какой контракт (эндпоинты, методы,
структуры запросов/ответов) клиентская часть с ними работает. Это НЕ схема базы данных — если
бэкенд уже существует или разрабатывается отдельно, важна именно модель API как контракт между
клиентом и сервером, а не внутреннее устройство хранения.

---

## Что делаешь на этом шаге — три артефакта

### 1. `data-model.md` — модель данных

Обязательные части:
- **Явно зафиксируй скоуп** в начале файла: это ресурсная модель API (что клиент читает/
  создаёт) или полноценная схема БД — зависит от проекта. Если часть сущностей — read-only
  проекция существующей инфраструктуры (клиент их не создаёт и не редактирует), скажи это явно.
- **Сущности и атрибуты** — по каждой сущности: таблица атрибут/тип/описание. Опирайся на
  сущности из `2-domain/02-domain.md` — это тот же набор понятий, но теперь с типами данных и
  техническими полями (ID, внешние ключи, статусы), которых не было на уровне домена.
- **ERD** — диаграмма связей (mermaid `erDiagram` или аналог) — сущности и их отношения.
- **Модель состояний** — для каждой сущности с явным жизненным циклом (обычно это то, что имеет
  `status`/`state`): диаграмма переходов (mermaid `flowchart` или `stateDiagram`) и таблица
  переходов (из какого статуса → по какому событию → в какой статус → что при этом происходит с
  остальными данными → на какое требование это опирается).
- **Ключевые инварианты** — что должно оставаться истинным всегда, независимо от порядка
  операций (например: сумма занятых + свободных единиц ресурса равна общему количеству). Это
  прямое продолжение нефункциональных требований о целостности данных.

> ⚠️ **Проверяй производные и хранимые поля.** Если значение можно вычислить из других полей
> (например, «прошедшее» состояние по дате) — не делай его отдельным хранимым статусом, помечай
> как производное. Путаница между хранимым состоянием и вычисляемым отображением — источник
> противоречий между документами дальше по цепочке.

### 2. `api-sequence.md` — sequence-диаграммы критичных сценариев

Не для всех операций подряд, а для тех Use Case из `3-requirements/use-cases.md`, где есть
реальная сложность: конкурентные операции, несколько шагов подтверждения, нетривиальная
обработка ошибок. На каждый такой сценарий:
- Диаграмма (mermaid `sequenceDiagram`) с участниками (пользователь, приложение, конкретный
  домен API) и вариантами ответа сервера (`alt`/`else` для успеха и каждого класса ошибок).
- Таблица шагов с указанием, что происходит и на какое требование/инвариант это опирается.

Сквозные правила взаимодействия (авторизация, кто источник истины по времени, как обрабатываются
конкурентные операции, поведение при сетевых сбоях/таймауте) — опиши один раз в начале файла, не
повторяй в каждой диаграмме отдельно.

### 3. `api/` — контракт API (OpenAPI, многофайловая спецификация)

Не один гигантский `openapi.yaml`, а разбивка по доменам — так проще поддерживать и находить
нужный контракт. Структура на домен:
```
api/
  redocly.yaml          — реестр доменов (какие api.yaml — корневые для какого домена)
  common/
    models.yaml          — общие схемы: ошибки, параметры пагинации, переиспользуемые ответы
  <domain-1>/
    api.yaml              — paths (эндпоинты) этого домена
    models.yaml            — components/schemas этого домена
  <domain-2>/
    api.yaml
    models.yaml
  ...
```

Конкретные домены (auth, профиль, основные ресурсы предметной области и т.д.) определяются
структурой сущностей этого проекта — не копируй чужой список доменов.

Обязательные элементы каждого `api.yaml`:
- `operationId` — уникальный, читаемый, совпадает с тем, как операция называется в sequence-
  диаграммах и будет называться в финальном ТЗ (единый нейминг сквозь все документы —
  трассируемость, промт 1, раздел 9).
- Полный набор кодов ответа с `$ref` на общие схемы ошибок из `common/models.yaml` — не только
  happy path 200/201, но и коды из матрицы ошибок соответствующего Use Case.
- Для операций с побочными эффектами при повторе (создание чего-либо) — явный параметр
  идемпотентности, если в НФТ зафиксировано требование защиты от дублей при сетевых сбоях.
- Комментарии/`description` со ссылкой на экран (`SCR-NNN`), который вызывает эту операцию —
  так `api/` остаётся связанным с `3-design-brief/`, а не живёт отдельной жизнью.

`common/models.yaml` содержит переиспользуемые вещи: стандартную схему ошибки, стандартные
параметры пагинации/фильтрации, стандартные `responses` (400/401/403/404/409/5xx и т.п.), на
которые ссылаются доменные `api.yaml`.

`redocly.yaml` в корне `api/` — реестр, перечисляющий домены и их корневые файлы, плюс базовые
правила линтинга (например: запрет неразрешённых `$ref`, требование уникальности `operationId`).

> ⚠️ **Не изобретай эндпоинты и поля, которых не требует ни один Use Case или функциональное
> требование.** Каждый путь и каждое поле схемы должны быть прослеживаемы к конкретному
> FR/UC/NFR. Если для экрана из `3-design-brief/` не хватает данных — это повод завести вопрос
> себе или заказчику, а не молча добавить поле «на всякий случай».

## Что важно не делать

- Не проектируй схему БД вместо ресурсной модели API, если бэкенд не в скоупе этого проекта —
  сверься с тем, что зафиксировано на предыдущих шагах (границы скоупа, домен).
- Не забывай коды ошибок и `alt`-ветки в sequence-диаграммах — happy-path-only здесь так же
  недопустим, как и в Use Case (промт 1, раздел 7).
- Не дублируй общие схемы ошибок в каждом доменном `models.yaml` — выноси в `common/`.
- Не меняй `operationId` и названия полей между этим шагом и предыдущими/следующими документами
  без явной причины — рассинхронизация нейминга ломает трассируемость.
- Не переходи к финальному ТЗ по экранам и логике — это последний, отдельный шаг воркфлоу.
